An HTAP table is a table within a Postgres Distributed (PGD) cluster that uses Postgres Analytics Accelerator (PGAA) to continuously synchronize its content with an analytical copy in object storage. 

While this dual-state allows you to run heavy analytics without slowing down your transactions, it requires double the storage: once in the local **heap** and once in the data lake.

Offloading to analytics is a manual, surgical operation used to remove the local heap copy and reclaim disk space. When you offload a table, the system:

- Truncates the local heap table, immediately freeing up disk space on the PGD nodes.
- Stops the real-time replication to the data lake.
- Changes the table access method to PGAA, pointing the Postgres table definition directly to the existing data in object storage.

The result is a cold table that remains fully queryable through Postgres but consumes no local storage. If transactional writes are required again in the future, you can restore the table and stream the data back from the lake into the local heap.

## Process overview

To surgically offload a table to the data lake, follow these steps:

1. Meet the [prerequisites](#prerequisites).
1. [Point to object storage](#pointing-to-object-storage): by choosing either a storage location or a catalog service.
1. [Offload the table to object storage](#offloading-a-table).
1. [Restore the table](#restoring-from-analytics) back to heap if transactional access is needed again.


## Prerequisites

- **Cluster:** PGD version 6.1 or later with PGAA and PGFS extensions installed.
- **Catalog:** An external REST-compatible catalog service (optional).
- **Storage:** S3-compatible object storage with valid credentials.
- **Permissions:** The database user must have `CREATE`, `ALTER`, and `EXECUTE` privileges for the PGD and PGAA functions.

## Pointing to object storage 

You must define the destination for the offloaded data. Because you are working within a distributed PGD cluster, you must wrap these configuration calls in `bdr.replicate_ddl_command` to ensure the settings are propagated to all nodes in the group.

### Option 1: Using a storage location

1. Create a storage location using the [PGFS extension](/edb-postgres-ai/current/ai-factory/pipeline/pgfs/functions/#creating-a-storage-location) and set it as the analytics storage for your node group.

  ```sql
  SELECT bdr.replicate_ddl_command($$
    SELECT pgfs.create_storage_location(
    'your-storage-location',
    'protocol://your-bucket-name/path/',
    '{"region": "region-name"}',
    '{"access_key_id": "...", "secret_access_key": "..."}'
    )
  $$);
  ```

1. Clear `analytics_write_catalog`. This must be `NULL` before an `analytics_storage_location` can be assigned:

  ```sql
  SELECT bdr.alter_node_group_option('your-pgd-group', 'analytics_write_catalog', NULL)
  ```

1. Set the analytics storage location. This command targets your specific PGD node group and starts the Seafowl engine:

  ```sql
  SELECT bdr.alter_node_group_option('your-pgd-group', 'analytics_storage_location', 'your-storage-location');
  ```

1. Verify that Seafowl has started. (look for `pgd seafowl_manager`):

  ```sql
  SELECT application_name FROM pg_stat_activity WHERE application_name LIKE '%seafowl%';
  ```

### Option 2: Using a catalog service
      
1. Configure a connection to an external catalog:

      ```sql
      SELECT bdr.replicate_ddl_command($$
        SELECT pgaa.add_catalog(
          'your-catalog-name',
          'iceberg-rest',
          '{
            "url": "https://your-catalog-rest-endpoint.com",
            "token": "YOUR_AUTH_TOKEN",
           }'
        )
      );
      ```

1. Set the catalog as the write destination:

  ```sql
  SELECT bdr.alter_node_group_option(
    SELECT bdr.alter_node_group_option('your-pgd-group', 'analytics_write_catalog', 'your-catalog-name')
    );
  ```

1. Verify that Seafowl has started. (look for `pgd seafowl_manager`):

  ```sql
  SELECT application_name FROM pg_stat_activity WHERE application_name LIKE '%seafowl%';
  ```

## Offloading a table

Use the `pgaa.convert_to_analytics()` function to convert a synchronized HTAP table into a cold, analytical table and remove the data from local disk.

```sql
CALL pgaa.convert_to_analytics('my-table'::regclass);
```

The function removes the local data and redefines the table's access method to PGAA to point solely to the external Iceberg or Delta files.

Alternatively, switch the access method with the `ALTER TABLE` command. Use the `pgaa.auto_truncate` option to ensure local disk space is freed immediately.

For storage locations:

```sql
ALTER TABLE public.my_table
  SET ACCESS METHOD pgaa,
  SET (
    pgaa.auto_truncate = 'true',
    pgaa.format = 'iceberg',
    pgaa.storage_location = 'your-storage-location',
    pgaa.path = 'public.my_table'
  )
```

For a catalog-managed table:

```sql
ALTER TABLE public.my_catalog_managed_table
  SET ACCESS METHOD pgaa,
  SET (
      pgaa.auto_truncate = 'true',
      pgaa.format = 'iceberg',
      pgaa.managed_by = 'my_iceberg_catalog',
      pgaa.catalog_namespace = 'public',
      pgaa.catalog_table = 'my_remote_table_name'
  );
```

## Restoring from analytics

If you need to resume transactional writes or bring the data back to local high-performance transactional storage, use the `pgaa.restore_from_analytics()` function.

```sql
CALL pgaa.restore_from_analytics('public.transactional_table');
```

The function checks the PGAA table's configuration, identifies its external Iceberg/Delta path in object storage, and rewrites the data back into the table's local storage structure. Once the data is locally present, the function sets the table's access method back to the local transactional endpoint again.

You can also use the `ALTER TABLE` command to restore a table from its cold state in the data lake back into a local transactional heap table. This manual method provides granular control over the transition back to local storage.

```sql
  ALTER TABLE public.my_table 
  SET ACCESS METHOD heap
```

!!! Warning
  Restoring a table to the heap access method does not remove the files from object storage. Manually delete the data via your object storage provider or use a tool like PyIceberg to drop the table from the catalog.
