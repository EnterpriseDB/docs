Hybrid Manager (HM) Lakehouse is a powerful tool for running analytical queries on your data.
It's built on top of Postgres, so you can use your existing Postgres knowledge and tools.

To get the best experience with Lakehouse, follow the [quick start](/pgaa/current/quick_start.mdx) to query benchmarking data.
Then you can try loading your own data and [query Delta Lake tables](/pgaa/current/how-to-lakehouse-read-with-without-catalog).

This section details some of the important things to know:

-   [Benchmarking datasets](/pgaa/current/reference/datasets.mdx)
-   [Queries in Lakehouse](/pgaa/current/reference/queries.mdx)
-   [Delta Table tools](/pgaa/current/reference/delta_tables.mdx)
-   [DirectScan and fallback modes](/pgaa/current/reference/directscan.mdx)
-   [Users](/pgaa/current/reference/users.mdx)
-   [Loading data](/pgaa/current/reference/loadingdata.mdx)
-   [PGAA functions](/pgaa/current/reference/pgaa_functions.mdx)

For information on the PGFS functions for managing storage location, see the [AI Factory PGFS documentation](/edb-postgres-ai/latest/ai-factory/pipeline/pgfs/).

* [Configuration parameters](configuration_parameters): The behavior of the PGAA extension is governed by Grand Unified Configuration (GUC) variables. These parameters allow you to switch executors, enable performance optimizations, and manage security credentials.

* [Functions](functions): PGAA introduces a suite of SQL functions for administrative tasks, such as mapping new tables, monitoring storage health, and launching maintenance background jobs.

* [Table options](table_options): When mapping or creating analytical tables, specific options allow you to define how data is read from or written to your object store.

* [Data types](data_types): PGAA maps native Postgres data types to optimized columnar formats in the data lake.

* [Datasets](datasets): Access pre-configured schemas and data loading instructions for analytical datasets to baseline your performance.
