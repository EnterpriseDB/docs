If you're having issues configuring the agent or are getting errors when trying to run it, you can troubleshoot. Test the agent’s functionalities in an isolated way to narrow down the source of the issue. 

## Testing connectivity from the agent to Hybrid Manager

To ensure there's a connection between the agent and Hybrid Manager (HM), run the agent:

```
beacon-agent --config
```

See [Agent CLI](/tools/agent/agent_ref) for other agent modes and options.

If the output displays no errors, your connection to HM is working properly. The issue might have to do with the connection to your databases. 

If the output displays errors, your connection to HM isn't working properly. Check the syntax of the YAML file, the access key, the CA path and contents, DSN string, and project ID one after the other to ensure the settings are correct. 

## Error messages and solutions

`"fail fetching server version: pq: SSL is not enabled on the server"`

If the database you are connecting to has SSL disabled, you must explicitly instruct the Agent to use an unencrypted connection. To do this, append the setting `?sslmode=disable` to the DSN used in the [Agent's configuration](../mon_with_agent/configure_agent), for example: 

```shell
export DSN="postgresql://postgres:password@localhost:5432/postgres?sslmode=disable"
```

## Schema change events not appearing in Hybrid Manager

If schema change events or other agent telemetry aren't appearing in the HM console, the activity service (`accm-server`) may be degraded or unavailable.

**Symptoms:**

- The agent logs show errors sending events to the platform.
- Schema changes are not reflected in the HM console despite the agent running normally.
- Other event-driven features (such as new query detection) are also not updating.

**Resolution:**

This is expected behavior when the activity service is down, it is not a defect in the agent or the monitored database. The agent continues to detect changes locally and queues events for delivery. Once the activity service recovers, queued events are delivered automatically and the HM console updates.

If the issue persists, verify that the `accm-server` pod is running and healthy in your HM Kubernetes deployment.
