## 1. Management workstation (Bastion host)

> **Optionality:** Strict requirement.

You require a designated machine to orchestrate the deployment. 
This can be a laptop (for public clusters) or a cloud-based Bastion host (for private clusters).

!!! Note
A **Bastion host** (or jump host/box) is strongly recommended for secure, private access to the Kubernetes API, Portal, and PostgreSQL endpoints. 
This host serves as your dedicated operational workstation for installation and validation activities.
For EDB Remote DBA (RDBA) or Managed Services contracts, a dedicated Bastion is **mandatory**.
!!!

### General requirements

* **Operating System:** Linux (AMD64/ARM64) or macOS.
    * *Note:* Windows users must use **WSL2** (Windows Subsystem for Linux).

* **Network Access:**
    * **Kubernetes API:** Must have network connectivity to the Cluster API Server (typically port 6443).
    * **Internet/Registry:** Must be able to reach the Container Registry to pull Helm charts and images.

* **Storage:** Minimal (enough to hold configuration files and certificates).

### Required tooling inventory

You do not need to install these tools in this phase. 
However, you must ensure your workstation environment allows for the installation and execution of the following binaries in the next phase (Phase 3):

**Core installation tools**
* `edbctl` (EDB Hybrid Manager CLI)
* `kubectl` (Kubernetes CLI) + `cnpg` plugin
* `helm` (Kubernetes package manager)
* `yq` (Command-line YAML processor)

**Utilities**
* `curl` (Download utility)
* `openssl` (Certificate management)
* `htpasswd` (Required only if using Static Users)

**Platform CLIs (Environment dependent)**
* `aws` (AWS CLI)
* `gcloud` (Google Cloud CLI)
* `oc` (OpenShift CLI)

**Operational tools (Post-install)**
* `pgdcli` (Postgres Distributed management)
* `sentinel` (Monitoring and failover management)

## 3. Compute (Node requirements)

Ensure your worker nodes meet the minimum sizing for your intended deployment topology (Minimum vs. Fully featured) as defined in the **[Planning Architecture](./planning_arch)** phase.

Hybrid Manager components require **AMD64/x86-64 nodes**. 
Mixed-architecture clusters (ARM64 + AMD64) are not supported for HM, even though PostgreSQL itself can run on ARM64.

### Node roles summary

| Node type | Purpose | Count | Kubernetes nodes | Required label |
| :--- | :--- | :--- | :--- | :--- |
| **Control plane** | Runs HM control plane & telemetry. | 3+ | Control or Worker | `edbaiplatform.io/control-plane: "true"` |
| **Data plane** | Runs Postgres databases. | 0 or 3+ | Worker | `edbaiplatform.io/postgres: "true"` |
| **AI model (GPU)** | *Optional:* Runs AI/ML workloads. | 0 or 2+ | Worker | `nvidia.com/gpu: "true"` |

Use Node Pools (e.g., AWS NodeGroups, RHOS Machine Sets) to manage resources and apply required labels/taints.

!!! Note
The HM system databases — the core platform database and the databases backing the Beacon Server, Migration Portal, and Transporter (the DMS service) — run on control-plane nodes, not the data plane. The data-plane pool hosts only the user-provisioned Postgres clusters.
!!!

In most cases, HM runs on Kubernetes worker nodes in cloud environments (since you do not have access to managed control nodes&mdash;see EKS control plane access), and can run on control nodes ([see Kubernetes control plane documentation](https://kubernetes.io/docs/concepts/architecture/control-plane-node-communication/)) or worker nodes for on-premises clusters.

### 3.1 Control plane nodes

> **Optionality:** Strict requirement.

Resource requirements for the control plane increase with the number of databases monitored. (Note: The Telemetry stack scales with the monitoring load).

**Sizing guidelines:**

| Resource | Minimum (Up to 10 DBs) | Standard (10–50 DBs) | Large (>50 DBs) |
| :--- | :--- | :--- | :--- |
| **CPU** | 8 vCPUs | 16 vCPUs | 16+ vCPUs |
| **Memory** | 32 GB RAM | 64 GB RAM | 64+ GB RAM |
| **Disk** | 100 GB SSD | 200 GB SSD | >200 GB SSD |
| **Quantity** | 3 nodes | 3 nodes | 3+ nodes |
| **Max pods/node** | ≥ 60 | ≥ 60 | ≥ 60 |

**Pod density (`maxPods`) — easy to miss, and not a CPU/memory problem.** The control-plane stack is pod-dense: in addition to telemetry and the service mesh, the **HM system databases run on control-plane nodes** (see [Node roles summary](#node-roles-summary)). An all-scenario install places **~150 pods** on the control-plane pool, which across the minimum 3 nodes is ~50 pods/node — so each control-plane node must allow **at least 60 pods**.

This is independent of CPU and memory: a node can have spare CPU/RAM and still reject pods once it hits its pod cap. On **AKS the Azure CNI default is `maxPods=30`**, which is too low — control-plane-pinned pods (for example `upm-beacon-server`) stay `Pending` with *"Too many pods"* and the install stalls. Set the limit **at node-pool creation time** (`az aks nodepool add … --max-pods 110`; you can't change it afterward) or use Azure CNI Overlay (default 250). GKE (default 110) and EKS (per-ENI IP limits) generally have enough headroom, but verify against the ~60-pods/node figure above.

### 3.2 Data plane nodes

> **Optionality:** Strict requirement.

The data plane nodes must be sized to host the **PostgreSQL** database clusters provisioned by users.
* **Sizing:** Depends entirely on expected database workloads, storage requirements, and performance SLAs as defined in [Phase 1](./planning_arch.mdx).

### 3.3 AI model nodes

> **Optionality:** Required only if utilizing GenAI/Agent Factory capabilities (Installation scenario `ai` is enabled).

* **Sizing:** Current recommendations require **Nvidia B200 GPUs** (or equivalent supported hardware).

### Node abstractions

To ensure the HM Control Plane runs on dedicated resources and Postgres workloads are on independent resources, or to partition dedicated resources for your AI workloads, you must apply specific configurations to each of these node pools before [deploying your Kubernetes cluster](./k8s_install/#set-your-node-abstractions).

### Node allocatable resources requirements

Hybrid Manager components have specific resource requirements that the nodes they're scheduled on must satisfy. When defining your node pools, ensure the allocatable resources meet the minimum requirements for the Hybrid Manager components and Postgres clusters you plan to run. Each scenario corresponds to a set of Hybrid Manager components and has its own resource requirements:

| Scenario | Total CPU | Total Memory | Total Storage | Total Ephemeral Storage |
|---------|---------|---------|---------|---------|
| ai | 3366m | 4749Mi | 4Gi | 250Mi |
| analytics | 200m | 456Mi | 0 | 0 |
| core | 10020m | 30445Mi | 454706Mi | 3Gi |
| dbaas | 0 | 0 | 0 | 0 |
| migration | 2880m | 6432Mi | 54Gi | 0 |

In addition, the Hybrid Manager operator itself requires `260m CPU` and `576Mi memory`.

## 4. Local networking

### Local network requirements

HM is not strictly opinionated about local networking logic. 
The standard networking capabilities provided by major Cloud Service Providers (AWS, Azure, GCP) are sufficient, provided they offer **low latency and high bandwidth** between availability zones.

* **On-premises:** Networking should follow best practices regarding switching, link bonding, and link redundancy to ensure high availability.

* **Protocol:** **IPv4 only**. Kubernetes must be configured for IPv4. IPv6 has not been thoroughly productized for Hybrid Manager; if IPv6 is required, it must be masked behind a Load Balancer so that internal communication remains IPv4.

* **Defined Network Address Spaces (CIDRs)**: Your cluster must be configured with distinct, non-overlapping IP ranges for:
    
    * Pod Network (clusterCIDR): The IP range from which all pods are assigned their IPs.
    * Service Network (serviceCIDR): The IP range from which all internal services (ClusterIPs) are assigned their virtual IPs.
    * Functional DNS and NTP: The cluster's internal DNS service (typically CoreDNS) must be running and able to resolve both internal cluster services (e.g. kubernetes.default) and external addresses.

#### Container Network Interface (CNI)

A functional CNI is a strict requirement for pod-to-pod networking.

* **Examples**: Calico, Cilium, Flannel

* **Context:** For the **data plane**, database I/O — not pod density — is the bottleneck, so CNI pod-density tuning rarely matters there. The **control plane** is different: it is pod-dense, so your CNI and per-node `maxPods` setting must allow at least 60 pods per control-plane node. See [3.1 Control plane nodes](#31-control-plane-nodes) for platform-specific guidance on setting `maxPods`. Cilium is more efficient at very high pod counts if you need it.

* **Validation:** Verify CNI setup and CIDR assignment:

```bash
kubectl get nodes -o custom-columns=NAME:.metadata.name,PODCIDR:.spec.podCIDR
kubectl get svc
```

## 5. Ingress

You must choose an ingress strategy that matches your infrastructure capabilities. HM can deploy either a load balancer or NodePort service for its own Portal and services, as well as individual load balancers and NodePorts for each Postgres database you create.
Choose an ingress strategy that matches your infrastructure capabilities. HM can deploy either a load balancer or NodePort service for its own portal and services. It can also deploy individual load balancers and NodePorts for each Postgres database you create.
The table below defines the supported combinations of DNS Management, Endpoint Types, and SSL Termination.

| Scenario | DNS Management | Endpoint Type | Portal SSL/TLS Termination |
| :--- | :--- | :--- | :--- |
| **CSP LB (ELB, ALB)** | Manual portal, dynamic postgres | Dynamic via Load Balancer Controller | Istio Ingress Gateway |
| **F5 + F5 K8s Controller** | Manual portal, dynamic postgres | Dynamic via Load Balancer Controller | Istio Ingress Gateway |
| **MetalLB** | Manual | Dynamic via Load Balancer Controller | Istio Ingress Gateway |
| **NodePort (Conventional LB)** | Manual (A record against LB) | Static (manual) | Istio Ingress Gateway |
| **NodePort (No LB)** | Manual (Round Robin against Nodes) | Static (manual) | Istio Ingress Gateway |

### 5.1 Load balancer controller

> **Optionality:** Required option for an optimal experience.

Examples of load balancer controllers range from the AWS Load Balancer controller to MetalLB or a BigIP F5 combined with DNS capability and their F5 Load Balancer Controller.

**General requirements**

* **Firewall Annotations:** If using a specific provider scheme (e.g., AWS Internet Facing), you know the correct `resourceAnnotations` to provision the load balancer correctly:

*AWS example annotation*: 

```yaml
globalParameters:
  load_balancer_mode: public
  load_balancer_provider: aws
resourceAnnotations:
  - name: istio-ingressgateway
    kind: Service
    annotations:
      service.beta.kubernetes.io/aws-load-balancer-scheme: internet-facing
```

*MetalLB example annotation*:

```yaml
globalParameters:
  load_balancer_mode: custom-only
resourceAnnotations:
  - name: istio-ingressgateway
    kind: Service
    annotations:
      metallb.io/address-pool: production-public-ips
```

**Configuration dependencies**

| Parameter | Custom Resource key | Value / Format |
| :--- | :--- | :--- |
| **Enable LB** | `spec.globalParameters.load_balancer_mode` | `"public"`, `"private"`, or `"disabled"` |
| **Provider** | `spec.globalParameters.load_balancer_provider` | `aws`, `azure`, or `gcp` |
| **Annotations** | `spec.resourceAnnotations[]` | Provider-specific list (see below). |


The `load_balancer_mode` parameter lets you instruct HM to create a load balancer service for both the Portal and the Postgres databases. If set to 
`disabled`, both the portal and the Postgres databases use NodePorts; otherwise, they use a load balancer. The `public` and `private` values only
 affect the Postgres databases and have no impact on the portal load balancer. Similarly, the `load_balancer_provider` parameter applies only to Postgres clusters.
  The `resourceAnnotations` parameter is relevant only for the HM Portal.

**Spot validation**

To verify that your controller is active, use the following command:

```bash
kubectl get pods -A | grep -i 'load\|lb\|router\|metallb\|gateway'
```

**Postgres ports (Data Plane)**

With a proper load balancer setup, the following are Postgres ports:

| Port | Protocol | Description |
| :--- | :--- | :--- |

**SSL Termination**

HM is exposed through the Istio Gateway by using TLS. If you are using a load balancer in front of Istio to access HM, you have two options:

1. **SSL passthrough**: Configure the load balancer to forward TCP traffic without modification.
2. **TLS termination**: Terminate the TLS session from the client at the load balancer, then have the load balancer establish a new TLS connection with Istio.

#### 5.1.2 NodePort alternative

### 5.2 DNS

> **Optionality:** DNS configuration is a **required option for an optimal experience**.

You must ensure proper DNS resolution both inside the cluster and for external access.

#### Internal cluster DNS & NTP

* **CoreDNS:** The cluster's internal DNS service (typically CoreDNS) must be running and able to resolve both internal cluster services (e.g., `kubernetes.default`) and external internet addresses.

* **NTP:** Functional NTP is required to ensure time synchronization across nodes.

* **Container Registry:** Your DNS service must be able to resolve the DNS entries for the container registry (whether local or EDB's).

#### Configuration dependencies

You must provision and control the following domains.
These are inputs into your HM configuration in the next phase:

| Domain Purpose | Custom Resource key | Description |
| :--- | :--- | :--- |
| **Portal domain** | `spec.globalParameters.portal_domain_name` | The host name for the HM Portal UI. |
| **Agent domain** | `spec.componentsParameters.upm-beacon.server_host` | The host name through which the Beacon Server API is reachable. |
| **Migration domain** | `spec.globalParameters.dms_domain_name` | The internal domain for the Transporter migration service. Required unless the `migration` installation scenario isn't enabled. |
| **NodePort base** | `spec.globalParameters.node_port_domain` | **(Scenario B Only)** The base domain for Postgres access if using NodePort. |

In case of NodePort approach, the above DNS entries should be round robin on the compute IP where the control-plane label is applied on the Kubernetes Nodes.

#### DNS configuration strategy

Your DNS configuration depends on your Ingress selection.

**Scenario A: Load balancer (recommended)**

* **Dependency:** Ensure `spec.globalParameters.load_balancer_mode:` is set to `"public"` or `"private"` in your HM configuration.

* **Portal:** In the case of an optimal load balancer controller like AWS ELB or an F5 with DNS capability, manually configure DNS to point to the resulting Load Balancer IP for `istio-ingress`. Domain names must be resolvable to a **locally routable IP** so that the ingress service on Kubernetes can properly route traffic according to the hostname in the HTTPS request.

* **Postgres:** You do *not* need to manually manage DNS for every database. 

Postgres DNS records are populated automatically as a function of the load balancer controller updating the status of the Service in the namespace dedicated to that Postgres cluster.

**Scenario B: NodePort alternative**

* **Prerequisite:** You disabled load balancers and defined a `node_port_domain` in the Ingress section.

* **Portal:** DNS entries should be a **Round Robin A Record** pointing to the compute IPs of the **Control Plane nodes** (nodes labeled `edbaiplatform.io/control-plane`).

* **Postgres:** The `node_port_domain` value is used as the base URL for all Postgres instances. It should be a **Round Robin DNS record** pointing to the IP addresses of the **Worker nodes** where the Postgres clusters are running.

## 6. Block storage (Database & logs)

> **Optionality:** Strictly required.

You must define a Kubernetes `StorageClass` that presents persistent storage volumes (PVCs).

The default storage class used for the **HM Control Plane** is set in your HM configuration.
The same storage class is an option for all Postgres instances' deployments as well.
However, for the **Data Plane** (your Postgres deployments), you may establish any number of additional storage classes in the cluster. 
All of them are available options for Postgres deployments, providing the flexibility needed for critical or intensive workloads.

### Control plane requirements

Block storage demands for the HM Control Plane are relatively light. 
A storage class roughly equivalent to the capability of **AWS gp2** is sufficient.

#### Data plane requirements (Postgres)

Postgres workloads may require specialized configurations depending on their specific I/O patterns.

* **Latency Sensitivity:** Postgres is generally more sensitive to latency—and specifically the *consistency* of latency—than raw throughput.

* **Example: Analytics:** Analytic capabilities perform as expected with approximately **16,000 IOPS** capacity per PVC.

* **Example: Azure:** In Azure, IOPS-capable storage (Premium SSD) is highly preferred—even when provisioning low IOPS capacity—due to its highly consistent latency compared to Standard tiers.

### Notable considerations

Hybrid Manager and containerized Postgres are particularly well suited to leverage **Local Storage**.
Unlike traditional applications, these solutions do not strictly depend on the PVC being replicated across the Kubernetes cluster by the storage provider. 
If a Postgres instance loses its underlying node (and local disk), the new pod does not need to remount the old PVC. 
Instead, the Hybrid Manager High Availability (HA) logic rebuilds the new node from its peers or backup.

### Configuration dependencies

| Parameter | Custom Resource key | Value / Format |
| :--- | :--- | :--- |
| **Storage Class** | `spec.globalParameters.storage_class` | The exact name of your K8s StorageClass. |

##### Validation

Verify available storage classes:

```bash
kubectl get sc
```

### 6.1 Volume snapshots

* **Optionality:** Required for an optimal experience (backups will be limited without it).

The underlying storage provider **must support Volume Snapshots** to enable the backup and recovery features of the platform.

* **Mechanism:** Hybrid Manager automatically utilizes volume snapshots if the capability is exposed via the Kubernetes API.

* **Implementation note:**
    * **Cloud:** Typically handled by the CSP's CSI driver (e.g., AWS EBS CSI), but often requires installing a "Snapshot Controller" add-on.
    * **On-Premises:** Your storage backend must support the CSI Snapshotter specification, and you must define a `VolumeSnapshotClass` that maps to it.

##### Validation

Verify that a snapshot class is configured and available:

```bash
kubectl get volumesnapshotclass
```

### 6.2 Excluding storage from Hybrid Manager discovery

In shared Kubernetes environments, some `StorageClasses` or `VolumeSnapshotClasses` may not be suitable for Postgres workloads — for example, GPU- or HM-only classes.
By default, every discovered class appears in the Hybrid Manager **Create Cluster** dropdowns, which can lead operators to pick a class on which a Postgres cluster will never successfully deploy.

To hide a class from the Hybrid Manager provisioning UI, apply the `beacon.enterprisedb.com/exclude-storage-from-dbaas` label with value `true`:

```bash
kubectl label storageclass <name> beacon.enterprisedb.com/exclude-storage-from-dbaas=true
kubectl label volumesnapshotclass <name> beacon.enterprisedb.com/exclude-storage-from-dbaas=true
```

Behavior:

* **Opt-in:** Only the exact value `"true"` excludes a class. An absent label, or any other value, preserves current behavior — the feature is a no-op for existing deployments.

* **Default-safe:** An excluded `StorageClass` is never promoted to Hybrid Manager's discovered default, even if it carries the `storageclass.kubernetes.io/is-default-class` annotation.

* **Takes effect on the next agent heartbeat:** No agent restart is required.

* **UI-only:** The label hides the class from Hybrid Manager's provisioning UI. Workloads that reference the class by name — for example, a custom resource or a direct `kubectl apply` — can still use it.

To restore a class to the provisioning UI, remove the label:

```bash
kubectl label storageclass <name> beacon.enterprisedb.com/exclude-storage-from-dbaas-
kubectl label volumesnapshotclass <name> beacon.enterprisedb.com/exclude-storage-from-dbaas-
```

##### Validation

Audit which classes are currently excluded:

```bash
kubectl get sc,volumesnapshotclass -l beacon.enterprisedb.com/exclude-storage-from-dbaas=true
```

## 7. Object storage

> **Optionality:** Strictly required.

Object storage is the backbone of disaster recovery and data mobility. It is used for:

* Velero backups (Cluster DR)
* Postgres WAL archiving (Point-in-Time Recovery)
* Unstructured data for GenAI/AIDB
* Parquet files for Lake Keeper
* Log files

### General requirements

* **Protocol:** Must be fully **S3 Compatible**.

* **Consistency:** In multi-location deployments, the object storage configuration must be available equally across all locations. (See [multi-dc guidance](/edb-postgres-ai/current/hybrid-manager/install/configuration/multi_dc/) for details.)

!!! Note
You must provision a Kubernetes Secret named `edb-object-storage` in the `default` namespace. 
The contents of this secret must be identical in every cluster. 
You are not expected to run `kubectl create secret` at this point in the process. 
Here, you are gathering the raw credentials (API keys, certificates, passwords) so you have them ready to inject into the cluster in [Phase 4: Preparing your environment](./environment_prep).
!!!

!!! Important
Your object storage must be dedicated entirely to the Hybrid Manager. 
Any prior data existing in your object storage interferes with the successful operation of the HM. 
Similarly, if HM is removed and reinstalled, then the object storage must be emptied prior to beginning the new installation.
!!!

## 8. Container registry

> **Optionality:** A local container registry is a **required option for an optimal experience**.

You require a container registry that is accessible from your Kubernetes Cluster to host HM and Postgres images.

* **Purpose:** To serve the specific application images required for the Control Plane and Data Plane.

* **Access Requirement:** The cluster must be able to pull images from this registry.

* **Permissions:** The authentication credential used must have permissions to **List Repositories**, **List Tags**, and **Read Tag Manifests**.

If your environment cannot access the public EDB repository, you must pull the artifacts and push them to your local private registry using `edbctl`.

### Choosing a registry approach

You have two registry options:

* **External private registry (recommended).** Use your organization's existing registry — for example, AWS ECR, Google Artifact Registry, Azure Container Registry, or Harbor. Best for environments with established registry tooling, scanning, and access control. See [Syncing images to a local private registry](../configuration/sync_images).

* **In-cluster container registry.** A registry embedded in the EDB HCP Operator itself. Suitable only for fully air-gapped clusters whose nodes have no public IPs. See [Using the in-cluster container registry](../configuration/in_cluster_registry).

!!! Warning In-cluster registry security
The in-cluster registry serves images over plain HTTP without authentication and exposes itself to nodes through a NodePort. It's only safe when every cluster node is on a private subnet with no public IP. If any node has a public IP, the registry — and every image it stores — is reachable from the internet without credentials.

If your nodes have public IPs, don't use the in-cluster registry. Use an external private registry instead.
!!!
