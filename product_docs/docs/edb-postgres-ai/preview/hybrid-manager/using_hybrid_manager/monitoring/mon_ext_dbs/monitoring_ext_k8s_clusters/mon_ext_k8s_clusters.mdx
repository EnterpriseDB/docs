## Installation methods

!!!important
Set the [environment values](/edb-postgres-ai/preview/hybrid-manager/using_hybrid_manager/monitoring/mon_ext_dbs/monitoring_ext_k8s_clusters/installation_requirements/#required-environment-values) before proceeding with the installation of the EDB Postgres AI agent for external Kubernetes cluster.
!!!


This chart is designed to work across different Kubernetes platforms (including OpenShift) with a unified configuration:

| Environment         | Configuration                                   | OpenShift Flag                           |
|---------------------|-------------------------------------------------|------------------------------------------|
| KIND, GKE, EKS, AKS | Uses image defaults (numeric or named)          | openshift.enabled: false (default)       |
| OpenShift (RHOS)    | Compatible via runAsUser: null (SCC assigns ID) | openshift.enabled: false/true (optional) |

Key points:

- **Universal Compatibility:** The chart defaults to `runAsUser: null`, allowing OpenShift SCCs to assign valid UIDs automatically, while standard Kubernetes falls back to the image's default user.
- **Security Context:** runAsNonRoot strategy varies by component to ensure compatibility with standard Kubernetes validation logic vs OpenShift named-user support.
- **OTEL collector:** Uses the image default user (nonroot) without explicit numeric IDs to satisfy both strict K8s validation and OpenShift SCCs.

## Downloading the Helm chart and customizing the values file

1. Obtain an EDB subscription token.

1. Download the EDB Postgres AI agent chart from the EDB repository:

   ```bash
   helm repo add enterprisedb-$EDB_REPO "https://downloads.enterprisedb.com/$EDB_SUBSCRIPTION_TOKEN/$EDB_REPO/helm/charts/"
   ```

1. Create a `custom-values.yaml` file that includes your Hybrid Manager (HM) access credentials:
   
   ```yaml
   # Required: Your HM access credentials
   agentConfig:
     accessKey: ${HM_ACCESS_KEY} 
     projectID: ${HM_PROJECT_ID}
     beaconServer:
       host: ${HM_URL}
       plaintext: false
     metrics:
       push:
         endpoint: ${HM_URL}
     webhook:
       enabled: true
       podSelector: ${POD_SELECTOR}
   # Required: Docker registry credentials for EDB images
   image:
     repository: docker.enterprisedb.com/${EDB_REPO}/upm-beacon-agent
     tag: ${AGENT_VERSION}
     imageCredentials:
       registry: docker.enterprisedb.com
       username: ${EDB_REPO}
       password: ${EDB_SUBSCRIPTION_TOKEN}
   
   # OTEL collector image
   otelImage:
     repository: "docker.enterprisedb.com/${EDB_REPO}/upm-beacon-external-metrics-receiver"
     tag: ${AGENT_VERSION}
   
   # Optional: Enable additional monitoring components
   additionalCharts:
     cert-manager:
       install: false 
     opentelemetry-collector:
       install: true  # Enable for metrics collection
     kube-state-metrics:
       install: true  # Enable for Kubernetes state metrics
     prometheus-node-exporter:
       install: true  # Enable for node metrics (requires privileged access)
   
   # Required if opentelemetry-collector.install is true:
   opentelemetry-collector:
     image:
       repository: "docker.enterprisedb.com/${EDB_REPO}/upm-beacon-external-metrics-receiver"
       tag: ${AGENT_VERSION}
   ```    

   For other configuration parameters and options, see the [Configuration reference](/edb-postgres-ai/preview/hybrid-manager/using_hybrid_manager/monitoring/mon_ext_dbs/monitoring_ext_k8s_clusters/configuration_reference/).

1. Apply the helm chart:

   - For standard Kubernetes (GKE, EKS, AKS, KIND):
   
   ```helm
   helm upgrade --install beacon-agent-k8s \
    enterprisedb-enterprise/beacon-agent-k8s \
    --create-namespace \
    --namespace upm-beacon \
    --values custom-values.yaml
   ```

   - For OpenShift clusters:
   Add the `--set openshift.enabled=true` flag:
   
   ```shell
   helm upgrade --install beacon-agent-k8s \
    enterprisedb-enterprise/beacon-agent-k8s \
    --create-namespace \
    --namespace upm-beacon \
    --set openshift.enabled=true \
    --values custom-values.yaml
   ```

   !!!important
   Grant SCC if using OTEL collector:

   ```shell
   oc adm policy add-scc-to-user privileged \
     system:serviceaccount:upm-beacon:beacon-agent-k8s-opentelemetry-collector
   kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-opentelemetry-collector-agent
   ```

   !!!

1. To allow your target namespace to pull the required images, copy the beacon-agent-pull-secret from the agent's namespace to the CloudNativePG™ cluster namespace:

   ```
   kubectl get secrets -n "$AGENT_NS" -oyaml beacon-agent-pull-secret | grep -v '^\s*namespace:\s' | kubectl apply -n "$TARGET_NS" -f -
   ```

## Patching clusters to enable the agent sidecar

This section shows how to patch a single cluster. You can adapt the instructions below to patch many clusters:

1. Define the target resource you wish to patch. Export the TARGET_NAME for your CloudNativePG™ cluster or PGD group (for EDB Postgres® AI for CloudNativePG™ Cluster) to which you wish to add the agent.

   ```shell
   export TARGET_UID="$(kubectl get "${TARGET_TYPE}" "${TARGET_NAME}" -n "${TARGET_NS}" -o jsonpath='{.metadata.uid}')"
  ```

1. Create the configuration patch. This template adds the necessary labels and ensures the agent can pull the required images:
  
  ```shell
   PATCH_CONTENT=$(cat <<EOF
   metadata:
     labels:
       k8s.enterprisedb.io/beacon: enabled
   spec:
     imagePullSecrets:
     - name: beacon-agent-pull-secret
     inheritedMetadata:
       labels:
         k8s.enterprisedb.io/beacon: enabled
         k8s.enterprisedb.io/beacon-project-id: "${HM_PROJECT_ID}"
         k8s.enterprisedb.io/beacon-resource-id: "${TARGET_NAME}-${TARGET_UID}"
   EOF
   )
  ```

1. Run the following command to merge the configuration changes into your resource:

   ```
   kubectl patch pgdgroup "${TARGET_NAME}" -n "${TARGET_NS}" --type='merge' -p "$PATCH_CONTENT"
   ```

1. Restart the cluster to bring the agent online.

   - **For CNPG clusters:**

     ```shell
     kubectl cnp restart "$TARGET_NAME"
     ```

   - **For PGD4K clusters:**
     
     Restart each node in the group:
 
    ```shell
    kubectl cnp restart region-a-1
    kubectl cnp restart region-a-2
    kubectl cnp restart region-a-3
    ```

## Configuring security contexts for external Kubernetes clusters

### For standard Kubernetes (KIND, GKE, EKS, AKS)

Configure your `custom-values.yaml` with default secure settings which include user and group IDs:

   ``` Yaml
   securityContext:
     allowPrivilegeEscalation: false
     capabilities:
       drop:
         - ALL
     privileged: false
     readOnlyRootFilesystem: true
     runAsNonRoot: true
     runAsUser: 10000
     runAsGroup: 10000
   
   podSecurityContext:
     seccompProfile:
       type: RuntimeDefault
   ```

### For OpenShift Kubernetes clusters

1. Enable the OpenShift mode to automatically configure security contexts for SCC compatibility.

1. Enable OpenShift support in your `custom-values.yaml` file:

   ```yaml
   openshift:
     enabled: true
   ```

   When the value is set to `openshift.enabled: true`, the chart automatically:
      
   - Omits `runAsUser` and `runAsGroup` from the container security contexts, allowing SCC to assign them.
   - Stores other security settings like `capabilities.drop`, `readOnlyRootFilesystem`, etc.
   - Uses numeric UID 65532 for the OTEL collector that is required for distroless image validation.

1. Alternatively, specify your configuration via the command line using the `-f custom-values.yaml` flag:

   ```Yaml
   In your custom-values.yaml for OpenShift
   securityContext:
     allowPrivilegeEscalation: false
     capabilities:
       drop:
         - ALL
     privileged: false
     readOnlyRootFilesystem: true
     runAsNonRoot: true
   ``

1. Leave the fields below empty as OpenShift will assign them automatically:
     
    ```yaml
    runAsUser:
    runAsGroup:
    ```
1. When deploying to OpenShift, ensure the `fsGroup` field remains undefined to allow the platform to manage volume permissions automatically:

   ```yaml
   podSecurityContext:
     seccompProfile:
       type: RuntimeDefault
     # Leave fsGroup empty for OpenShift
     # fsGroup:
   ```
1. To ensure OpenShift compatibility, configure the OTEL collector subchart with a restricted security context:

   ```yaml
   opentelemetry-collector:
     podSecurityContext:
       seccompProfile:
         type: RuntimeDefault
     securityContext:
       runAsNonRoot: true
       allowPrivilegeEscalation: false
       capabilities:
         drop:
           - ALL
   ```

1. Configure the security context to prohibit root access, manage volume permissions via file system groups, and restrict system calls using the default seccomp profile.

    ```yaml
        securityContext:
          enabled: true
          runAsNonRoot: true
          runAsUser:
          runAsGroup:
          fsGroup:
          seccompProfile:
            type: RuntimeDefault
     ```

   !!! note
   Implementing a standardized security context for your deployments ensures seamless compatibility and enhanced security across diverse container platforms.
   
   - Distroless image support: The UID/GID `65532` is the standard "nonroot" user for distroless images, which lacks a traditional `/etc/passwd` file.
   
   - Numeric validation: Setting a specific numeric ID is required for the `runAsNonRoot: true` directive to verify the user identity correctly.
   
   - OpenShift compatibility: While OpenShift's Privileged SCC allows any UID, using 65532 ensures a consistent, secure identity across both OpenShift and standard Kubernetes environments.
   !!!

   **Key configuration notes:**
   
   - OpenShift ID management: The `runAsUser`, `runAsGroup`, and `fsGroup` fields are left empty to allow OpenShift’s SCC to automatically assign the correct UID and GID ranges for the project.
   
   - Security context activation: The setting `enabled: true` ensures the subchart applies the specified security settings rather than falling back to its internal defaults.
   
   - Non-root enforcement: The `runAsNonRoot: true` flag remains active to guarantee that the container never runs with root privileges, even when IDs are dynamically assigned by the platform.
   
   - Seccomp hardening: Utilizing the `RuntimeDefault` profile ensures the container uses the container runtime's default secure syscall filter.
         
1. Harden the container-level security context by enforcing a read-only filesystem, disabling privilege escalation, and dropping all default Linux capabilities:

   ```yaml
     containerSecurityContext:
       readOnlyRootFilesystem: true
       allowPrivilegeEscalation: false
       capabilities:
         drop:
           - ALL
   ```
   
   Alternatively, you can apply these settings via the command line:
      
   ```shell
   helm upgrade --install beacon-agent-k8s \
     . \
     --namespace upm-beacon \
     --create-namespace \
     # ... your other flags ...
     --set securityContext.runAsUser=null \
     --set securityContext.runAsGroup=null \
     --set podSecurityContext.fsGroup=null \
     --set opentelemetry-collector.securityContext.runAsUser=null \
     --set opentelemetry-collector.securityContext.runAsGroup=null \
     --set kube-state-metrics.podSecurityContext=null
   ```

## Additional requirements for OpenShift

OpenShift assigns a specific UID range to each namespace.  For example:  1000880000-1000889999.

For example: 1000880000-1000889999.

Specifying an explicit runAsUser (like 10000) often falls outside the UID range range, triggering SCC violations. Leave these values empty so OpenShift's admission controller automatically assigns appropriate values from the allowed range.

### Recommended runAsUser configurations by component

This table outlines the required runAsUser settings for EDB Postgres AI agent components to ensure compatibility with different SCC.

| Component               | runAsUser | Reason                                                                                                                                                                                     |
|-------------------------|-----------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| beacon-agent            | Empty     | Uses restricted SCC; OpenShift assigns UID from namespace range                                                                                             |
| kube-state-metrics      | Empty     | Uses restricted SCC; OpenShift assigns UID from namespace range                                                                                             |
| opentelemetry-collector | 65532     | Uses privileged SCC for `hostPath` or `hostPorts`.                                                                                                              |
| opentelemetry-collector | 65532     | Requires explicit numeric UID because `runAsNonRoot: true` cannot verify non-numeric users like `nonroot`. Privileged SCC allows any UID, so `65532` works. |

### Granting SCC for Daemonsets 

To collect node-level metrics and logs on OpenShift, the `opentelemetry-collector` and `prometheus-node-exporter` subcharts require elevated permissions to access `hostPath`, `hostPorts`, and `hostNetwork`. Grant the privileged SCC to their respective service accounts using the commands below.

!!! note
Since OpenShift includes a built-in node-exporter, your deployment may conflict with existing monitoring; consider using the platform's native exporter instead.
!!!

#### Permission setup and deployment

1. Create the namespace (if not already present):

   ```shell
   oc new-project upm-beacon
   ```

1. Grant the privileged SCC:

  ``` shell
   oc adm policy add-scc-to-user privileged \
     system:serviceaccount:upm-beacon:beacon-agent-k8s-opentelemetry-collector
   
   oc adm policy add-scc-to-user privileged \
     system:serviceaccount:upm-beacon:beacon-agent-k8s-prometheus-node-exporter
  ```

1. Deploy or update the Helm chart:
   
   ```shell
   helm upgrade --install beacon-agent . -f custom-values.yaml -n upm-beacon
   ```

1. Restart the Daemonsets:
   ```shell
   kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-opentelemetry-collector-agent
   kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-prometheus-node-exporter
   ```

!!! note 
On OpenShift, `prometheus-node-exporter` may conflict with OpenShift's built-in `node-exporter`. 
For example, if port 9100 is already in use. 
We recommend seting  `additionalCharts.prometheus-node-exporter.install: false` and relying on the `opentelemetry-collector` for container metrics via cadvisor/kubelet scraping.
!!!
