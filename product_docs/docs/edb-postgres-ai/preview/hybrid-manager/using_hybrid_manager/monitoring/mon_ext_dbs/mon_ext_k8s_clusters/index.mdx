# Monitoring external Kubernetes clusters

You can configure the PGAI Agent to run anywhere and activate specific capabilities by assigning it one or more roles: Webhooks, Providers, Dispatchers, or Runners. A single agent can be configured to perform any combination of these functions simultaneously.

This Helm chart deploys the PGAI Agent for [EDB Postgres for Kubernetes](/postgres_for_kubernetes/latest/) or [CloudNativePG](https://cloudnative-pg.io/) clusters, integrating with the HM console to centralize monitoring, metrics collection, and resource management.

## Architecture

The Standalone Kubernetes Agent consists of:

### PGAI Agent

**Key functions:**
  - Resource discovery and monitoring
  - Webhook injection for sidecar containers
  - Metrics collection and forwarding
  - Communication with HCP

### OpenTelemetry Collector (optional):

[Are only the two last features optional or OpenTelemetry as a whole?]: #

**Key functions:**
  - Container metrics (cAdvisor) collection and processing
  - Kubelet metrics
  - Application logs
  - Custom metrics

**Optional:**
- Kube State Metrics: Exposes Kubernetes object state metrics
- Optional: Prometheus node exporter: Exposes node-level metrics

!!! important Important:
When deploying Cloud-native Postgres (CNP) or CloudNativePG (CNPG) clusters or PGDGroups in different namespaces, you must create the required secrets in each namespace separately.
