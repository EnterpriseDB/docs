# Monitoring external Kubernetes clusters

To use the EDB Postgres AI agent to monitor [EDB Postgres® AI for CloudNativePG™](/postgres_for_kubernetes/latest/) or [CloudNativePG™](https://cloudnative-pg.io/) clusters and integrate them with the HM console, you have to use a Helm chart for installation. After configuration, you will get centralized monitoring, metrics collection, and resource management.

You can configure the EDB Postgres AI agent to run anywhere and activate specific capabilities by assigning it one or more roles: Webhooks, Providers, Dispatchers, or Runners. A single agent can be configured to perform any combination of these functions simultaneously. 

## Architecture

The EDB Postgres AI agent used for Kubernetes deployments consists of:

### EDB Postgres AI agent

**Key functions:**
  - Resource discovery and monitoring
  - Webhook injection for sidecar containers
  - Metrics collection and forwarding
  - Communication with HM

### OpenTelemetry Collector (optional):

[Are only the two last features optional or OpenTelemetry as a whole?]: #

**Key functions:**
  - Container metrics (cAdvisor) collection and processing
  - Kubelet metrics
  - Application logs
  - Custom metrics

**Optional:**
- Kube State Metrics: Exposes Kubernetes object state metrics
- Optional: Prometheus node exporter: Exposes node-level metrics

!!! important Important:
When deploying Cloud-native Postgres (CNP) or CloudNativePG (CNPG) clusters or PGDGroups in different namespaces, you must create the required secrets in each namespace separately.
