# Monitoring external Kubernetes clusters
The Standalone Kubernetes Agent collects data from Beacon Estates. You can configure the Kubernetes Agent to run where you want and can decide what capabilities you want to activate. You can configure the Kubernetes agent to run as "Webhooks", "Providers", "Dispatchers", and/or "Runners". Depending on the configuration, a single agent can function as one or more agent types.

The agent deployed by this Helm chart is intended to be installed on an external Kubernetes cluster that aren't managed by HM to manage [EDB Postgres for Kubernetes](/postgres_for_kubernetes/latest/) or [CloudNativePG](https://cloudnative-pg.io/) clusters. It integrates with the HM Console to provide comprehensive monitoring, metrics collection, and resource management.

## Architecture
The Standalone Kubernetes Agent consists of:

### Beacon Agent

**Key functions:**
  - Resource discovery and monitoring
  - Webhook injection for sidecar containers
  - Metrics collection and forwarding
  - Communication with HCP


### OpenTelemetry Collector (optional): Collects and processes:
 
**Key functions:**
  - Container metrics (cAdvisor)
  - Kubelet metrics
  - Application logs
  - Custom metrics
- Optional: Kube State Metrics: Exposes Kubernetes object state metrics
- Optional: Prometheus node exporter: Exposes node-level metrics

!!! important Important:
When deploying Cloud-native Postgres (CNP) or CloudNativePG (CNPG) clusters or PGDGroups in different namespaces, you must create the required secrets in each namespace separately.
