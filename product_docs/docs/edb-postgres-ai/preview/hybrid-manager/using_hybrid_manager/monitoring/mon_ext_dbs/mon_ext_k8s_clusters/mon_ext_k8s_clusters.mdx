## Installation Methods

### Environment-Specific Configuration

This chart is designed to work across different Kubernetes platforms (including OpenShift) with a unified configuration:

| Environment         | Configuration                                   | OpenShift Flag                           |
|---------------------|-------------------------------------------------|------------------------------------------|
| KIND, GKE, EKS, AKS | Uses image defaults (numeric or named)          | openshift.enabled: false (default)       |
| OpenShift (RHOS)    | Compatible via runAsUser: null (SCC assigns ID) | openshift.enabled: false/true (optional) |

Key points:

- **Universal Compatibility:** The chart defaults to runAsUser: null, allowing OpenShift SCCs to assign valid UIDs automatically, while standard Kubernetes falls back to the image's default user.
- **Security Context:** runAsNonRoot strategy varies by component to ensure compatibility with standard K8s validation logic vs OpenShift named-user support.
- **OpenTelemetry collector:** Uses the image default user (nonroot) without explicit numeric IDs to satisfy both strict K8s validation and OpenShift SCCs.

For OpenShift deployments, the default configuration is sufficient.

```
helm upgrade --install beacon-agent-k8s \
  . \
  --create-namespace \
  --namespace upm-beacon \
  --values custom-values.yaml
```

!!!note
Enabling the `openshift.enabled=true` flag is only required when using deprecated templates.
!!!

### Installing the Helm chart with customized values file


1. Install the Helm chart for standard Kubernetes (KIND, GKE, EKS, AKS):
   
      ```
      helm upgrade --install beacon-agent-k8s \
        . \
        --create-namespace \
        --namespace upm-beacon \
        --values custom-values.yaml
      ```

1. Deploy the PGAI agent o cluster B using the OpenShift (RHOS) configuration:
   
      ```
      helm upgrade --install beacon-agent-k8s \
        . \
        --create-namespace \
        --namespace upm-beacon \
        --set openshift.enabled=true \
        --values custom-values.yaml
      
      # Grant SCC if using opentelemetry-collector:
      oc adm policy add-scc-to-user privileged \
        system:serviceaccount:upm-beacon:beacon-agent-k8s-opentelemetry-collector
      kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-opentelemetry-collector-agent
      ```

### Installing the Helm chart with the CLI

    Deploy the Beacon Agent to Cluster B by passing the monitoring components, credentials, and agent configuration as command-line arguments:
      
      ```
      helm upgrade --install beacon-agent-k8s \
        --create-namespace \
        --set additionalCharts.cert-manager.install=true \
        --set additionalCharts.opentelemetry-collector.install=true \
        --set additionalCharts.kube-state-metrics.install=true \
        --set image.imageCredentials.registry="${registry}" \
        --set image.imageCredentials.username="${username}" \
        --set image.imageCredentials.password="${password}" \
        --set image.repository=<repo url> \
        --set image.tag=<image tag> \
        --set otelImage.repository=<repo url> \
        --set otelImage.tag=<image tag> \
        --set opentelemetry-collector.image.repository="${repo url}" \
        --set opentelemetry-collector.image.tag=${image tag} \
        --set agentConfig.beaconServer.host="${server_url}" \
        --set agentConfig.metrics.push.endpoint="${server_url}" \
        --set agentConfig.accessKey="${access_key_agent}" \
        --set agentConfig.projectID="${project_agent}" \
        --set rootCA="${cacert}" \
        --set agentConfig.webhook.podSelector="k8s.enterprisedb.io/podRole=instance" \ # for cnp/pg4k cluster
        # --set agentConfig.webhook.podSelector="cnpg.io/podRole=instance" \ # for cnpg cluster
        .
      ```

## Creating and connecting external CNP clusters

You can create an external CNP cluster and connect it to an instance in HM.

### Requirements

- HM cluster: Make sure your HM instance is active and the PGAI agent server is running and reachable.
- External CNP cluster: Confirm that the target Kubernetes cluster has the CNP or CNPG operator pre-installed.

1. Get your external Kubernetes cluster ready by deploying the CNP or CNPG operator.

   Create a namespace in the CNP cluster if it doesn't exist yet:
   
   ```
   export cluster_ns="test-cnp"
   kubectl create ns "${cluster_ns}"
   ```
   See [Install the appropriate PostgreSQL operator for operator installation](/edb-postgres-ai/preview/hybrid-manager/using_hybrid_manager/monitoring/mon_ext_dbs/mon_ext_k8s_clusters/Installation%20requirements/#installing-the-appropriate-postgresql-operator).

1. Get your access key for the PGAI agent
   In the HM console, create a machine user and assign the Estate Ingest Role to the machine user you created.
   
   !!!note
   Ensure that the access key's respective machine user is assigned the Estate Ingest Role. This is required for both the Beacon Agent and the external CNP cluster.
   !!!

   Set the environment variable in the CNP cluster:

   ```
   export access_key_agent="<ACCESS_KEY_AGENT>"
   export project_agent="<PROJECT_ID_AGENT>"
   ```

1. Get your access key for external PGDGroups
   Create another machine user for the external CNP cluster and assign the Estate Ingest Role to the machine user you created.

   !!!note 
   Ensure that the access key's respective machine user is assigned the Estate Ingest Role. This is required for both the Beacon Agent and the external CNP cluster.
   !!!

   Set the environment variables for the PGDGroups :

   ```
   export access_key_pgdgroup="<ACCESS_KEY_PGDGROUP>"
   export project_pgdgroup="<PROJECT_ID_PGDGROUP>"
   ```

1. Create a secret for external PGDGroups

  With the access key you created in HM, create a secret for the PGDGroup in the CNP cluster:

### Label Descriptions:
   
   These labels are essential for:
   
   - k8s.enterprisedb.io/beacon: Enables beacon monitoring
   - k8s.enterprisedb.io/beacon-project-id: Associates metrics with the correct HCP project
   - k8s.enterprisedb.io/beacon-resource-id: Provides unique cluster identification using Kubernetes UID
   - All three labels will be inherited by pods/PVCs and used for metric labeling

### Security context configuration for standard Kubernetes clusters

By default, the chart uses secure settings with explicit user or group IDs:

   ```
   # In your custom-values.yaml
   securityContext:
     allowPrivilegeEscalation: false
     capabilities:
       drop:
         - ALL
     privileged: false
     readOnlyRootFilesystem: true
     runAsNonRoot: true
     runAsUser: 10000
     runAsGroup: 10000
   
   podSecurityContext:
     seccompProfile:
       type: RuntimeDefault
   ```

### Security context configuration for OpenShift clusters

Enable the OpenShift mode to automatically configure security contexts for SCC compatibility.

In your custom-values Yaml file, run:

   ```
   openshift:
     enabled: true
   ```

When the value is `openshift.enabled: true`, the chart automatically:

- Omits `runAsUser` and `runAsGroup` from the container security contexts, allowing SCC to assign them.
- Stores other security settings like capabilities drop, readOnlyRootFilesystem, etc.
- Uses numeric UID 65532 for the OpenTelemetry collector that is required for distroless image validation.

Alternatively, you can apply `"-f custom-values.yaml"` through the command line:

   ```
   In your custom-values.yaml for OpenShift
   securityContext:
     allowPrivilegeEscalation: false
     capabilities:
       drop:
         - ALL
     privileged: false
     readOnlyRootFilesystem: true
     runAsNonRoot: true
     # Leave these empty - OpenShift will assign them automatically
     runAsUser:
     runAsGroup:
   
   podSecurityContext:
     seccompProfile:
       type: RuntimeDefault
     # Leave fsGroup empty for OpenShift
     # fsGroup:
   
   # Configure subcharts for OpenShift compatibility
   opentelemetry-collector:
     podSecurityContext:
       seccompProfile:
         type: RuntimeDefault
     securityContext:
       runAsNonRoot: true
       allowPrivilegeEscalation: false
       capabilities:
         drop:
           - ALL
       # runAsUser 65532 is the 'nonroot' user in distroless images.
       # Required because runAsNonRoot needs a numeric UID to verify.
       # Privileged SCC (required for hostPath) allows any UID, so this works on OpenShift.
       runAsUser: 65532
       runAsGroup: 65532
   
   kube-state-metrics:
     # Pod-level security context - leave runAsUser/runAsGroup/fsGroup empty for OpenShift
     securityContext:
       enabled: true
       runAsNonRoot: true
       runAsUser:
       runAsGroup:
       fsGroup:
       seccompProfile:
         type: RuntimeDefault
         
     # Container-level security context
     containerSecurityContext:
       readOnlyRootFilesystem: true
       allowPrivilegeEscalation: false
       capabilities:
         drop:
           - ALL
   ```
   
   Or through command line overrides:
   
   ```
   helm upgrade --install beacon-agent-k8s \
     . \
     --namespace upm-beacon \
     --create-namespace \
     # ... your other flags ...
     --set securityContext.runAsUser=null \
     --set securityContext.runAsGroup=null \
     --set podSecurityContext.fsGroup=null \
     --set opentelemetry-collector.securityContext.runAsUser=null \
     --set opentelemetry-collector.securityContext.runAsGroup=null \
     --set kube-state-metrics.podSecurityContext=null
   ```

### Requirements for OpenShift:

OpenShift assigns a specific UID range to each namespace. 

For example: 1000880000-1000889999.

When you specify an explicit runAsUser like 10000, it falls outside this range and causes Security Context Constraints (SCC) violations. By leaving these values empty, OpenShift's admission controller automatically assigns appropriate values from the allowed range.

### runAsUser differences between components:

| Component               | runAsUser | Reason                                                                                                                                                                                     |
|-------------------------|-----------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| beacon-agent            | Empty     | Uses restricted SCC (Security Context Constraints); OpenShift assigns UID from namespace range                                                                                             |
| kube-state-metrics      | Empty     | Uses restricted SCC (Security Context Constraints); OpenShift assigns UID from namespace range                                                                                             |
| opentelemetry-collector | 65532     | Uses privileged SCC (Security Context Constraints) for hostPath or hostPorts.                                                                                                              |
| opentelemetry-collector | 65532     | Requires explicit numeric UID because `runAsNonRoot: true` cannot verify non-numeric users like `nonroot`. Privileged SCC (Security Context Constraints) allows any UID, so `65532` works. |

### OpenShift: Granting SCC (Security Context Constraints) for Daemonsets 

If you enable opentelemetry-collector or prometheus-node-exporter subcharts on OpenShift, these daemonsets require privileged access to collect node-level metricsm including hostPath volumes, hostPorts, hostNetwork.
Grant the privileged SCC to their service accounts:

   ```
   # Required for opentelemetry-collector daemonset (cadvisor/kubelet metrics, logs)
   oc adm policy add-scc-to-user privileged \
     system:serviceaccount:upm-beacon:beacon-agent-k8s-opentelemetry-collector
   
   # Required for prometheus-node-exporter daemonset (node metrics)
   # Note: OpenShift already has its own node-exporter, so this may conflict.
   # Consider disabling prometheus-node-exporter and using OpenShift's built-in monitoring instead.
   oc adm policy add-scc-to-user privileged \
     system:serviceaccount:upm-beacon:beacon-agent-k8s-prometheus-node-exporter
   
   # Restart the daemonsets to pick up the SCC changes
   kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-opentelemetry-collector-agent
   kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-prometheus-node-exporter 
   ```

!!! note 
On OpenShift, prometheus-node-exporter may conflict with OpenShift's built-in node-exporter. 
For example if port 9100 is already in use. 
It's recommended to set additionalCharts.prometheus-node-exporter.install: false and rely on the opentelemetry-collector for container metrics via cadvisor/kubelet scraping.
!!!
