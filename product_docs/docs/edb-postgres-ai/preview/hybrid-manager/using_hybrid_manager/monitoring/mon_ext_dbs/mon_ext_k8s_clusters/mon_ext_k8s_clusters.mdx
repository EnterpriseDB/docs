## Installation Methods

### Environment-Specific Configuration

This chart is designed to work across different Kubernetes platforms (including OpenShift) with a unified configuration:

| Environment         | Configuration                                   | OpenShift Flag                           |
|---------------------|-------------------------------------------------|------------------------------------------|
| KIND, GKE, EKS, AKS | Uses image defaults (numeric or named)          | openshift.enabled: false (default)       |
| OpenShift (RHOS)    | Compatible via runAsUser: null (SCC assigns ID) | openshift.enabled: false/true (optional) |

Key points:

- **Universal Compatibility:** The chart defaults to runAsUser: null, allowing OpenShift SCCs to assign valid UIDs automatically, while standard Kubernetes falls back to the image's default user.
- **Security Context:** runAsNonRoot strategy varies by component to ensure compatibility with standard K8s validation logic vs OpenShift named-user support.
- **OpenTelemetry collector:** Uses the image default user (nonroot) without explicit numeric IDs to satisfy both strict K8s validation and OpenShift SCCs.

For OpenShift deployments, the default configuration is generally sufficient. Explicitly setting openshift.enabled=true is only required if specific deprecated templates need activation:

```
helm upgrade --install beacon-agent-k8s \
  . \
  --create-namespace \
  --namespace upm-beacon \
  --values custom-values.yaml
```

   **Required:** Docker registry credentials for EDB images:

   ```
   image:

     tag: ""
     imageCredentials:
   ```

### Option B: Install with command-line parameters on cluster B

```
helm upgrade --install beacon-agent-k8s \
  --create-namespace \
  --set additionalCharts.cert-manager.install=true \
  --set additionalCharts.opentelemetry-collector.install=true \

  --set image.imageCredentials.username="${username}" \
  --set image.imageCredentials.password="${password}" \

  --set otelImage.repository=<repo url> \
  --set otelImage.tag=<image tag> \
  --set opentelemetry-collector.image.repository="${repo url}" \
  --set opentelemetry-collector.image.tag=${image tag} \
  --set agentConfig.beaconServer.host="${server_url}" \
  --set agentConfig.metrics.push.endpoint="${server_url}" \
  --set agentConfig.accessKey="${access_key_agent}" \
  --set agentConfig.projectID="${project_agent}" \
  --set rootCA="${cacert}" \
  --set agentConfig.webhook.podSelector="k8s.enterprisedb.io/podRole=instance" \ # for cnp/pg4k cluster
  # --set agentConfig.webhook.podSelector="cnpg.io/podRole=instance" \ # for cnpg cluster
  .
```

## Guide for Creating an External CNP Cluster Joining a HCP

This comprehensive guide walks you through the complete process of creating an external CNP cluster that joins a HCP instance.

### Prerequisites for This Guide

Cluster HM (HCP): Your HCP instance with Beacon server running
Cluster CNP (External): External Kubernetes cluster with CNP (or CNPG) operator installed

### Step-by-Step Instructions

Step 1: Prepare the External Cluster
Get your external Kubernetes cluster ready with the CNP (or CNPG) operator deployed.

On the CNP cluster, create a namespace if it doesn't exist:

```
export cluster_ns="test-cnp"
kubectl create ns "${cluster_ns}"
```
See Install the appropriate PostgreSQL operator for operator installation.

Step 2: Get Access Keys for Beacon Agent
Create a machine user with the Estate Ingest Role in your HCP console for the Beacon Agent.

!!!note Note: Ensure that the access key's respective machine user is assigned the Estate Ingest Role. This is required for both the Beacon Agent and the external CNP cluster.
!!!

Set the environment variable on the CNP cluster:

```
export access_key_agent="<ACCESS_KEY_AGENT>"
export project_agent="<PROJECT_ID_AGENT>"
```

### Step 3: Get Access Keys for External PGDGroup
Create another machine user with the Estate Ingest Role for the external CNP cluster.

!!!note 
Ensure that the access key's respective machine user is assigned the Estate Ingest Role. This is required for both the Beacon Agent and the external CNP cluster.
!!!

Set the environment variables in the PGDGroup cluster:

```
export access_key_pgdgroup="<ACCESS_KEY_PGDGROUP>"
export project_pgdgroup="<PROJECT_ID_PGDGROUP>"
```

### Step 4: Create Secret for External PGDGroup

Create secret with access key for the PGDGroup in the CNP cluster:

```
kubectl create secret generic beacon-access-key \
    --from-literal=access-key="${access_key_cluster}" -n "${pgdgroup_ns}"
```

### Step 6: Get Beacon Server URL from HCP

Get the server host and port in the cluster in HM:

```
export server_host=$(kubectl -n upm-beacon get gateways.networking.istio.io beacon-server -oyaml | yq '.spec.servers[0].hosts[0]')
export server_port=$(kubectl -n upm-beacon get gateways.networking.istio.io beacon-server -oyaml | yq '.spec.servers[0].port.number')
export server_url="${server_host}:${server_port}"
```

Transfer the server_url value to Cluster B:
Set the server URL in the CNP cluster with the output of the previous step:

```
export server_url="<server_url>"  #Output of previous step
```

Step 7: Get Image Registry Credentials
On Cluster CNP
For customer deployment, you probably will use the CloudSmith registry

```
 export registry=docker.enterprisedb.com
 export username="${USERNAME}"
 export password="${PASSWORD}"
```

Step 8: Install Standalone Beacon Agent
Follow [step 3](/edb-postgres-ai/preview/hybrid-manager/using_hybrid_manager/monitoring/mon_ext_dbs/mon_ext_k8s_clusters/#step-3-get-access-keys-for-external-pgdgroup) from previous the section.

!!! note 
For PGDGroup clusters, use: --set agentConfig.webhook.podSelector="k8s.enterprisedb.io/podRole=instance"
!!!

Step 10: Add Required Labels for Metrics Collection
After creating the cluster, add the other two required labels to the inheritedMetadata which are required for metrics.

## On Cluster B: Get the PGDGroup UID for the beacon-resource-id

```
export pgdgroup_uid="$(kubectl get pgdgroup external-pgd-group -n test-pgd -o jsonpath='{.metadata.uid}')"


kubectl patch pgdgroup external-pgd-group -n test-pgd --type='merge' -p='
spec:
  inheritedMetadata:
    labels:
      k8s.enterprisedb.io/beacon: enabled
      k8s.enterprisedb.io/beacon-project-id: "'"${project_pgdgroup}"'"
      k8s.enterprisedb.io/beacon-resource-id: "'"${pgdgroup_name}-${pgdgroup_uid}"'"
'
```

Label Descriptions:

These labels are essential for:

- k8s.enterprisedb.io/beacon: Enables beacon monitoring
- k8s.enterprisedb.io/beacon-project-id: Associates metrics with the correct HCP project
- k8s.enterprisedb.io/beacon-resource-id: Provides unique cluster identification using Kubernetes UID
- All three labels will be inherited by pods/PVCs and used for metric labeling

Step 11: Verify in HCP Console
Log in to your HCP console. Check the Estate page - the external PGDGroup should now appear with metrics being collected.

### Security Context Configuration

For Standard Kubernetes Clusters

By default, the chart uses secure settings with explicit user/group IDs:

```
# In your custom-values.yaml
securityContext:
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
  privileged: false
  readOnlyRootFilesystem: true
  runAsNonRoot: true
  runAsUser: 10000
  runAsGroup: 10000

podSecurityContext:
  seccompProfile:
    type: RuntimeDefault
```

For OpenShift Clusters
Enable OpenShift mode to automatically configure security contexts for SCC compatibility.

In your custom-values Yaml file, run:

```
openshift:
  enabled: true
```

When openshift.enabled: true, the chart automatically:

- Omits runAsUser and runAsGroup from container security contexts (allowing SCC to assign them)
- Keeps other security settings (capabilities drop, readOnlyRootFilesystem, etc.)
- Uses numeric UID 65532 for the OpenTelemetry collector (required for distroless image validation)

Alternatively, set via command line with `"-f custom-values.yaml"`:

```
In your custom-values.yaml for OpenShift
securityContext:
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
  privileged: false
  readOnlyRootFilesystem: true
  runAsNonRoot: true
  # Leave these empty - OpenShift will assign them automatically
  runAsUser:
  runAsGroup:

podSecurityContext:
  seccompProfile:
    type: RuntimeDefault
  # Leave fsGroup empty for OpenShift
  # fsGroup:

# Configure subcharts for OpenShift compatibility
opentelemetry-collector:
  podSecurityContext:
    seccompProfile:
      type: RuntimeDefault
  securityContext:
    runAsNonRoot: true
    allowPrivilegeEscalation: false
    capabilities:
      drop:
        - ALL
    # runAsUser 65532 is the 'nonroot' user in distroless images.
    # Required because runAsNonRoot needs a numeric UID to verify.
    # Privileged SCC (required for hostPath) allows any UID, so this works on OpenShift.
    runAsUser: 65532
    runAsGroup: 65532

kube-state-metrics:
  # Pod-level security context - leave runAsUser/runAsGroup/fsGroup empty for OpenShift
  securityContext:
    enabled: true
    runAsNonRoot: true
    runAsUser:
    runAsGroup:
    fsGroup:
    seccompProfile:
      type: RuntimeDefault
  # Container-level security context
  containerSecurityContext:
    readOnlyRootFilesystem: true
    allowPrivilegeEscalation: false
    capabilities:
      drop:
        - ALL
```

Or via command line overrides:

```
helm upgrade --install beacon-agent-k8s \
  . \
  --namespace upm-beacon \
  --create-namespace \
  # ... your other flags ...
  --set securityContext.runAsUser=null \
  --set securityContext.runAsGroup=null \
  --set podSecurityContext.fsGroup=null \
  --set opentelemetry-collector.securityContext.runAsUser=null \
  --set opentelemetry-collector.securityContext.runAsGroup=null \
  --set kube-state-metrics.podSecurityContext=null
```

### Why this is required for OpenShift:

OpenShift assigns a specific UID range (e.g., 1000880000-1000889999) to each namespace. When you specify an explicit runAsUser (like 10000), it falls outside this range and causes SCC violations. By leaving these values empty, OpenShift's admission controller automatically assigns appropriate values from the allowed range.

Note on runAsUser differences between components:

| Component               | runAsUser | Reason                                                                                                                                                                                              |
|-------------------------|-----------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| beacon-agent            | Empty     | Uses restricted SCC; OpenShift assigns UID from namespace range                                                                                                                                     |
| kube-state-metrics      | Empty     | Uses restricted SCC; OpenShift assigns UID from namespace range                                                                                                                                     |
| opentelemetry-collector | 65532     | Uses privileged SCC (for hostPath/hostPorts); requires explicit numeric UID because runAsNonRoot: true cannot verify non-numeric users like nonroot. Privileged SCC allows any UID, so 65532 works. |

### OpenShift: Granting SCC for Daemonsets (Required for Metrics Collection)

If you enable opentelemetry-collector or prometheus-node-exporter subcharts on OpenShift, these daemonsets require privileged access to collect node-level metrics (hostPath volumes, hostPorts, hostNetwork). You must grant the privileged SCC to their service accounts:

```
# Required for opentelemetry-collector daemonset (cadvisor/kubelet metrics, logs)
oc adm policy add-scc-to-user privileged \
  system:serviceaccount:upm-beacon:beacon-agent-k8s-opentelemetry-collector

# Required for prometheus-node-exporter daemonset (node metrics)
# Note: OpenShift already has its own node-exporter, so this may conflict.
# Consider disabling prometheus-node-exporter and using OpenShift's built-in monitoring instead.
oc adm policy add-scc-to-user privileged \
  system:serviceaccount:upm-beacon:beacon-agent-k8s-prometheus-node-exporter

# Restart the daemonsets to pick up the SCC changes
kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-opentelemetry-collector-agent
kubectl -n upm-beacon rollout restart daemonset beacon-agent-k8s-prometheus-node-exporter
```

!!! note Note: On OpenShift, prometheus-node-exporter may conflict with OpenShift's built-in node-exporter (port 9100 already in use). It's recommended to set additionalCharts.prometheus-node-exporter.install: false and rely on the opentelemetry-collector for container metrics via cadvisor/kubelet scraping.
!!!
