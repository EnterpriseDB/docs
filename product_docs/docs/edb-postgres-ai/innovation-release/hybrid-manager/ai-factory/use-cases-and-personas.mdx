## Overview

AI Factory on Hybrid Manager enables diverse organizational use cases through sovereign AI infrastructure. Organizations leverage these capabilities to build intelligent applications while maintaining complete control over data and models within their Kubernetes environments.

## Key Use Cases

### Internal Knowledge Management

Organizations deploy AI assistants that provide employees with natural language access to internal documentation, policies, and institutional knowledge.

**Implementation Pattern**

Financial services firms index compliance documentation, regulatory filings, and internal procedures into knowledge bases. Employees query these systems using conversational interfaces, receiving accurate answers with source citations while maintaining data sovereignty.

**Technical Components**

-   Knowledge bases indexing SharePoint, Confluence, and database content
-   LLM endpoints for query understanding and response generation
-   Retrieval systems with role-based filtering for access control

See [Knowledge Base Creation](/edb-postgres-ai/1.3/hybrid-manager/ai-factory/gen-ai/create-knowledge-base/) for implementation guidance.

### Customer Service Automation

AI-powered customer support systems handle routine inquiries while escalating complex issues to human agents.

**Implementation Pattern**

E-commerce companies deploy assistants that access order systems, product catalogs, and return policies through secure APIs. These assistants resolve 60-70% of customer inquiries automatically while maintaining full audit trails for quality assurance.

**Technical Components**

-   REST API tools for order management system integration
-   Knowledge bases containing product documentation and FAQs
-   Thread tracking for conversation history and escalation workflows

See [Customer Service Agent Quickstart](/edb-postgres-ai/1.3/hybrid-manager/ai-factory/gen-ai/quickstart-customer-service-agent-ui/) for a working example.

### Enterprise RAG Systems

Retrieval-augmented generation applications ground LLM responses in organizational data, reducing hallucination while maintaining accuracy.

**Implementation Pattern**

Healthcare organizations implement RAG systems over clinical guidelines, research papers, and treatment protocols. Medical professionals receive evidence-based recommendations with citations to source materials, supporting clinical decision-making while ensuring traceability.

**Technical Components**

-   Vector embeddings stored in PostgreSQL with pgvector
-   Hybrid search combining semantic and keyword matching
-   Reranking models for result optimization

See [RAG Pipeline Documentation](/edb-postgres-ai/1.3/ai-factory/pipeline/knowledge_base/) for detailed architecture.

### AI-Enabled APIs

Organizations expose AI capabilities through secure APIs for integration with existing applications.

**Implementation Pattern**

Software companies provide intelligent features within their products by deploying model endpoints accessible through standard REST interfaces. These APIs handle tasks like document summarization, sentiment analysis, and content generation while maintaining predictable performance characteristics.

**Technical Components**

-   InferenceServices exposing OpenAI-compatible endpoints
-   API gateways with authentication and rate limiting
-   Load balancing across model replicas for high availability

See [Model Serving Guide](/edb-postgres-ai/1.3/ai-factory/model/serving/) for deployment instructions.

## User Personas

### Platform Engineers

Platform engineers manage AI Factory infrastructure, ensuring reliable model serving and optimal resource utilization.

**Primary Responsibilities**

-   Configure GPU nodes and resource allocation
-   Deploy and monitor InferenceServices
-   Manage model library and registry connections
-   Implement security policies and network configurations

**Key Workflows**

1.  Provision GPU resources following [GPU Setup Guide](/edb-postgres-ai/1.3/ai-factory/model/setup-gpu)
2.  Configure model registries per [Model Library Documentation](/edb-postgres-ai/1.3/ai-factory/model/library)
3.  Monitor inference performance using [Observability Tools](/edb-postgres-ai/1.3/ai-factory/model/observability)

### Data Scientists

Data scientists leverage AI Factory to deploy models and build intelligent applications without managing infrastructure complexity.

**Primary Responsibilities**

-   Create and optimize knowledge bases
-   Configure retrieval strategies for RAG applications
-   Evaluate model performance and accuracy
-   Design assistant behaviors and tool integrations

**Key Workflows**

1.  Build knowledge bases following [Knowledge Base Management](/edb-postgres-ai/1.3/hybrid-manager/ai-factory/gen-ai/manage-knowledge-base/)
2.  Configure retrievers using [Retriever Creation Guide](/edb-postgres-ai/1.3/hybrid-manager/ai-factory/gen-ai/create-retriever/)
3.  Deploy assistants through [Assistant Configuration](/edb-postgres-ai/1.3/hybrid-manager/ai-factory/gen-ai/create-assistant/)

### Production Operator

Production Operators take a tested flow from a developer and run it as a deployed, production-grade service in a Hybrid Manager project. Their work begins where the developer's ends: the operator doesn't build flows, but owns deploying the flow, supplying its production credentials and parameters, and operating it afterwards. The deployment project can be one the developer has no access to, giving a clean separation between development and production.

**What they receive from the developer**

Depending on whether development and production share a cluster, the operator receives one of two things:

- **Same cluster:** access to a *published flow*. The developer publishes the flow as a bundle, includes the operator's project in its access list, and notifies the operator. No file changes hands.
- **Separate clusters:** the flow exported as a `.json` file, plus the list of resources it depends on (knowledge bases, model clusters, Postgres databases, and the machine-user roles required). The operator imports it into the production cluster and publishes it there. See [Share a Langflow flow between Hybrid Manager instances](./langflow/sharing-flows-between-instances).

In both cases the published bundle carries the flow's structure and parameter names but no credential values; the operator supplies those at deploy time.

**Primary tooling**

Production Operators work primarily through the **Hybrid Manager console** (HM UI). They don't need direct `kubectl` access for routine operations, though it's useful for advanced troubleshooting. Key HM UI areas they use:

- **Launchpad → Langflow** — to import and publish flows (separate-cluster case).
- **Asset Library → Published Flows** — to find published bundles and deploy them.
- **Estate → Flows** — to manage deployed flows: replicas, endpoints, and access.
- **Estate → InferenceServices** — to adjust inference service instances and monitor model server health.
- **Project Settings → Secrets** — to create and rotate credentials.

**Key workflow**

The flow reaches the deployment project by one of two routes, then follows the same deploy path:

1. Get the flow into the target:
   - *Same cluster:* the developer publishes the flow with the deployment project in its access list and notifies the operator.
   - *Separate clusters:* the developer exports the flow as JSON; the operator imports it into the production cluster and publishes it. See [Share a Langflow flow between Hybrid Manager instances](./langflow/sharing-flows-between-instances).
1. Deploy the published flow to the deployment project, supplying credentials, parameters, and replica count at deploy time. The deployment project need not be the one the flow was published to, so the developer can have zero access to production. See [Flow deployment](./langflow/flow-deployment).
1. Set up alerting for the deployed flow and its dependencies — see [Observability](./observability).

Project-level isolation, per-flow Kubernetes secrets, and network policy for a deployed flow are provided by the deployment itself, not configured by hand. For organisation-wide controls (external TLS, secret rotation, air-gapped egress), see [Security and compliance](./security-and-compliance).

## Implementation Considerations

### Scalability Planning

Different use cases require varying levels of scale:

-   **High-volume customer service**: Multiple model replicas with auto-scaling
-   **Internal knowledge systems**: Moderate scaling with caching optimization
-   **Specialized assistants**: Single instances with dedicated resources

### Security Requirements

Use cases determine security configurations:

-   **Public-facing applications**: External ingress with authentication
-   **Internal systems**: Cluster-local access with RBAC
-   **Regulated industries**: Enhanced audit logging and encryption

### Performance Optimization

Optimize based on use case characteristics:

-   **Real-time applications**: Low-latency model serving with edge caching
-   **Batch processing**: Higher throughput with larger batch sizes
-   **Interactive assistants**: Balanced latency and throughput
