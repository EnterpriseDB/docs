Pipeline Designer uses a two-phase wizard to create pipelines. The first phase configures the pipeline's target cluster, database, and processing options. The second phase defines the source table, processing steps, and destination.

## Phase 1: Pipeline options

1.  Navigate to **AI Factory > Pipeline Designer** in your Hybrid Manager project.
2.  Click **Create Pipeline**.
3.  Enter a **pipeline name**. Names must be lowercase, start with a letter, contain only letters, digits, and underscores, and be at most 46 characters long.
4.  Select a **cluster** from the estate. The dropdown shows all eligible clusters in your project:
    -   **Appliance clusters** (PSR or PGD/AHA): For PGD clusters, you also select a data group. Witness groups are excluded. Only the write leader node is eligible for pipeline execution.
    -   **Self-managed PostgreSQL instances**: Registered on-premises or external PostgreSQL instances with a beacon-agent connected.
    -   Only clusters running PostgreSQL 16 or later appear. Replica clusters are excluded.
5.  Select a **database** on the chosen cluster.
6.  Configure the **processing type**:
    -   **Disabled** (default): No automatic processing. You trigger execution manually.
    -   **Live**: Real-time processing triggered by row changes. See [Concepts: Processing modes](concepts#live).
    -   **Background**: Asynchronous batch processing at a configurable interval.
7.  If you selected Background, configure the **batch size** (default: 100) and **sync interval** (default: 30 seconds).
8.  Click **Next** to proceed to step configuration.

## Phase 2: Step configuration

The step designer shows a visual pipeline with two fixed endpoints: a **Table Source** card on the left and a **Destination Table** card on the right. You build your pipeline by adding steps between these endpoints.

### Configuring the source

1.  Click the **Table Source** card.
2.  Select the **schema** containing your source table.
3.  Select the **table** to use as the pipeline source.
4.  Select the **key column**: a column with unique values that identifies each row (typically a primary key or unique identifier).
5.  Select the **data column**: the column containing the content to process (text, HTML, PDF bytes, or image bytes, depending on your first step).

!!! Note
    The `visual_pipeline_user` role must have SELECT access to the source table. If you see a permission error during pipeline creation, see [VPU and permissions](vpu-and-permissions) for the required grants.

### Adding steps

1.  Click the **+** button on the arrow connecting two nodes in the pipeline.
2.  Select a step type from the dropdown. Available step types depend on the position in the pipeline and the output type of the preceding step:
    -   **First step**: ParsePDF, PerformOCR, PdfToImage, ParseHTML, ChunkText, SummarizeText, or KnowledgeBase (if it is the only step).
    -   **Middle steps**: ParseHTML, ChunkText, SummarizeText (provided the preceding step produces text output), or PerformOCR (provided the preceding step produces binary image output).
    -   **Last step**: KnowledgeBase (if you want vector storage), or any compatible text processing step.
3.  Configure the step's parameters:
    -   **ChunkText**: Set the desired chunk length (required) and optionally the maximum length.
    -   **SummarizeText**: Select a text completion model from the model picker.
    -   **PerformOCR**: Select an OCR model from the model picker.
    -   **KnowledgeBase**: Select an embedding model, choose the data format (Text or Image), and optionally configure the distance operator and vector index type.
4.  Repeat to add more steps, up to the 10-step maximum.

### Step ordering

Pipeline Designer validates step ordering in real time. Invalid step placements are flagged visually. The rules are:

-   Steps consuming raw document input (ParsePDF, PdfToImage) must be the first step. PerformOCR accepts binary image input and can appear first or after a step that produces image output (such as PdfToImage).
-   KnowledgeBase must be the last step.
-   Each step's output type must match the next step's expected input type.

You can reorder steps using the up and down controls on each step card. Steps can also be removed by clicking the delete control.

### Selecting models

Steps that require a model (KnowledgeBase, SummarizeText, PerformOCR) display a model picker that lists models available on the selected database. Models are discovered from the AIDB model registry on the target database, which includes:

-   **HM-hosted models**: Models deployed through [Model Serving](../models) and synchronized into AIDB via KServe.
-   **External models**: Models registered through [external inference services](external-inference-services) (OpenAI, AWS Bedrock, and others).
-   **On-database models**: Built-in AIDB models running locally within PostgreSQL.

The model picker filters models by function type (embedding, text completion, OCR) based on the step being configured.

!!! Important
    When creating pipelines on self-managed PostgreSQL instances or secondary HM locations, HM-hosted KServe models may not be reachable from those clusters. Use externally registered models (such as OpenAI) or on-database models for remote clusters. See [Known issues: Inference routing on remote clusters](known-issues#inference-routing-on-remote-clusters) for details.

### Deploying the pipeline

1.  Verify that all steps are configured and the pipeline is valid (no red indicators).
2.  The **destination table** name defaults to `pipeline_{your_pipeline_name}`. For pipelines ending with a KnowledgeBase step, this becomes the vector storage table.
3.  Click **Deploy**.

Pipeline Designer creates the pipeline, associated database objects (destination tables, triggers or background processing artifacts), and begins monitoring its status. The pipeline appears in the pipeline list with a "Creating" status that transitions to its operational status once setup is complete.

## Creating a knowledge base pipeline

A knowledge base pipeline is any pipeline whose last step is KnowledgeBase. When you deploy such a pipeline:

1.  AIDB creates a vector table to store embeddings.
2.  A knowledge base entry is registered in AIDB's knowledge base registry.
3.  The knowledge base becomes queryable through the Pipeline Designer's [Knowledge Bases](knowledge-bases) view.

To create a simple knowledge base pipeline:

1.  Follow the wizard above.
2.  Add a **ChunkText** step to split your source text into segments.
3.  Add a **KnowledgeBase** step as the final step.
4.  In the KnowledgeBase step, select a **text embedding model** and set the data format to **Text**.
5.  Deploy the pipeline.

For PDF or image sources, add a ParsePDF or PerformOCR step before ChunkText to convert binary data to text.

## After creation

Once a pipeline is created:

-   **Step structure is fixed.** You cannot add, remove, or reorder steps. To change the pipeline structure, delete it and create a new one.
-   **Processing parameters are editable.** You can change the processing type, batch size, and sync interval at any time from the pipeline detail page.
-   **Manual execution is always available.** Regardless of the processing mode, you can trigger a manual run from the pipeline detail page.

## Langflow integration

If you are using the [Langflow-based Gen AI builder](/edb-postgres-ai/current/ai-factory/langflow/), knowledge bases created through Pipeline Designer are accessible as data sources within Langflow flows. The Langflow KB component queries AIDB knowledge bases using the same vector retrieval mechanism as Pipeline Designer's built-in query tool.
