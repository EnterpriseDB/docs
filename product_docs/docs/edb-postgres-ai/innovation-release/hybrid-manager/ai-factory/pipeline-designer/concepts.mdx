Pipeline Designer is a visual interface for [AIDB pipelines](/edb-postgres-ai/current/ai-factory/pipeline/). The concepts on this page focus on how the designer presents and constrains AIDB functionality; for the underlying pipeline, knowledge base, and model concepts, see the [AIDB documentation](/edb-postgres-ai/current/ai-factory/pipeline/).

## Pipeline structure in the designer

A pipeline is a sequence of one to ten processing steps that transform data from a source table into a destination. You configure the source, steps, and destination through the [pipeline creation wizard](creating-pipelines).

Pipeline names must be lowercase, start with a letter, and contain only letters, digits, and underscores. The maximum length is 46 characters (AIDB reserves up to 17 characters of the PostgreSQL identifier limit for internal suffixes).

## Step types

Pipeline Designer exposes the AIDB [data preparation operations](/edb-postgres-ai/current/ai-factory/pipeline/preparers/) and [knowledge base](/edb-postgres-ai/current/ai-factory/pipeline/knowledge_base/) functionality as pipeline steps. For detailed descriptions of what each operation does, its parameters, and its input/output types, see the [AIDB preparers primitives documentation](/edb-postgres-ai/current/ai-factory/pipeline/preparers/primitives).

The available step types are: ParsePDF, PerformOCR, ParseHTML, ChunkText, SummarizeText, KnowledgeBase, and PdfToImage.

!!! Note "SQL enum casing"
    The step type names shown above use the display names from the Pipeline Designer UI. The corresponding AIDB SQL enum values use different casing: `ParsePdf`, `PerformOcr`, `ParseHtml`, `ChunkText`, `SummarizeText`, `KnowledgeBase`, `PdfToImage`. If you reference step types in SQL calls to `aidb.create_pipeline()`, use the SQL enum casing.

## Designing multi-step pipelines

Pipelines can chain up to ten steps. The designer enforces the following ordering and compatibility constraints when you build a multi-step pipeline:

**Ordering constraints enforced by the designer:**
-   Steps that consume raw document data (ParsePDF, PdfToImage) must be placed first. PerformOCR also accepts binary input but can appear later in the pipeline (for example, after PdfToImage).
-   The KnowledgeBase step, if present, must be placed last.
-   Middle steps (ChunkText, SummarizeText, ParseHTML) can appear in any order between the first and last positions.

**Type compatibility:**
Each step produces output of a specific type (text or bytes), and the next step must accept that type as input. The designer validates this automatically and prevents you from deploying incompatible step sequences. For example, you cannot place a ChunkText step (which expects text input) immediately after a PdfToImage step (which produces bytes output) without an intermediate OCR step to convert bytes to text.

**Common multi-step patterns:**

| Pattern | Steps | Use case |
| ------- | ----- | -------- |
| PDF to knowledge base | ParsePDF, ChunkText, KnowledgeBase | Index PDF documents for semantic search |
| Image OCR to knowledge base | PerformOCR, ChunkText, KnowledgeBase | Extract and index text from scanned documents |
| HTML processing | ParseHTML, ChunkText, SummarizeText | Clean, chunk, and summarize web content |
| Full document pipeline | ParsePDF, ChunkText, SummarizeText, KnowledgeBase | Parse, chunk, summarize, and index documents |

**Step limit**: A pipeline supports a maximum of 10 steps. Steps cannot be added, removed, or reordered after the pipeline is created. To change the step structure, delete the pipeline and create a new one.

## Processing modes

Pipeline Designer lets you select an [AIDB auto-processing mode](/edb-postgres-ai/current/ai-factory/pipeline/capabilities/auto-processing) during pipeline creation. You can change the mode later without recreating the pipeline. The three modes are:

-   **Disabled** (default): No automatic processing. You trigger execution manually from the designer UI or through the AIDB API.
-   **Live**: Real-time processing using database triggers. Each INSERT, UPDATE, or DELETE on the source table triggers pipeline processing within the same transaction. This adds latency to application writes but ensures immediate processing.
-   **Background**: Asynchronous batch processing using AIDB's background worker. Application writes are not blocked. You configure the batch size (default: 100) and sync interval (default: 30 seconds, range: 1 second to 2 days) in the designer.

Background mode is the recommended choice for most workloads. For a detailed explanation of how each mode works, including change detection and batch processing behavior, see the [AIDB auto-processing documentation](/edb-postgres-ai/current/ai-factory/pipeline/capabilities/auto-processing).

## Data flow and intermediate results

When a pipeline executes, data flows through AIDB's internal envelope from one step to the next. For details on how AIDB handles data lineage, chunking identifiers, and step-to-step data transfer, see the [AIDB pipelines documentation](/edb-postgres-ai/current/ai-factory/pipeline/).

In Pipeline Designer, each intermediate step has a **Save Step Results** toggle (enabled by default) that persists the step's output to a destination table. This lets you inspect intermediate results at each stage of the pipeline, not just the final output. The toggle is hidden on the last step, whose output is always written to the pipeline's destination.
