This page describes how to move a Langflow flow between Hybrid Manager (HM) instances: a developer exports the flow as a portable artifact, and an operator imports and reconfigures it in another HM instance. For example, you might move a flow from a development HM to a production HM managed by a Production Operator.

This shares a flow between instances. It does not, on its own, give you a production-grade deployment: after import the flow still runs in the shared Langflow flow builder, configured against the target instance's resources. To run a flow as an isolated, independently scalable, callable service, publish and deploy it. See [Flow deployment](./flow-deployment).

## Before you begin

Make sure the following are in place before you start.

**Developer environment**

- A working Langflow flow built with EDB components, tested in a development HM project.

- Langflow access via Launchpad in the HM console.

- An access key (machine user) with the roles required by the components in the flow:
  
  - **AI Model Manager** (organization level) — required for EDB Embedding and EDB Model Server components.
  
  - **Project Viewer** (project level) — required for EDB Knowledge Base and EDB Database components.

**Target environment**

- A separate HM project in the target instance (for example, a production project).

- GPU nodes deployed if the flow uses model serving components. See [Set up GPU nodes](../gpu-recommended) for guidance.

- A machine user in the target project with the same role assignments as above.

- Any Postgres clusters and knowledge bases that the flow depends on, recreated or already present in the target project.

- Model clusters for any EDB Embedding or EDB Model Server components, deployed in the target project.

## Stage 1: Export the flow (developer)

Langflow flows are stored as JSON. Exporting a flow produces a self-contained file that captures the flow's structure, component configuration, and connections — but not environment-specific secrets or endpoint references. You reconfigure those in stage 2.

1. Open Langflow via **Launchpad** in the development HM console.

1. Open the flow you want to move.

1. Select the **Options** menu (⋮) in the top toolbar, then select **Export**.

1. Save the exported `.json` file to a location accessible to the operator who will import it.

!!!Note
The exported file doesn't contain HM Machine User Keys, passwords, or Global Variable values. Those are scoped to the Langflow instance and must be reconfigured in the target instance.
!!!

## Stage 2: Import and configure in the target instance (operator)

### Import the flow

1. Log into the **target** HM console and open Langflow via Launchpad.

1. On the Langflow home screen, select **New Flow**, then **Import from file**.

1. Select the `.json` file exported by the developer.

1. Verify that the flow loads correctly — all components should appear with their connections intact. Components that reference environment-specific values will show empty or invalid fields until you reconfigure them in the next step.

### Set up Global Variables

EDB components read credentials and connection details from Langflow Global Variables. Set these up before configuring individual components.

1. In Langflow, go to **Profile** → **Settings** → **Global Variables**.

1. Create variables for the following, using the target instance's values:

   | Variable            | Description                                 |
   |---------------------|---------------------------------------------|
   | HM Machine User Key | Access key for the target machine user.     |
   | Database user       | Postgres user for the target cluster.       |
   | Database password   | Password for the target Postgres user.      |

   See [Configure Global Variables](https://docs.langflow.org/configuration-global-variables#create-a-global-variable) in the Langflow documentation for instructions.

### Reconfigure EDB components

Each EDB component in the flow must point to the target instance's resources. Work through each component in turn.

**EDB Knowledge Base component**

1. In the component, set **HM Machine User Key** to the Global Variable you created.

1. Under **Hybrid Manager Project**, select the target project.

1. Under **Hybrid Manager Database**, select the target cluster that contains the knowledge base.

1. Set **Database Connection Type**, **User for the database**, and **Password for database** to the target instance's values.

1. Under **Database Name**, select the database where the knowledge base was created in the target instance.

**EDB Database component**

1. Set **HM Machine User Key** to the Global Variable you created.

1. Select the target project and cluster.

1. Set the connection type, user, password, and database name to the target instance's values.

**EDB Embeddings component**

1. Set **HM Machine User Key** to the Global Variable you created.

1. Under **Hybrid Manager Model Server Cluster Instance**, select the target model cluster.

**EDB Model Server component**

1. Set **HM Machine User Key** to the Global Variable you created.

1. Confirm the **Input** field and output configuration match your expected behavior.

!!!Tip
If a component can't resolve the project because the portal domain defaults to the API endpoint, select **Controls** at the top of the EDB Database or Knowledge Base component and set the **HM Portal URL** to `http://istio-internalgateway.istio-system.svc.cluster.local:7080`.
!!!

### Validate the imported flow

Before relying on the imported flow, run a functional check:

1. Select the **Playground** icon in the top toolbar to open the interactive test panel.

1. Send a representative input through the flow and verify the output is correct.

1. Check that all components return results without errors — look for green status indicators on each component after running.

1. If any component shows an error, select it and review the configuration against the steps above.

---

## What's next

Importing a flow into another HM instance leaves it running in that instance's shared Langflow flow builder, available only to the user who imported it. To run it as a production-grade service, with project-level isolation, its own HTTP and MCP endpoints, independent scaling, and credentials supplied at deploy time, publish and deploy the flow. See [Flow deployment](./flow-deployment), which covers high availability (replicas), observability, and access control for deployed flows.
