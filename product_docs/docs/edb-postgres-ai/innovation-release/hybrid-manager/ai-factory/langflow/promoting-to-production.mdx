Moving a Langflow-based AI application from development to production involves two stages: 
- the developer exports the flow as a portable artifact, and
- a production operator imports, reconfigures, and deploys it in the production Hybrid Manager (HM) project. 
This page walks through both stages.

## Before you begin

Make sure the following are in place before starting the promotion workflow.

**Developer environment**

- A working Langflow flow built with EDB components, tested in a development HM project.

- Langflow access via Launchpad in the HM console.

- An access key (machine user) with the roles required by the components in the flow:
  
  - **AI Model Manager** (organization level) — required for EDB Embedding and EDB Model Server components.
  
  - **Project Viewer** (project level) — required for EDB Knowledge Base and EDB Database components.

**Production environment**

- A separate HM project configured for production use.

- GPU nodes deployed if the flow uses model serving components. See [Set up GPU nodes](../gpu-recommended) for guidance.

- A production machine user with the same role assignments as above, scoped to the production project.

- Any Postgres clusters and knowledge bases that the flow depends on, recreated or already present in the production project.

- Model clusters for any EDB Embedding or EDB Model Server components, deployed in the production project.

---

## Stage 1: Export the flow (developer)

Langflow flows are stored as JSON. Exporting a flow produces a self-contained file that captures the flow's structure, component configuration, and connections — but not environment-specific secrets or endpoint references. You reconfigure those in stage 2.

1. Open Langflow via **Launchpad** in the development HM console.

1. Open the flow you want to promote.

1. Select the **Options** menu (⋮) in the top toolbar, then select **Export**.

1. Save the exported `.json` file to a location accessible to the production operator.

!!!Note
The exported file doesn't contain HM Machine User Keys, passwords, or Global Variable values. Those are scoped to the Langflow instance and must be reconfigured in the production environment.
!!!

---

## Stage 2: Import and configure in production (production operator)

### Import the flow

1. Log into the **production** HM console and open Langflow via Launchpad.

1. On the Langflow home screen, select **New Flow**, then **Import from file**.

1. Select the `.json` file exported by the developer.

1. Verify that the flow loads correctly — all components should appear with their connections intact. Components that reference environment-specific values will show empty or invalid fields until you reconfigure them in the next step.

### Set up Global Variables

EDB components read credentials and connection details from Langflow Global Variables. Set these up before configuring individual components.

1. In Langflow, go to **Profile** → **Settings** → **Global Variables**.

1. Create variables for the following, using production values:

   | Variable            | Description                                 |
   |---------------------|---------------------------------------------|
   | HM Machine User Key | Access key for the production machine user. |
   | Database user       | Postgres user for the production cluster.   |
   | Database password   | Password for the production Postgres user.  |

   See [Configure Global Variables](https://docs.langflow.org/configuration-global-variables#create-a-global-variable) in the Langflow documentation for instructions.

### Reconfigure EDB components

Each EDB component in the flow must point to production resources. Work through each component in turn.

**EDB Knowledge Base component**

1. In the component, set **HM Machine User Key** to the Global Variable you created.

1. Under **Hybrid Manager Project**, select the production project.

1. Under **Hybrid Manager Database**, select the production cluster that contains the knowledge base.

1. Set **Database Connection Type**, **User for the database**, and **Password for database** to your production values.

1. Under **Database Name**, select the database where the production knowledge base was created.

**EDB Database component**

1. Set **HM Machine User Key** to the production Global Variable.

1. Select the production project and cluster.

1. Set the connection type, user, password, and database name to production values.

**EDB Embeddings component**

1. Set **HM Machine User Key** to the production Global Variable.

1. Under **Hybrid Manager Model Server Cluster Instance**, select the production model cluster.

**EDB Model Server component**

1. Set **HM Machine User Key** to the production Global Variable.

1. Confirm the **Input** field and output configuration match your expected production behavior.

!!!Tip
If a component can't resolve the project because the portal domain defaults to the API endpoint, select **Controls** at the top of the EDB Database or Knowledge Base component and set the **HM Portal URL** to `http://istio-internalgateway.istio-system.svc.cluster.local:7080`.
!!!

### Validate before deploying

Before treating the flow as production-ready, run a functional check:

1. Select the **Playground** icon in the top toolbar to open the interactive test panel.

1. Send a representative input through the flow and verify the output is correct.

1. Check that all components return results without errors — look for green status indicators on each component after running.

1. If any component shows an error, select it and review the configuration against the steps above.

---

## What's next

Once the flow is running correctly in production, consider the following:

- **High availability and load balancing** — For production workloads, configure replica counts and load balancing for your Gen AI Builder components. See [Deploy AI Factory with Hybrid Manager UI](../deploy-with-hm-ui) for deployment options.

- **Observability** — Set up monitoring and alerting for the flow's dependent services (model clusters, Postgres clusters). See [Observability](../observability) for guidance.

- **Access control** — Review machine user roles and project membership to ensure production access is limited to authorised users only.
