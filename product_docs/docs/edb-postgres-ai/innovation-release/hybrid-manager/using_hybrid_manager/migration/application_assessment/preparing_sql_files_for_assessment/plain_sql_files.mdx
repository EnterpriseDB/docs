If you already have SQL scripts — for example, scripts used for database initialization, maintenance jobs, or SQL embedded in source code files — you can upload them directly as plain `.sql` files without any Oracle-side configuration.

## Format requirements

Plain SQL files must follow these formatting rules for Hybrid Manager (HM) to parse them correctly:

-   Every SQL statement must end with a semicolon (`;`).
-   Every PL/SQL block (anonymous blocks, stored procedures, and triggers) must end with a forward slash (`/`) on its own line, preceded by a blank line.
-   Files must use UTF-8 encoding.

**Example:**

```sql
SELECT employee_id, first_name FROM employees WHERE department_id = 10;

BEGIN
    UPDATE employees SET salary = salary * 1.1 WHERE department_id = 10;
    COMMIT;
END;

/
```

## Limitations for plain SQL files

Unlike trace or extractor files, plain SQL files aren't tied to a live Oracle session, which introduces some limitations:

-   **SQL\*Plus commands:** HM recognizes the following SQL\*Plus commands:

    - `EXIT`
    - `SHOW ERRORS` (`SHOW ERR`)
    - `START`
    - `WHENEVER`
    - `EXEC`
    - `DEFINE`
    - `SET`
    - `COLUMN`
    - `VARIABLE` 
    
    All other SQL\*Plus commands are unsupported and may cause parsing errors or be silently skipped. Supported commands aren't validated against an actual database, since there's no active session to resolve them against.

-   **No schema context:** Because plain SQL files don't link to a specific Oracle session or audit trail, HM can't determine the schema owner of the objects referenced. HM assesses statements without schema-level context, which may reduce the precision of compatibility results compared to trace or extractor files. Providing Oracle User UIDs at upload time helps narrow the scope of the assessment. As a result, the **Schema Name** column in the application's **Overview** tab appears empty.

-   **MP Project column:** The **MP Project** column in the application's **Overview** tab appears empty for plain SQL uploads.

-   **Procedure and function queries:** Queries executed from within stored procedures or functions are captured as separate individual statements in the assessment results.
