If you already have SQL scripts — for example, scripts used for database initialization, maintenance jobs, or SQL embedded in source code files — you can upload them directly as plain `.sql` files without any Oracle-side configuration.

## Format requirements

Plain SQL files must follow these formatting rules for Hybrid Manager (HM) to parse them correctly:

-   Every SQL statement must end with a semicolon (`;`).
-   Every PL/SQL block (anonymous blocks, stored procedures, triggers) must be terminated by a forward slash (`/`) on its own line, preceded by a blank line.
-   Files must be encoded in UTF-8.

**Example:**

```sql
SELECT employee_id, first_name FROM employees WHERE department_id = 10;

BEGIN
    UPDATE employees SET salary = salary * 1.1 WHERE department_id = 10;
    COMMIT;
END;

/
```

## Limitations

-   **SQL\*Plus commands:** Support for SQL\*Plus commands (such as `SET`, `DEFINE`, `SPOOL`, `PROMPT`, and `@`) is limited. Not all SQL\*Plus commands are recognized, and unsupported commands may cause parsing errors or be silently skipped.
-   **No schema context:** Because plain SQL files aren't linked to a specific Oracle session or audit trail, HM can't determine the schema owner of the objects referenced. Statements are assessed without schema-level context, which may reduce the precision of compatibility results compared to trace or extractor files. Providing Oracle User UIDs at upload time helps narrow the scope of the assessment.
-   **No validation of SQL\*Plus commands:** Commands in `.sql` files that reference schema objects aren't validated against an actual database, since there's no active session to resolve them against.
