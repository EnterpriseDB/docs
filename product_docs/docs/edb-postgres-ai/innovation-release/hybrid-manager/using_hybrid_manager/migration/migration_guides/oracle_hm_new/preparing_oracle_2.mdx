Before you perform a migration of an Oracle source using Hybrid Manager (HM), you must configure a dedicated database user and alter several database configurations. These steps ensure that all migration components, including the EDB Postgres AI agent (`beacon-agent`) and the DMS agent (`cdcagent`), can establish a stable connection, discover your schemas, and perform data extraction operations.

## Prerequisites

- Collect the names of the schemas you want to migrate.
- Identify your Oracle architecture (CDB/PDB multitenant vs. Legacy non-CDB).

## Creating a migration user

To simplify the migration process, we recommend creating a single **Common User** (for multitenant) or a dedicated system user (for non-CDB). This unified user handles both schema assessment via the EDB Postgres AI agent and live data migration via the DMS agent.

1.  Log in to your Oracle database using the SYS account with SYSDBA privileges:

    ```shell
    sqlplus sys/<PASSWD>@//<HOST>:<PORT>/<DB_IDENTIFIER> as sysdba
    ```

    Where `<DB_IDENTIFIER>` is the **Root Container** (e.g., `CDB1`) for multitenant or the SID/Service Name for non-CDB.

    !!!important
    All user creation and permission granting must be performed while connected as `SYS AS SYSDBA`. Using the `SYSTEM` account or a standard DBA account to create common users can lead to inconsistent metadata and authentication errors.
    !!!

1.  Create the migration user:

    <TabContainer syncKey="db_type">
    <Tab title="CDB/PDB">
    
    In a multitenant environment, you must create a **Common User**. Common users exist across all containers and must be prefixed with `C##`.

    1.  Check whether you are in the root: 

        ```sql
        SHOW CON_NAME;
        __OUTPUT__
        CON_NAME
        ------------------------------
        CDB$ROOT
        ```

    1.  Create a new user:

        ```sql
        -- Create the common user
        CREATE USER C##MIG_USER IDENTIFIED BY "<MIG_PASSWD>" CONTAINER=ALL;
        
        -- Allow the user to transition between containers
        GRANT CREATE SESSION, SET CONTAINER TO C##MIG_USER CONTAINER=ALL;
        ```

    </Tab>
    
    <Tab title="non-CDB">
    
    Create a standard database user for all migration processes: 

    ```sql
    CREATE USER MIG_USER IDENTIFIED BY "<MIG_PASSWD>";
    GRANT CREATE SESSION TO MIG_USER;
    ```
    
    </Tab>
    </TabContainer>

3.  Run the following grants to provide the user with the necessary metadata discovery and log mining privileges. 

    <TabContainer syncKey="db_type">
    <Tab title="CDB/PDB">

    ```sql
    -- Discovery: Required for beacon-agent to see schemas and metadata
    GRANT SELECT ANY DICTIONARY TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT_CATALOG_ROLE TO C##MIG_USER CONTAINER=ALL;
    GRANT EXECUTE_CATALOG_ROLE TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ANY TABLE TO C##MIG_USER CONTAINER=ALL;

    -- Data Movement: Required for DMS agent to handle LOBs and tracking
    GRANT CREATE TABLE TO C##MIG_USER CONTAINER=ALL;
    ALTER USER C##MIG_USER QUOTA UNLIMITED ON USERS CONTAINER=ALL;
    GRANT FLASHBACK ANY TABLE TO C##MIG_USER CONTAINER=ALL;
    GRANT LOCK ANY TABLE TO C##MIG_USER CONTAINER=ALL;
    GRANT CREATE SEQUENCE TO C##MIG_USER CONTAINER=ALL;

    -- CDC/LogMining: Required for live migration
    GRANT LOGMINING, SELECT ANY TRANSACTION TO C##MIG_USER CONTAINER=ALL;
    GRANT EXECUTE ON DBMS_LOGMNR TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$LOG TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$LOGFILE TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$ARCHIVED_LOG TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$DATABASE TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$LOGMNR_LOGS TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$LOGMNR_CONTENTS TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$ARCHIVE_DEST_STATUS TO C##MIG_USER CONTAINER=ALL;
    GRANT SELECT ON V_$TRANSACTION TO C##MIG_USER CONTAINER=ALL;
    GRANT EXECUTE ON SYS.DBMS_LOGMNR_D TO C##MIG_USER CONTAINER=ALL;
    ```

    Change to the PDB and grant explicit select grant on your tables: 

    ```sql
    -- Connect to the PDB
    ALTER SESSION SET CONTAINER = <PDB_NAME>;

    -- Grant explicit select on your app tables
    GRANT SELECT ON <SCHEMA_NAME>.<TABLE_NAME> TO C##MIG_USER;
    GRANT SELECT ON <SCHEMA_NAME>.<TABLE_NAME> TO C##MIG_USER;
    ```

    </Tab>
    <Tab title="non-CDB">

    ```sql
    GRANT SELECT ANY DICTIONARY, SELECT_CATALOG_ROLE TO MIG_USER;
    GRANT SELECT ANY TABLE, CREATE TABLE TO MIG_USER;
    ALTER USER MIG_USER QUOTA UNLIMITED ON USERS;

    -- CDC/LogMining
    GRANT LOGMINING, SELECT ANY TRANSACTION TO MIG_USER;
    GRANT EXECUTE ON DBMS_LOGMNR TO MIG_USER;
    GRANT SELECT ON V_$LOG, V_$LOGFILE, V_$ARCHIVED_LOG, V_$DATABASE TO MIG_USER;
    ```

    Explicit select grant on your tables: 

    ```sql
    -- Grant explicit select on your app tables
    GRANT SELECT ON <SCHEMA_NAME>.<TABLE_NAME> TO C##MIG_USER;
    GRANT SELECT ON <SCHEMA_NAME>.<TABLE_NAME> TO C##MIG_USER;
    ```

    </Tab>
    </TabContainer>

## Verifying the migration user 

Before moving to the agent configuration, verify that your new migration user can authenticate over the network and has sufficient visibility into your application data.

1.  Log in to the database as the new user:

    <TabContainer syncKey="db_type">
    <Tab title="CDB/PDB">
    
    Log in to the specific Pluggable Database (PDB) where your data resides. You must use the PDB's Service Name (e.g., ORCLPDB1) to ensure the session context is correct for schema discovery.
    
    ```bash
    sqlplus C##MIG_USER/password@//<HOST>:1521/<PDB_SERVICE_NAME>
    ```
    
    </Tab>
    <Tab title="Non-CDB (Legacy)">
    
    Log in to the database instance using the standard Service Name or SID.
    
    ```sql
    sqlplus MIG_USER/password@//<HOST>:1521/<SERVICE_NAME>
    ```
    
    </Tab>
    </TabContainer>

1.  Run the following to ensure the user can see your application tables:

    ```sql
    -- Replace YOUR_SCHEMA with your actual schema names in UPPERCASE
    SELECT owner, table_name 
    FROM dba_tables 
    WHERE owner IN ('YOUR_SCHEMA_1', 'YOUR_SCHEMA_2')
    ORDER BY owner, table_name;
    ```

| Issue | Meaning | How to fix | 
|-------|---------|------------|
| ORA-01017 | Invalid username/password | Ensure you are connecting to the PDB Service Name, not the CDB SID. |
| Empty Results | Permission or Case-Sensitivity | Verify `SELECT ANY DICTIONARY` was granted and schema names are 'UPPERCASE'. |


## Configure database logging

For live migration (CDC), Oracle must be in `ARCHIVELOG` mode and have Supplemental Logging enabled at the database level.

### Enable `ARCHIVELOG` mode

Migration tools require the database to be in ARCHIVELOG mode to ensure that data changes are captured during the transfer.

1.  Connect to the root container or database instance as SYSDBA:

    <TabContainer syncKey="db_type">
    <Tab title="CDB/PDB">

    Connect to the root container:

    ```shell
    sqlplus sys/<PASSWD>@//<HOST>:<PORT>/<CDB_SERVICE_NAME> as sysdba
    ```

    </Tab>
    <Tab title="Non-CDB (Legacy)">

    Connect to the database instance:

    ```shell
    sqlplus sys/<PASSWD>@//<HOST>:<PORT>/<SID_OR_SERVICE> as sysdba
    ```

    </Tab>
    </TabContainer>

1.  Verify the status:

    ```sql
    SELECT log_mode FROM v$database;
    ```

    If the status is `NOARCHIVELOG`, you must enable it. 

1.  Configure the Fast Recovery Area (FRA) and enable logging:

    !!!Tip
    - Create a directory for Fast Recovery Area (FRA) in advance.
    - Ensure the `db_recovery_file_dest_size` is large enough to hold at least 24 hours of redo logs during peak migration loads to prevent the database from hanging.
    !!!

    ```sql
    ALTER SYSTEM SET db_recovery_file_dest_size = <RECOVERY_FILE_DEST_SIZE> SCOPE=BOTH;
    ALTER SYSTEM SET db_recovery_file_dest = '<RECOVERY_FILE_DEST>' SCOPE=BOTH;
    ```

1.  Restart the database while you enable `ARCHIVELOG`:

    ```sql
    SHUTDOWN IMMEDIATE;
    STARTUP MOUNT;
    ALTER DATABASE ARCHIVELOG;
    ALTER DATABASE OPEN;
    ```

1.  Confirm the change: 

    ```sql
    archive log list;
    ```

### Enable supplemental logging

Supplemental logging ensures that the redo logs contain enough information for the DMS agent to identify changed rows.

1.  Connect to the root container or database instance as SYSDBA:

    <TabContainer syncKey="db_type">
    <Tab title="CDB/PDB">

    Connect to the root container:

    ```shell
    sqlplus sys/<PASSWD>@//<HOST>:<PORT>/<CDB_SERVICE_NAME> as sysdba
    ```

    </Tab>
    <Tab title="Non-CDB (Legacy)">

    Connect to the database instance:

    ```shell
    sqlplus sys/<PASSWD>@//<HOST>:<PORT>/<SID_OR_SERVICE> as sysdba
    ```

    </Tab>
    </TabContainer>

1.  Enable Minimal and Primary Key logging:

    ```sql
    ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
    ALTER DATABASE ADD SUPPLEMENTAL LOG DATA (PRIMARY KEY) COLUMNS;
    ALTER SYSTEM SWITCH LOGFILE;
    ```

1.  Verify the status:

    ```sql
    SELECT supplemental_log_data_min, supplemental_log_data_pk FROM v$database;
    ```

    Both should return `YES`.

## Enable table-level supplemental logging

While database-level logging (enabled in the previous step) provides the foundation, you must also specify which individual tables require full column logging. This ensures that the DMS agent receives a _before_ and _after_ image of every row change.

!!! Note
- Make sure supplemental logging is enabled on tables that you want to migrate. Without it, the synchronization doesn't work, as it's a requirement for Debezium.

- For Oracle, the terms *schema* and *user* are used interchangeably. When you run the SQL commands for data migration, to avoid errors, be sure to set the schema name to the correct user account.

- For Oracle XE compatibility restrictions, see [limitations](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/limitations/#oracle-xe-compatibility-restrictions).
!!!

<TabContainer syncKey="db_type">
<Tab title="CDB/PDB">

Table-level logging must be enabled within the Pluggable Database (PDB) where your data resides. Connect as SYSDBA.

```sql
sqlplus sys/<PASSWD>@//<HOST>:<PORT>/<PDB_NAME> as sysdba
```

Enable logging for your specific tables. You must run this command for every table you intend to migrate:

```sql
ALTER TABLE <SCHEMA_NAME>.<TABLE_NAME> ADD SUPPLEMENTAL LOG DATA (ALL) COLUMNS;
```

</Tab>

<Tab title="Non-CDB">

Enable logging for your specific tables. You must run this command for every table you intend to migrate:

```sql
ALTER TABLE <SCHEMA_NAME>.<TABLE_NAME> ADD SUPPLEMENTAL LOG DATA (ALL) COLUMNS;
```

</Tab>
</TabContainer>

!!!tip Generating commands for a whole schema

If you have dozens of tables, don't run them manually. You can generate the SQL commands for an entire schema by running this query (replace YOUR_SCHEMA_1 with your schema name):

```sql
SELECT 'ALTER TABLE ' || owner || '.' || table_name || ' ADD SUPPLEMENTAL LOG DATA (ALL) COLUMNS;' 
FROM dba_tables 
WHERE owner = 'YOUR_SCHEMA_1';
```

Copy the output and execute it to update the entire schema at once.
!!!

<details><summary>More information...</summary> 

Supplemental logging refers to the capture of additional information in Oracle redo logs, such as *before* state. This extra redo log information is needed for log-based applications like EDB DMS to capture change events. See Oracle's [Supplemental Logging documentation](https://docs.oracle.com/en/database/oracle/oracle-database/19/sutil/oracle-logminer-utility.html#GUID-D857AF96-AC24-4CA1-B620-8EA3DF30D72E) for more information.

To use Debezium for data migration, you must enable supplemental logging at both the database and table levels. This is crucial for Debezium to correctly capture change data:

- **Database-level supplemental logging** is the foundation for change data capture (CDC). It ensures that every change message includes a tracking number, which is typically the primary key of the modified row. This allows you to uniquely identify which specific record was changed. Without this, Debezium can't link a change event to its source row.

- **Table-level supplemental logging** provides the detailed information. It ensures that the change message includes a before image of the entire row, that is, a complete copy of all columns before the change occurred. This is especially useful for understanding exactly what was changed, as Debezium can compare the before and after states. Without this, Debezium captures only the updated column and not the full context of the change.


</details>

## Prepare tablespaces for the migration user

The Data Migration Service (DMS) and the underlying Debezium engine require dedicated tablespaces to store internal metadata, log-mining details, and tracking offsets. This keeps migration-related overhead separate from your application data.

**Initial size:** 25MB is the minimum for testing. Production environments typically require 512MB or more.

**File paths:** Always consult your Database Administrator (DBA) to determine the correct directory for `.dbf` files on your server.

<TabContainer syncKey="db_type">
<Tab title="CDB/PDB">

In a multitenant environment, you must create a tablespace in the Root Container (CDB) for global log mining and a tablespace in the Pluggable Database (PDB) for table-specific tracking.

1.  Create the CDB tablespace (run in CDB$ROOT as SYSDBA):

    ```sql
    CREATE TABLESPACE <TS_NAME> 
    DATAFILE '<CDB_PATH>/<TS_NAME>.dbf' 
    SIZE 25M REUSE AUTOEXTEND ON MAXSIZE UNLIMITED;

    -- Grant quota to the common user
    ALTER USER C##MIG_USER QUOTA UNLIMITED ON <TS_NAME>;
    ```

1.  Switch to the PDB:

    ```sql
    ALTER SESSION SET CONTAINER = <PDB_NAME>;
    ```

1.  Create the PDB tablespace:

    ```sql
    CREATE TABLESPACE <TS_NAME> 
    DATAFILE '<PDB_PATH>/<TS_NAME>.dbf' 
    SIZE 25M REUSE AUTOEXTEND ON MAXSIZE UNLIMITED;

    -- Grant quota to the common user within the PDB
    ALTER USER C##MIG_USER QUOTA UNLIMITED ON <TS_NAME>;
    ```

    !!!tip
    Using the same name for both tablespaces (e.g., MIGRATOR_TS) simplifies configuration later.
    !!!

1.  Change the default tablespace in both the CDB and the PDBÖ

    ```sql 
    -- In the CDB ROOT
    ALTER SESSION SET CONTAINER = CDB$ROOT;
    ALTER USER C##MIG_USER DEFAULT TABLESPACE MIGRATOR_TS;
    
    -- In the PDB
    ALTER SESSION SET CONTAINER = ORCLPDB1;
    ALTER USER C##MIG_USER DEFAULT TABLESPACE MIGRATOR_TS;
    ```

</Tab>
<Tab title="Non-CDB">

In a standalone environment, you only need to create a single tablespace to handle both log mining and metadata tracking.

```sql
CREATE TABLESPACE <TS_NAME> 
DATAFILE '<PATH>/<TS_NAME>.dbf' 
SIZE 25M REUSE AUTOEXTEND ON MAXSIZE UNLIMITED;

ALTER USER MIG_USER QUOTA UNLIMITED ON <TS_NAME>;
```

Change the default tablespace:

```sql 
-- In the CDB ROOT
ALTER SESSION SET CONTAINER = CDB$ROOT;
ALTER USER C##MIG_USER DEFAULT TABLESPACE MIGRATOR_TS;
```

</Tab>
</TabContainer>

<details><summary>More information...</summary> 

To perform data migration, both DMS and its underlying Debezium library require a ​​dedicated user​​. Before creating this user, set up the required tablespaces.

For CDB architecture, you need two tablespaces:
- **CDB-level tablespace** &mdash; This tablespace is for Debezium's internal processes. It's used for log mining and other management operations that must run at the container database level. 
- **PDB-level tablespace** &mdash; This one is for Debezium operations that are specific to your pluggable database. It handles processes related to the individual tables you're capturing within that PDB.

</details>

### Run the config validation script

The EDB DMS Reader installation includes a helper script, oracleConfigValidation.sh, which performs a comprehensive check on your Oracle source. It verifies everything from your `ARCHIVELOG` status to the specific C## user grants we just configured.

1.  Navigate to the EDB DMS agent folder in `/opt/cdcagent/reader/`.

1.  Create a shell array containing the tables you intend to migrate in `schema.table` format:

    ```bash
    arr=(test1.table1 test1.table2 test1.table3 test1.table4)
    ```

    For example: 
    ```bash
    arr=(HR_APP.EMPLOYEES FINANCE_APP.SALARIES)
    ```

1.  Execute the script using your `SYSDBA` credentials and your new C## migration user.

    ```bash
    ODB_SYSDBA_USERNAME=sys ODB_SYSDBA_PASSWORD=password ODB_HOST=localhost ODB_PORT=5432 ORACLE_SID=postgres ODB_DBZ_USERNAME=debezium ODB_DBZ_PASSWORD=dbz LOG_MINER_TBS_NAME=LOGMINER_TBS ./oracleConfigValidation.sh "${arr[@]}"
    ```

    You get a response similar to the following:

    ```shell
    *** [Transporter] - Validate system user connectivity
    [Pass] Provide username and password of system user can connect to database.
    
     *** [Transporter] - Validate debezium user connectivity
    [Pass] Provide username and password of debezium user can connect to database.
    
     *** [Transporter] - Validate Archive Log mode
    [Pass] Database log mode is Archive Mode.
    
     *** [Transporter] - Validate fast recovery area parameters
    [Pass] Recovery area path was set.
    [Pass] Recovery area size limit meets requirements.
    [Pass] Recovery area remaining size meets requirements.
    
     *** [Transporter] - Validate if database supplemental logging is enabled
    [Pass] Supplemental logging check for database is passed.
    
     *** [Transporter] - Validate if table supplemental logging is enabled
    [Pass] All source tables enable supplemental logging.
    
     *** [Transporter] - Validate redo log count&size
    [Pass] Redo log count meets the requirements.
    [Pass] Redo log size meets the requirements.  
    [Pass] Redo logs are all active.
    
     *** [Transporter] - Validate if tablespace for migration with Limited Privileges were created
    [Pass] LOGMINER_TBS tablespace has enough datafiles.
    [Pass] LogMiner DataFile Size meet the requirements.
    [Pass] LogMiner DataFile Max Size meet the requirements.
    [Pass] LogMiner DataFile are auto-extendable.
    
     *** [Transporter] - Validate if user for migration with Limited Privileges were created
    Oracle version is 19, which is 12c or newer.
    [Pass] This user has all sys privileges that migration needs.
    [Pass] This user has all role privileges that migration needs.
    [Pass] This user has all select privilege on specific tables that migration needs.
    [Pass] This user has all execute privilege on specific tables that migration needs.
    
     *** [Transporter] - Validate if user for migration with LOGMINER_TBS tablespace
    [Pass] User for migration use LOGMINER_TBS tablespace.
    ```
    
    Your database is ready for CDC migration.

!!! Note 
    Address any `[Failed]` or `[Suggestion]` statuses by modifying the source database settings as recommended here. `[Failed]` checks are blocking issues that you must resolve before the DMS can execute the data migration. `[Suggestion]` checks allow migration but can negatively affect performance if left unaddressed.
