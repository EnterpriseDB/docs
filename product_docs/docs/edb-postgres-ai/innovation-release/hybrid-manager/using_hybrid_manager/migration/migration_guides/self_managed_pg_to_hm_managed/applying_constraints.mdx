<TabContainer syncKey="pg_schema_path">
<Tab title="Simplified (recommended)">

If you used the **Schema and data** migration scope, HM applied the foreign key, check, and exclusion constraints automatically — either concurrently with streaming (if you ran **Finalize Schema Concurrently** during cutover) or in the post-cutover **Schema Finalization** stage after **Mark as Completed**. See [Finalizing migration](finalizing_migration) for the full flow.

Verify that all constraints are present in the destination database before proceeding to verification.

</Tab>
<Tab title="Manual">

At the beginning of your data migration journey, you prepared and imported the schema of your source database, excluding foreign key, check, and exclusion constraints. Connect to the destination database and reapply those constraints now.

## Primary key and unique constraints

For primary key and unique constraints, you already created the tables and constraints in the destination Postgres database. This allowed DMS to map them to the source objects and migrate data successfully. You don't need to do anything else.

The same applies to `NOT NULL` constraints if you included them in your schema import.

## Foreign key, check, and exclusion constraints

You can now reapply any foreign key, check, or exclusion constraints you excluded during the schema preparation and import.

</Tab>
</TabContainer>


## Ensuring data integrity

Rows in tables that don't have primary key or unique constraints were migrated with at-least-once delivery. It's possible that these rows are duplicates. Deduplication can be performed as part of the verification.

## Verification

To verify that all data was migrated, compare the source and destination databases. You can use [LiveCompare](/livecompare/latest/).
