EDB DMS migrates data only — not schema. You must import your schema to the destination database before the data migration begins so that the migration has tables to populate.

## Export schema with Migration Portal

Use Migration Portal to convert and migrate your Oracle schemas.

### Prerequisites

- You [registered your Oracle database with HM](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/migration_guides/oracle_to_hm_managed/) via the EDB Postgres AI agent.

- Optional: Your HM administrator [enabled and configured the AI Copilot](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/enabling_migration_copilot/) to help resolve schema incompatibilities in Migration Portal.

### Open your Migration Portal project

After registering your Oracle database with the EDB Postgres AI agent, a Migration Portal project is **created automatically** — one per registered database, based on the `resource_id` in your `beacon_agent.yaml`.

!!!warning
Schemas added after registration aren't reflected in the auto-created project.
!!!

To find your project: 

1.  Navigate to your Oracle database's detailed view. (**Estate > Migrations > [your database]** or alternatively, **[your project] > Migrate > Databases > [your database]**).

1.  Select the **Schemas** tab.

1.  Select the project link in the **MP Project** column to open the Migration Portal's project-scoped workspace for this database.

1.  Use Migration Portal to view the ingested DDL. Resolve any compatibility issues identified using the AI Copilot or QuickHelp.

<details><summary>Create a Migration Portal project manually</summary>

Automatic project creation won't trigger if:

- The ingested DDL exceeds the 300 MB default limit (or your [customized limit](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/schema_migration/migration_portal/controlling_project_creation/#customizing-the-maximum-size-of-migration-portal-projects)).
- You [disabled automatic project creation](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/schema_migration/migration_portal/controlling_project_creation/#disabling-the-automatic-creation-of-migration-portal-projects).

In these cases, create a project manually:

1. In the **Estate** page, select the **Migrations** tab, then select your database.

1. In the detailed view, select the **Schemas** tab, then select **New MP Project** or the plus sign next to a schema.

    !!!tip
    The **Dependencies** column shows which schemas the current one relies on. Group a schema with all its dependencies in a single project to ensure referenced objects exist in the destination.
    !!!

1. Enter the project details and select **Continue**.

1. Select the schemas to include and select **Create New Project**.

    !!!note
    You can't add schemas to the project after creation. Use a separate project to migrate additional schemas to the same destination.
    !!!

HM loads the schemas into Migration Portal in the background. Once ready, a project link appears — select it to open Migration Portal and begin resolving incompatibilities using the AI Copilot or QuickHelp.

</details>

## Import

After resolving all compatibility issues in Migration Portal, import the schema to the destination database.

### Constraint handling

Import constraints in two stages to avoid performance degradation during migration:

- **Include now** (before data migration): primary key, unique, and NOT NULL constraints. EDB DMS requires primary key and unique constraints to guarantee exactly-once delivery. NOT NULL constraints have negligible performance impact.

- **Exclude now, reapply later** (after data migration): foreign key, check, and exclusion constraints. These cause unnecessary CPU and memory overhead while data is actively being migrated.

When following the steps below, disable **Constraints** in Migration Portal and keep a record of what you excluded so you can reapply them after the migration completes.

!!!note
For tables without primary key or unique constraints, EDB DMS provides at-least-once delivery only. You can perform deduplication during migration verification.
!!!

<TabContainer>
<Tab title="Online migration (recommended)">

Use this option when you don't need to rename any schemas, tables, or columns.

1. Select **Migrate to** > **Online migration**.

1. Select your schemas. In the object list beneath the database, disable **Constraints**. Select **Next**.

1. Enter the host name of your destination cluster.

    !!!note
    For HM-managed clusters, find the host name under **Clusters > [cluster name] > Connect**.
    !!!

1. Select **Next** and wait for the import to complete.

1. Connect to the destination database and confirm the schemas are present.

</Tab>
<Tab title="Offline migration (with object renaming)">

Use this option when you need to map schemas, tables, or columns to different names in the destination.

1. Select **Migrate to** > **Offline migration**.

1. Select your schemas. In the object list beneath the database, disable **Constraints**. Select **Next**.

1. Select **Download SQL file**.

1. Edit the object names directly in the downloaded DDL. Ensure that renamed objects are also updated in any dependent views, constraints, or other references. See [Mapping schemas, tables, and columns](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/migrate/migrations/#mapping-schema-tables-and-columns-during-a-migration) for details.

1. Connect to the destination database and apply the modified DDL file.

</Tab>
</TabContainer>
