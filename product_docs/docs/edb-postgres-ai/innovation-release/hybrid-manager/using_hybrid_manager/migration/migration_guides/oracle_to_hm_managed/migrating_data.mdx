With the EDB DMS agent running and your schema either imported or ready for HM to import, you're ready to create a migration in the HM console. This page walks through the full migration lifecycle: creating the migration, monitoring its progress, finalizing it, and handling post-migration constraints.

The steps below vary depending on the approach you chose on the previous page. Select the tab that matches your choice.

## Creating a migration

A migration establishes a sync between your Oracle source and a destination database cluster managed by HM.

1.  In the HM console, select the project you're using for the migration.

1.  Select **Migrate > Migrations**.

1.  Select **Create Migration**.

1.  Assign a meaningful name to the migration — for example, `oracle-prod-migration`.

1.  Select the source of the migration. This corresponds to the `DBCONFIG_ID` you set when configuring the DMS agent.

1.  In **Scope**, select the option that matches your approach:

    <TabContainer syncKey="migration_scope">
    <Tab title="Simplified (recommended)">

    Select **Schema and data**. HM imports the converted schema from the resolved Migration Portal project, creates tables without constraints to keep the initial snapshot fast, applies primary key and unique constraints after the snapshot completes (required for exactly-once delivery), and applies the remaining schema either concurrently with streaming (via **Finalize Schema Concurrently**) or automatically in the post-cutover **Schema Finalization** stage. See [Finalizing a streaming migration](#finalizing-a-streaming-migration).

    </Tab>
    <Tab title="Manual">

    Select **Data only**. The schema was already imported to the destination via Migration Portal.

    </Tab>
    </TabContainer>

    !!!note
    You can use **Schema only** if you want to migrate the database structure without migrating any table data.
    !!!

1.  Select the migration type:
    - [**Snapshot**](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/terminology/#snapshot-migration) — copies data once and completes. Use this for a one-time, offline migration.
    - [**Snapshot + Streaming**](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/terminology/#snapshot--streaming-migration) — copies an initial snapshot, then continuously streams changes from the source. Use this to minimize downtime.

1.  Under **Destination Type**, select **HM-managed Postgres** and select your destination database cluster and database. Then select **Next**.

1.  Select the tables and columns to migrate. Modify table and column names if needed.

1.  Select **Create Migration**.

The HM console displays the new migration. If you chose **Snapshot + Streaming**, the EDB DMS agent streams data changes continuously while the migration shows a **Running** state.

!!!note
The EDB DMS agent streams data changes only — not DDL changes (schema modifications).
!!!

## Monitoring progress

### Migration list view

The **Migrations** page shows a summary of all migrations in the project. The **Migration progress** column displays three metrics per migration:

- **Total number of tables** — total tables included in this migration
- **Snapshot tables completed** — tables that have finished the snapshot stage
- **Migrated rows** — cumulative rows migrated across all tables

### Migration details view

Select a migration from the list to open the details page, which has three tabs:

- **Overview** — shows the current migration state and general information about the source and destination databases. For **Snapshot + Streaming** migrations, the migration is caught up when the status is **Streaming**. For **Snapshot** migrations, it's finished when the status is **Completed**.

- **Tables** — shows all tables selected for migration. For each table, you can see source and destination table and schema names, snapshot status, mapped columns, migrated rows, and migration rate. Select the **+** icon to expand a table and view column-level mappings.

- **Error history** — lists all errors logged for this migration. Select the **i** icon to open the full error log for a specific error. For stuck or failed migrations, active errors here indicate where to start troubleshooting.

## Finalizing a streaming migration

!!!note
This section applies only to **Snapshot + Streaming** migrations. For **Snapshot** migrations, the DMS marks the migration as completed automatically.
!!!

### Monitoring catch-up before cutover

Before stopping writes on the source, give the destination time to catch up. Use the **Migration details** page to track row counts and replication lag. Begin the cutover only after the destination is consistently close to the source.

### Finalizing schema concurrently (recommended)

<TabContainer syncKey="migration_scope">
<Tab title="Simplified (recommended)">

If you used the **Schema and data** scope, HM can apply most schema objects while the stream is still running: primary keys, unique constraints, indexes (built with `CREATE INDEX CONCURRENTLY`, so live writes aren't blocked), foreign keys (added as `NOT VALID`), check constraints, `NOT NULL` constraints, views, and triggers. Only foreign key validation is deferred until after cutover.

Running this work concurrently shrinks the read-only cutover window. The larger your tables and the more indexes they have, the greater the time saved.

When the destination has nearly caught up:

1.  On the **Migration details** page, an alert titled **Schema Finalization Available** appears. Select **Finalize Schema Concurrently**.

1.  HM applies the deferred objects while streaming continues. The migration remains in the **Running** state, with a progress sub-step labeled **Finalizing Schema Concurrently**.

1.  When the sub-step completes, proceed to the cutover.

!!!note
**Finalize Schema Concurrently** is available only for migrations created with the **Schema and data** scope with streaming enabled.
!!!

If you skip this step, the same objects are applied serially after you mark the migration as completed. See [Schema Finalization stage](#schema-finalization-stage). The cutover window is larger in that case because indexes are built non-concurrently and constraints are validated under exclusive locks.

</Tab>
<Tab title="Manual">

Not applicable to the manual path. You applied the schema before migration and will reapply deferred constraints manually after cutover. See [Applying deferred constraints and verifying data](#applying-deferred-constraints-and-verifying-data).

</Tab>
</TabContainer>

### Performing the cutover

When you're ready to switch traffic from the source to the destination:

1.  Stop all writes on the source database.

1.  Verify in the HM console that the destination has applied the last source change.

1.  In the HM console, select **Migrate > Migrations** in your project.

1.  Select the checkmark next to your migration to mark it as completed, or open the migration details page and select **Mark as Completed**.

This stops the streaming process.

### Schema Finalization stage

<TabContainer syncKey="migration_scope">
<Tab title="Simplified (recommended)">

For migrations created with the **Schema and data** scope, the migration enters a **Schema Finalization** stage automatically after **Mark as Completed**. What runs in this stage depends on whether you ran concurrent finalization first:

- **If you ran concurrent finalization**: HM validates the foreign keys that were added as `NOT VALID` and rebuilds sequences so they continue from the correct values on the destination. This stage is short.
- **If you skipped concurrent finalization**: HM applies the full set of deferred objects now (primary keys, unique constraints, indexes, foreign keys, check constraints, `NOT NULL`, views, triggers), validates the foreign keys, and rebuilds sequences. This stage takes substantially longer and runs while the source is read-only.

The migration is fully complete only after the **Schema Finalization** stage finishes. Progress is shown on the **Migration details** page.

</Tab>
<Tab title="Manual">

No automatic schema finalization runs for the **Data only** scope. Continue to [Applying deferred constraints and verifying data](#applying-deferred-constraints-and-verifying-data) to reapply the foreign key, check, and exclusion constraints you deferred before migration.

</Tab>
</TabContainer>

## Applying deferred constraints and verifying data

### Reapplying foreign key, check, and exclusion constraints

<TabContainer syncKey="migration_scope">
<Tab title="Simplified (recommended)">

HM applied all deferred constraints and rebuilt sequences automatically during the **Schema Finalization** stage. No manual action is needed. See [Schema Finalization stage](#schema-finalization-stage).

</Tab>
<Tab title="Manual">

During the schema import, you excluded foreign key, check, and exclusion constraints to avoid performance overhead during data migration. Now that migration is complete, connect to the destination database and reapply those constraints.

Primary key, unique, and `NOT NULL` constraints were already applied before migration and don't require any action.

</Tab>
</TabContainer>

### Deduplicating rows when needed

For tables that had no primary key or unique constraints, the EDB DMS agent provides at-least-once delivery, which can result in duplicate rows. Perform deduplication as part of your verification step.

### Verifying migrated data

Compare the source and destination databases to confirm all data was migrated successfully. You can use [LiveCompare](/livecompare/latest/) for this.

### Assessing your application SQL before cutover

With the database migration complete, the next step before switching your application to EPAS is to confirm that the SQL queries your application executes are compatible with the target database. Even a successful schema and data migration doesn't guarantee that application-level SQL — embedded in your code, ORM, or scripts — will work without changes.

Use HM's application assessment to identify incompatible queries and review suggested transformations before application cutover. See [Application assessment](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/application_assessment/) for an overview, or go directly to [Creating an application record](/edb-postgres-ai/current/hybrid-manager/using_hybrid_manager/migration/application_assessment/creating_an_application_record/) to get started.
