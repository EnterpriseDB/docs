When you configure an automation with **Approval Required**, Hybrid Manager (HM) doesn't run the operation immediately. Instead, it creates a pending task and waits for a team member to review it. **Task Manager** is where that review happens — it gives you a centralized view of all pending, running, and completed automation tasks across your project, so you can stay in control of what runs and when.

## Accessing Task Manager

**Task Manager** is available at two levels:

- **Project level** — Within a project, in the left navigation, select **Task Manager**. This view shows all pending, running, failed, and completed tasks across every cluster in the project.

- **Cluster level** — Within a project, in the left navigation, select **Clusters** > **Postgres Clusters**, then select a cluster. In the cluster view, select the **Task Manager** tab. This view shows only the tasks for that specific cluster.

## Reading the task list

Each row in the task list represents a single automation task. The following columns are shown by default:

| Column | Description |
|---|---|
| Task Name | The name of the automation task. Select the name to open the task detail sidebar. |
| Status | The current state of the task. See [Understanding task status](#understanding-task-status). |
| Created | When the task was created. |
| Duration | How long the task took to execute. Shown as a dash (`-`) for tasks that haven't run yet. |
| Source | The automation rule that triggered the task. |
| Target | The cluster targeted by the task. Not shown when viewing **Task Manager** from within a specific cluster. |

You can customize which columns are displayed using the settings icon at the top of the task list. **Priority** is an optional column not shown by default.

## Understanding task status

Each task moves through a lifecycle that you can track in the **Status** column:

| Status | Meaning |
|---|---|
| `Pending` | The task is waiting — either for approval (if approval is required) or for its maintenance window to open. |
| `Gate Satisfied` | All gates are satisfied. The task will execute when its maintenance window opens, or immediately if no window is set. |
| `Running` | The task is currently executing. |
| `Completed` | The task finished successfully. |
| `Failed` | The task couldn't run or encountered an error. When a specific failure reason is available, the status badge shows an error code instead of `Failed`. See [Failure error codes](#failure-error-codes). |
| `Cancelled` | The task was cancelled before it ran. |

### Failure error codes

When a task fails, the status badge shows a specific error code to indicate why:

| Error code | Meaning |
|---|---|
| `MaintenanceWindowFailed` | The maintenance window closed before the task could run. If the automation's cooldown period has passed, HM recreates the task on the next evaluation cycle. |
| `ApprovalFailed` | Approval was rejected, or the approval timeout expired before the task was approved. If the automation's cooldown period has passed, HM recreates the task on the next evaluation cycle. |
| `ExecutionError` | The task started but encountered an error during execution. Check the task detail sidebar for the error message. |
| `Timeout` | The task exceeded the maximum allowed execution time. |

## Approving or rejecting a task

When a task is in **Pending** status and requires your approval:

1. Select the task name or its status badge to open the task detail sidebar.
1. Review the task details — including the automation type, target cluster or data group, and the approval description set by the automation owner.
1. Select **Approve** to allow the task to proceed, or **Deny** to reject it.
1. If the automation was configured with **Require Reason**, add an explanation in the **Reason** field before confirming your decision.

After approval, the task moves to **Running** if no maintenance window was configured, or to **Gate Satisfied** if a maintenance window configuration exists. After rejection, the task is closed. If the automation's cooldown period has passed, HM recreates the task on the next evaluation cycle.

!!!note
To approve a task, you need the same permissions required to perform the underlying operation on the cluster. For example, approving an automatic image upgrade requires the same permissions as performing a manual upgrade. HM's RBAC system enforces this.
!!!

### Approving from a notification

You can also approve or reject a task directly from a notification. HM can deliver task notifications through two types of channels:

- **In-app** — A notification appears in the bell icon menu in the top-right corner of the HM console. Select the notification to open the pending task detail page.
- **External channels** — If you've configured email, Pager Duty, or a webhook, HM sends the notification through those channels as well. Email notifications include a direct link to the task detail page.

In all cases, the notification links you to the task detail page where you can approve or reject without navigating to Task Manager first.

## Monitoring task history

**Task Manager** retains a history of all completed, failed, and cancelled tasks. Use it to audit what ran, when, and with what outcome.

To narrow down the list, use the available filters:

- **Status** — Filter by one or more task phases (for example, show only `Pending` or `Failed` tasks).
- **Date range** — Limit results to a specific time window. Preset ranges of 7, 15, 30, and 90 days are available, or set a custom range.
- **Search** — Find tasks by name.

Select any task to open its detail sidebar.
All task activity is also recorded in the [Activity Log](../../monitoring/activity_log/), which provides an org- or project-wide audit trail across all resource types, not just tasks.
