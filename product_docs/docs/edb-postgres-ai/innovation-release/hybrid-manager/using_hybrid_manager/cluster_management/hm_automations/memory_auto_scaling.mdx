Memory auto-scaling monitors the average memory utilization of your cluster and automatically scales up memory resources when utilization exceeds a configured threshold. This prevents performance degradation during periods of high demand without requiring manual intervention.

!!!warning
Memory scaling requires a database restart. Configure a [maintenance window](maintenance_windows) to control when the restart occurs and avoid unexpected downtime on production clusters.
!!!

You configure automations per cluster. Scaling actions aren't coordinated across cluster replicas — HM scales each cluster independently.

## Memory auto-scaling parameters

When you configure an automated operation, there are specific parameters for the memory auto-scaling automation in addition to the general automation fields:

| Field | Description |
|---|---|
| **Memory Threshold** | The average memory utilization percentage that triggers a scaling event (for example, `80`). Min: 1, Max: 99. |
| **Lookback Window (minutes)** | The number of minutes over which HM averages memory utilization before comparing it to the threshold (for example, `5`). This smooths out short-lived spikes. Min: 1, Max: 60. |
| **Scaling Method** | Choose how to increase memory resources: **Scale by Quantity** adds a fixed amount each time (for example, `1Gi`, `512Mi`); **Scale by Percentage** increases by a percentage of the current value. |
| **Maximum Memory** | The upper limit for memory scaling. The automation stops scaling once this value is reached (for example, `32Gi`). |
| **Maintenance Window** | The maintenance window during which the scaling operation and required database restart are allowed to run. Because memory scaling requires a restart, leaving this empty isn't recommended for production clusters. See [Maintenance windows](maintenance_windows). |
| **Approval Required** | A user must approve all tasks generated by this automation before they run. |
| **Approval Not Required** | Scaling tasks run automatically without manual review. |

Once active, HM evaluates average memory utilization at each interval using the configured lookback window. When the threshold is crossed, a task is created in **Task Manager**. You can monitor task status — pending, completed, or rejected — from either the cluster-level **Task Manager** tab or the project-level **Task Manager** in the left navigation.
