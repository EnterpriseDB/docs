Apache Airflow® is an open-source workflow orchestration platform deployed and managed within Hybrid Manager (HM). Its extensible Python framework supports integration with a wide range of external technologies, and a web-based UI accessible from the HM dashboard provides visibility into workflow execution and debugging. Airflow scales from single-node deployments to distributed configurations for large-scale workloads.

## Workflows as code in Apache Airflow

You define Airflow workflows entirely in Python. This “workflows as code” approach brings several advantages:

- **Dynamic:** You define pipelines in code, enabling dynamic Dag generation and parameterization.
- **Extensible:** The Airflow framework includes a wide range of built-in operators that you can extend to fit your needs.
- **Flexible:** Airflow leverages the Jinja templating engine, allowing rich customizations.

## Requirements

### Technical requirements:
 
  - External PostgreSQL: Required for Airflow's metadata store.
  - Storage: Git repository access for DAG management.

### Configuration requirements:
 
  - Security keys: Encryption requires `fernetKey` and `apiSecretKey` (minimum 32 characters).
  - Git sync: Optional configuration for syncing DAGs from public or private Git repositories using SSH keys.

### Supported data sources

Airflow connects to nearly any data source via "Connections," including PostgreSQL, MySQL, AWS, and Snowflake.

### Application database

Airflow requires an external database to store its application metadata, including questions, dashboards, and settings. You must create this database before deploying Airflow.
EDB recommends PostgreSQL for production. Airflow also supports MySQL 8.0.17 or later and MariaDB 10.2.2 or later.

Have these connection details ready before installation:
- Host and port
- Database name
- Username and password

### Memory

Airflow uses approximately one quarter of available system RAM by default. Allocate at least 2 GB for Airflow, reserving additional capacity for other system processes.

### Supported data sources

Airflow officially supports drivers for the following databases:
- PostgreSQL (any currently supported version)
- MySQL 8.0.17 or later and MariaDB 10.2.2 or later
- MongoDB, SQL Server, SQLite
- Cloud data warehouses: Amazon Athena, Amazon Redshift, BigQuery, Databricks, Snowflake, SparkSQL, Starburst
- Additional databases: ClickHouse, Druid, Oracle, Presto, Vertica
- Cloud-managed services such as Amazon RDS and Aurora

## Preconfiguration
  1. Generate a Fernet key: `openssl rand -base64 32`.
  1. Provide the external PostgreSQL connection string in the HM UI.
  1. (Optional) Provide Git repository URL and SSH key for DAG syncing.
  1. Log in to the Airflow UI via Beacon.
  1. Navigate to **Admin** > **Connections** to add credentials for your external databases.
  1. Verify DAGs are appearing via the Git Sync process.

## Installing Airflow in HM

Install Airflow directly in HM.

1. Navigate to **Asset Library** then select **Apps** in the HM console.
1. Select Airflow and enter required values in the fields.

!!! note
Once you have deployed Airflow in HM, you can access it in the HM dashboard under **Apps** or **Asset library**.
!!!

### Post-deployment configuration

Once you deploy Airflow, open the Airflow from the HM UI. A setup wizard in the Airflow interface guides you through these steps:

1. Create an admin account in Airflow to manage all subsequent user additions, database connections, and system configuration.
1. Choose between self-service analytics, analytics embedded in applications, or both. You can change this later.
1. Provide your database hostname, port, name, and credentials. You can also skip this step and explore the sample database included with Airflow.
1. Choose whether to allow anonymous usage data collection. Airflow never collects anything about your data or question results.
