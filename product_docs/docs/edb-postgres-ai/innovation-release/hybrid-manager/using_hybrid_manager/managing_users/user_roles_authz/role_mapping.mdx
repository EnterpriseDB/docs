Role mapping rules allow you to automatically assign roles to users based on their identity provider (IdP) attributes. When a user logs in, Hybrid Manager (HM) evaluates the configured rules and assigns the appropriate roles without manual intervention.

This feature is useful for organizations that want to:

- Automatically grant roles based on IdP group membership.
- Reduce manual role assignment overhead.
- Ensure consistent role assignments across users from the same IdP groups.

## How role mapping works

When a user authenticates through an external IdP, HM receives information about their IdP identity and group memberships. Role mapping rules define which roles HM assigns based on these attributes.

Each rule consists of:

- **Conditions**: Criteria that must match for the rule to apply (for example, IdP identifier and group membership).
- **Roles**: The roles to assign when all conditions match.

### Rule evaluation

- All conditions within a single rule must match (AND logic).
- Multiple rules are evaluated independently (OR logic).
- A user receives the union of roles from all matching rules.

### Role assignment sources

Users can have roles from two sources:

- **Manual**: Roles that an administrator assigns directly through the HM console. An administrator can modify or remove these roles.
- **Mapping**: Roles that HM assigns automatically through role mapping rules. You can't modify these roles in the HM console. To change a user's mapped roles, update the role mapping rules or the user's group membership in the IdP.

## Prerequisites

Before configuring role mapping rules:

1. Configure an external IdP (SAML, OIDC, or LDAP) with group attribute mapping enabled.
1. Ensure users have group memberships configured in your IdP.
1. Have organization owner or appropriate administrative permissions.

!!! important
If you configure the HM IdP integration to retrieve group membership information, each user must be a member of at least one group in the IdP. Users with no group memberships can't sign in to HM.
!!!

## Configuring role mapping rules

You can configure role mapping rules at two levels:

- **Organization level**: Assigns organization-scoped roles (Organization Administrator, Organization Owner, etc.)
- **Project level**: Assigns project-scoped roles (Project Owner, Project Editor, Project Viewer, etc.)

### Creating organization-level rules

!!! important
Only an Organization Owner can create or edit organization-level role mapping rules.
!!!

1. In the HM console, navigate to **User Management**.

1. Select the **Role Mapping** tab.

1. Select **Add Role Mapping Rule**.

1. Configure the rule conditions:

   | Condition             | Description                                        |
   | :-------------------- | :------------------------------------------------- |
   | **Identity Provider** | Select the IdP this rule applies to.                |
   | **Group**             | Enter the IdP group name that users must belong to. |

1. Select the organization roles to assign when the conditions match.

1. Select **Save**.

### Creating project-level rules

!!! important
Only an Organization Owner or Project Owner can create or edit project-level role mapping rules.
!!!

You can add project-level role mapping rules to an existing project or during project creation.

#### Adding rules to an existing project

1. In the HM console, navigate to the project.

1. Go to **Users**.

1. Select the **Role Mapping** tab.

1. Select **Add Role Mapping Rule**.

1. Configure the rule conditions (Identity Provider and Group).

1. Select the project roles to assign when the conditions match.

1. Select **Save**.

#### Adding rules during project creation

1. When creating a new project, enable **Automated Role Assignments**.

1. Configure the role mapping rules as part of the project setup.

!!! note
Roles that HM assigns through role mapping rules can't be modified in the HM console. To change a user's mapped roles, update the role mapping rules or the user's group membership in the IdP.
!!!

## Example configurations

### Assigning organization owner role to IT administrators

Create an organization-level rule:

- **Identity Provider**: Your corporate SAML IdP
- **Group**: `IT-Admins`
- **Roles**: Organization Owner

HM automatically assigns all users in the `IT-Admins` group the Organization Owner role when they log in.

### Assigning project roles based on team membership

Create project-level rules for a data analytics project:

**Rule 1** - Data engineers get editor access:

- **Identity Provider**: Corporate LDAP
- **Group**: `data-engineering`
- **Roles**: Project Editor

**Rule 2** - Data analysts get viewer access:

- **Identity Provider**: Corporate LDAP
- **Group**: `data-analysts`
- **Roles**: Project Viewer

Users who are members of both `data-engineering` and `data-analysts` groups receive both Project Editor and Project Viewer roles.

## Role reconciliation

When you modify role mapping rules, HM automatically reconciles role assignments for affected users:

- **Adding a rule**: Users matching the new rule receive the configured roles on their next login.
- **Removing a rule**: HM revokes mapping-assigned roles from the removed rule; manually assigned roles remain unchanged.
- **Modifying a rule**: HM removes the previous assignments and applies new assignments based on the updated rule.

HM immediately reconciles roles for users who have previously logged in when rules change. Users who have never logged in receive the appropriate role assignments on their first login.

## Invalidating user sessions

When you update group memberships in your IdP (for example, adding a user to a new group), currently logged-in users don't automatically receive the updated group information until they log in again.

To ensure that users receive the correct roles based on their current group memberships, you can invalidate their sessions to force re-authentication.

1. In the HM console, go to **Settings** > **Identity Providers**.
1. Select the IdP whose users need to be re-authenticated.
1. Select **Invalidate Sessions**.
1. Confirm the action.

HM terminates all active sessions for users from that IdP. Users must log in again, at which point their authentication token includes the current group memberships from the IdP.

!!! note
Session invalidation affects all users from the selected IdP, not just users whose groups changed. Use this feature after making significant group membership changes in your IdP.
!!!

## Removing role mapping rules

When you remove a role mapping rule:

1. HM automatically revokes roles assigned only through mapping from affected users.
1. Manually assigned roles remain in place.
1. The change takes effect immediately for users who have previously logged in.

!!! note
Removing a role mapping rule doesn't affect users' ability to log in. It only removes the automatically assigned roles from that rule.
!!!

## Activity log

Changes to role mapping rules are recorded in the activity log. When you add or remove rules, HM logs an update action indicating whether the scope is organization or project, and the number of rules configured.

You can view role mapping activity in the organization activity log by navigating to **Settings** > **Activity Log**. Project-specific role mapping changes can also be viewed in the project's activity log by navigating to the project and selecting **Activity Log**.
