This Innovation Release includes features and enhancements.

**Release Date:** May 13, 2026

!!!info Release cadence update

HM now follows a dual release strategy: **LTS** and **Innovation Releases**. For details on this new model, see [Hybrid Manager dual release strategy](/edb-postgres-ai/current/hybrid-manager/release_notes/#hybrid-manager-dual-release-strategy).
!!!

## Release highlights

-   **Postgres-to-Postgres schema migration:** The Data Migration Service now supports complete schema migrations between Postgres databases, including proper handling of Postgres-specific types such as custom domain types.

-   **Enhanced agentic migration:** The Migration Agent Langflow template has been updated with improved architecture using Hybrid Manager's unified schema migration feature, with added support for Llama 3 Nemotron 49B and Nano 4B models.

-   **HM chatbot enhancements:** Added Anthropic Claude as a chatbot provider and bundled Hybrid Manager and PostgreSQL documentation for citation-backed answers.

-   **Role mapping improvements:** Multiple roles can now be assigned in a single role mapping rule, streamlining identity provider group-to-role assignments.

### Migrations

| Type | Description | Jira ref. |
| ---- | ----------- | --------- |
| Enhancement | Added concurrent schema finalization for `schema_and_data` migrations, allowing you to start schema operations while data migration continues running. You can now click **Finalize Schema Concurrently** on the migration detail page after the snapshot stage completes, which applies indexes, foreign keys, and other schema objects in the background using non-blocking DDL variants. The migration detail page displays concurrent schema finalization progress alongside existing snapshot and streaming indicators. When you later complete the migration, only lightweight validation steps remain, significantly reducing cutover time compared to waiting for all schema operations during the completion phase. | ET-5022 |
| Enhancement | You can now migrate schemas from Postgres to Postgres using the Data Migration Service. The DMS now handles Postgres-specific database objects—including custom domain types and other Postgres-native type definitions—during schema migration, ensuring schema objects are applied to the target before other schema objects that depend on them. This enables complete, end-to-end schema migration between Postgres databases. | MIG-8858 |
| Enhancement | Enhanced the Simple Migration Agent Langflow template with an improved architecture that leverages Hybrid Manager 1.4's unified schema migration feature, replacing the legacy Migration Portal approach. The updated workflow now includes streamlined agent components, automatic password generation and database setup, and removal of redundant components. Added support for Llama 3 Nemotron 49B and Llama 3 Nemotron Nano 4B models, providing additional AI model options for migration workflows. Enhanced database password security by moving cluster creation operations outside of LLM processing and leveraging global secrets management to prevent credential exposure. Fixed an issue whereby the list of tools in a MCP Tools component wouldn't update after selecting an MCP server until the page was refreshed. | MIG-8057 |
| Enhancement | Optimized the migration creation form to skip unnecessary table loading and validation when the migration scope is set to schema-only. The form now only fetches table data when required for data-inclusive migration scopes, improving performance and user experience. | ET-5050 |
| Bug fix | Added support for migrating EPAS PACKAGE and PACKAGE BODY schema objects from EPAS source databases to EPAS destination databases. The Data Migration Service now properly handles package and package body statements during schema migration, including correct dependency ordering to ensure package bodies are applied after referenced tables. | MIG-8903 |
| Bug fix | Added support for migrating EPAS non-PUBLIC `SYNONYM` schema objects from EPAS source projects. The Data Migration Service now properly handles non-public synonym statements during schema migration. `PUBLIC SYNONYM` objects are extracted and uploaded into the Migration Portal project; however, the DMS doesn't yet support migrating them to the destination database. | MIG-8905 |
| Bug fix | Fixed an issue whereby the schema migration capability couldn't migrate database objects for schemas that didn't contain any tables. The migration process now handles all schemas regardless of whether they include tables. | ET-5159 |
| Bug fix | Fixed an issue whereby schema-only migrations dropped PRIMARY KEY constraints and their dependent foreign key and named CHECK constraints when data migration was disabled for specific schemas but enabled for others. The transporter now correctly applies PRIMARY KEY constraints during the finalize schema phase for tables not included in data migration. | ET-5483 |
| Bug fix | Fixed an issue whereby the Data Migration Service validation incorrectly required all schemas in the source database to have a healthy Migration Portal project, even when those schemas weren't included in the migration. The validation now only checks schemas that are actually referenced by the migration's table list. | MIG-8918 |
| Bug fix | Fixed an issue whereby the transformed SQL file had missing line endings when the source SQL file contained Windows-style CR LF line endings. | MIG-8301 |
