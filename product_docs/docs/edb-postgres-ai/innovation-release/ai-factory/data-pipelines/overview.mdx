As a core component of EDB Postgres AI, Pipelines transforms Postgres into a robust platform for AI data management. It abstracts away the complexities of AI workflows by combining `pgvector's` search capabilities with automated data preparation and lifecycle management.

## Integrated data lifecycle

Pipelines automates the journey from raw data to searchable insights. By managing data ingestion, embedding generation, and indexing, it ensures your vector stores stay synchronized with your source data.

- **Flexible model integration**: Choose from supported local models (HuggingFace) or connect to OpenAI-compatible external APIs.

- **Preprocessing automation**: Built-in data preparation steps handle data cleaning and transformation directly within the AI Accelerator.

- **Eliminate stale data**: Auto-processing keeps embeddings current, significantly reducing the hallucinations often caused by outdated vector data.

- **Unified semantic search**: The Intelligent Knowledge Base feature allows you to query text and images using a single function call, regardless of whether the data lives in Postgres tables or S3-compatible object storage.

## Pipeline architecture

Data Pipelines in EDB Postgres AI automate the lifecycle of transforming raw data into vector-searchable insights. They abstract the complexity of cleaning, chunking, and embedding data, ensuring your knowledge bases remain synchronized with your source data.


The AIDB extension introduces the concept of a Pipeline that consists of one or more sequential steps:

1. Data processing configuration

   The core of every pipeline is a data processing configuration which defines how data should be processed before it is indexed.

   Depending on your data source, you can define the two types of processing steps:

   - **Table processing step**: Use `aidb.create_table_preparer()` to source data from existing Postgres tables.

   - **Volume processing step**: Use `aidb.create_volume_preparer()` to source unstructured data like PDFs or images from an external object storage via PGFS.

   **Key configuration parameters**:

   - **Source**: The specific Postgres table or PGFS volume containing the raw input data.

   - **Embedding model**: The specific model used to generate vectors (for example OpenAI, HuggingFace, or local models).

   - **Destination**: The knowledge base where the processed vectors will be stored and indexed.

   **Pulling in data (PGFS)**

   To process files stored outside of Postgres, Pipelines utilize Postgres File System (PGFS) and volumes.

   - **PGFS integration**: PGFS acts as the bridge to cloud storage (S3, GCS, Azure). It allows the database to mount external buckets as if they were local directories.

   - **Volumes**: Once a PGFS location is defined, you create an AIDB volume. This volume allows the Pipeline to scan the external storage, detect new files, and pull them into the processing stream.

1. Orchestration

   Orchestration manages the execution of the pipeline, determining when and how data is processed.

   - **Auto-processing**: You can configure pipelines to run as background processes or keep them perfectly synchronized with your live data. This flexibility allows you to define custom batch sizes and triggers, choosing between synchronous (immediate) or asynchronous (delayed) processing modes depending on your performance needs.

   To eliminate the need for manual intervention, use the `aidb.set_auto_preparer()` function to enable Auto-processing mode, which automatically manages the data flow as changes occur..

   - **Background workers**: The actual work of embedding and indexing is handled by Background workers.

     - **Bulk processing**: For initial data loads, use `aidb.bulk_data_preparation()` to process large datasets in parallel.

     - **Continuous workers**: For ongoing updates, background workers poll the data processing configurations and handle rate-limiting and retries when communicating with external embeddings APIS.

   - **Observability**: You can monitor the status and progress of each auto-processing mode.

1. Pipeline steps

   These transformation steps define the lifecycle of your raw data. You can ingest source data directly from Postgres tables using `aidb.create_table_preparer()` or from S3-compatible object storage via `aidb.create_volume_preparer()`.

   The pipeline then moves this data through a series of modular stages, converting unstructured files into structured, AI-ready vectors.

## Supported models

The AIDB extension supports a diverse range of Open Encoder LLMs from HuggingFace (running locally on your Postgres node) and OpenAI Encoders (via cloud API). You can view all available options in the `aidb.model_providers` table.
