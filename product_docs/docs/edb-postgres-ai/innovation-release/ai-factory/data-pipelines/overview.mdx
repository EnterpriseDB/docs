As a core component of EDB Postgres AI, Pipelines transforms Postgres into a robust platform for AI data management. It abstracts away the complexities of AI workflows by combining `pgvector's` search capabilities with automated data preparation and lifecycle management.

## Integrated data lifecycle

Pipelines automates the journey from raw data to searchable insights. By managing data ingestion, embedding generation, and indexing, it ensures your vector stores stay synchronized with your source data.

- **Flexible model integration**: Choose from supported local models (HuggingFace) or connect to OpenAI-compatible external APIs.

- **Preprocessing automation**: Built-in data preparation steps handle data cleaning and transformation directly within AI pipelines.

- **Eliminate stale data**: Auto-processing keeps embeddings current, significantly reducing the hallucinations often caused by outdated vector data.

- **Unified semantic search**: The Intelligent Knowledge Base feature allows you to query text and images using a single function call, regardless of whether the data lives in Postgres tables or S3-compatible object storage.

## Pipeline architecture

AI pipelines automate the lifecycle of transforming raw data into vector-searchable insights. They abstract the complexity of cleaning, chunking, and embedding data, ensuring your knowledge bases remain synchronized with your source data.


The AIDB extension introduces the concept of a Pipeline that consists of one or more sequential steps:

### Creating a pipeline

   The core of every pipeline is a data processing configuration which defines how data should be processed before it is indexed. You can create a pipeline using the `aidb.create_pipeline()` function, which allows you to specify the data source, the transformations to apply, and the destination for the processed data.

   - **Data source**: Depending on your data source, you can pull data directly from Postgres tables or from external cloud storage (like S3) using PGFS.

   - **Modular step configuration**: The power of a pipeline lies in its ability to chain atomic operations, known as steps. You can create simple single-step pipelines or complex multi-step pipelines that perform a series of transformations on your data.

   - **Orchestration & destination**: Once the pipeline is defined, it manages the heavy lifting of AI processing. It automatically tracks which records have been processed and stores the final output, typically high dimensional vectors into a Knowledge base for semantic search.

### Pulling in data (PGFS)

   To process files stored outside of Postgres, Pipelines utilize Postgres File System (PGFS) and volumes.

   - **PGFS integration**: PGFS acts as the bridge to cloud storage (S3, GCS, Azure). It allows the database to mount external buckets as if they were local directories.

   - **Volumes**: Once a PGFS location is defined, you create an AIDB volume. This volume allows the Pipeline to scan the external storage, detect new files, and pull them into the processing stream.

### Orchestration

   Orchestration manages the execution of the pipeline, determining when and how data is processed.

   - **Auto-processing**: Auto-processing defines the rules for how and when a pipeline stays synchronized with its source. While pipelines are disabled by default to allow for configuration, orchestration provides two primary execution paths:
     - **Manual triggering**: Using `aidb.run_pipeline()`, you can execute a one-time sync of all existing data, which is ideal for initial migrations or historical processing.

    - **Automated modes**: By using `aidb.update_pipeline()`, you can enable modes like Live or Background. This ensures the pipeline automatically reacts to source data changes, keeping Knowledge Bases up-to-date without manual intervention.

   - **Background workers**: Orchestration leverages background infrastructure to handle the heavy lifting of AI tasks without impacting standard database performance.

     - **Asynchronous processing**: In Background mode, workers poll the source at defined `background_sync_interval` periods (ranging from 1 second to 2 days).

     - **Resource management**: Workers manage the complexity of batch sizes and communication with AI models, ensuring that tasks like embedding generation or OCR are performed efficiently in the background.

   - **Observability**: Observability provides the transparency needed to manage automated workflows through specialized views.

     - **Health monitoring**: Views like `aidb.pipeline_metrics` (or the alias pipem) allow you to track the real-time Status of a pipeline, such as UpToDate, Processing, or Failed.

     - **Telemetry**: You can monitor specific metrics, including the count of unprocessed rows in a table's backlog or the number of scans completed for external volumes, ensuring every record is accounted for.

### Pipeline steps

   These transformation steps define the lifecycle of your raw data. You can ingest source data directly from Postgres tables using `aidb.create_pipeline()`.

   The pipeline then moves this data through a series of modular stages, converting unstructured files into structured, AI-ready vectors.

## Supported models

The AIDB extension supports a diverse range of Open Encoder LLMs from HuggingFace (running locally on your Postgres node) and OpenAI Encoders (via cloud API). You can view all available options in the `aidb.model_providers` table.
