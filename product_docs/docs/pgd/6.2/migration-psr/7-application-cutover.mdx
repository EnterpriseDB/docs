## Switching application traffic

With all of the above verification, the PGD cluster is now ready to take over the application traffic.

The method for redirecting write traffic depends on your PGD version and routing architecture. Most modern setups support a write leader switch, which minimizes downtime.

PGD 6.x (Connection Manager): Use the Connection Manager to orchestrate the switch. It provides native support for managing write leaders and ensuring traffic is routed to the correct primary node.

Third-Party Load Balancers: If you are using HAProxy or other external load balancers, trigger a switch-over based on the health check endpoints provided by PGD.


1. The Core Concept: The Write Leader
The Connection Manager works by monitoring the cluster's Raft consensus. It identifies which node is designated as the write_leader. When you want to move traffic, you aren't manually changing application strings; you are telling PGD to promote a different node to be the leader.

2. Performing a Manual Switchover
To move traffic for maintenance or rebalancing, you use the pgd command-line tool. This gracefully transitions the "Write Leader" role from one node to another.

Step A: Identify the Current Leader
First, see where the traffic is currently going:

```bash
pgd show-nodes
```

Look for the node marked with the L (Leader) or W (Writer) attribute.

Step B: Execute the Switch
Use the set-writer command to nominate a specific node to take over the write traffic:

```bash
pgd set-writer <target-node-name>
```

The Connection Manager stops accepting new write transactions on the old leader.

Synchronization: It ensures pending transactions are flushed/replicated.

Promotion: The target node is promoted to Write Leader via Raft.

Redirection: The Connection Manager instantly updates its internal routing table to point to the new node. Your application, connected via the CM endpoint, stays connected throughout.

3. Monitoring the Transition
As your developer mentioned, you should monitor the replication streams during this process to ensure the "handoff" is clean.

Check Replication Rates: 

```sql
SELECT * FROM bdr.node_replication_rates;
```

Verify the New Leader:
Check the logs of the Connection Manager service to see the switching leader event confirmation.

4. Integration with External Tools (HAProxy)
If you are using HAProxy in front of your Connection Managers, HAProxy doesn't need to know about the Postgres nodes. It only needs to health-check the Connection Manager's port (usually 6432 or a dedicated HTTP health port).

The Workflow: HAProxy → Connection Manager → Write Leader.

The Switch: When the CM changes the Write Leader, the health check remains green, and traffic flows through the same pipe to the new destination.
