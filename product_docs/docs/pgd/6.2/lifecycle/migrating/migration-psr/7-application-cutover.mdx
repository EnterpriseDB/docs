With the PGD cluster fully synchronized, this phase details the final transition from your Postgres Streaming Replication (PSR) environment to the Postgres Distributed (PGD) cluster.

## Performing lag verification

The primary health indicator for cutover readiness is the replication lag. You must ensure that the PGD cluster has ingested all outstanding changes from the source node before shifting traffic.

Run the following command on the seed node to ensure that replication lag on the migration subscription is near zero:

```sql
SELECT * FROM bdr.subscription_summary;
```

## Switching the application traffic

Once lag is minimal, the PGD cluster is ready to take over the application traffic by pointing your application to the [Connection Manager](/pgd/latest/connection-manager). The specific method (e.g., updating DNS, Kubernetes ConfigMaps, or `.env` files) depends on your specific application architecture.

To ensure a consistent cutover with no data loss, follow this sequence:

1. Stop all write traffic to the PSR cluster.
1. Update the connection strings to point your application to the Connection Manager.
1. Resume application traffic. The Connection Manager will now route requests to the PGD nodes.

## Performing validation

After the application is live on the PGD cluster, perform final verification checks to ensure the PGD mesh is healthy, receiving data from the application, and propagating across all nodes in the PGD cluster. This involves re-verifying the replication streams to confirm that writes are propagating across all nodes.

1. Confirm that all nodes are in an 'active' state and that internal subscriptions are streaming correctly. On any PGD node, run:

    ```sql
    SELECT * FROM bdr.node_summary;
    SELECT * FROM bdr.subscription_summary;
    ```

1. Check the replication rates to ensure the cluster is processing the new load without building up internal lag. On any PGD node, run:

    ```sql
    SELECT * FROM bdr.node_replication_rates;
    ```

1. Since you stopped the traffic to the old PSR cluster, verify that the migration subscriptions on your PGD nodes show zero lag and no new data incoming. On each PGD node, run:

    ```sql
    SELECT subname, pid, received_lsn, latest_end_lsn, pg_size_pretty(pg_wal_lsn_diff(latest_end_lsn, received_lsn)) as lag FROM pg_stat_subscription;
    ```
