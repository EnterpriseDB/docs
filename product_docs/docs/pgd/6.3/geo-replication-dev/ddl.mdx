DDL operations in a geo-distributed cluster acquire locks across all nodes simultaneously, held for the duration of the statement. Keep DDL transactions small, avoid mixing them with DML, and schedule them during low-traffic periods to minimize the impact on application throughput.

## Using bdr.replicate_ddl_command()

PGD replicates DDL automatically, but for greater control over execution order and locking behavior, use `bdr.replicate_ddl_command()` to run schema changes. It executes the DDL on the local node and replicates it to all other nodes in the specified replication sets, ensuring consistent execution order:

```sql
SELECT bdr.replicate_ddl_command(
    'ALTER TABLE public.transactions ADD COLUMN notes text'
);
```

To target specific replication sets rather than all nodes, check available replication sets first with `SELECT set_name FROM bdr.replication_sets;`, then pass the set name as an array:

```sql
SELECT bdr.replicate_ddl_command(
    'CREATE INDEX idx_transactions_channel ON public.transactions(channel)',
    ARRAY['<replication_set_name>']
);
```

Using `CREATE INDEX CONCURRENTLY` inside `bdr.replicate_ddl_command()` isn't supported. Create concurrent indexes outside of the function, or use a standard `CREATE INDEX` inside it.

## Running DDL safely

Lock acquisition time is proportional to round-trip latency across all nodes, so DDL that takes milliseconds on a local cluster can take seconds in a multi-region setup. Apply these practices to keep the impact minimal.

- **Make additive changes:** Adding a column, adding an index, or creating a new table are low-risk. Dropping columns, renaming columns, and changing data types require more care because they can break running application code on nodes that haven't received the schema change yet. In a rolling deployment, keep backward compatibility until all application versions that depend on the old schema are retired.

- **Add nullable columns first:** If a new column has a `NOT NULL` constraint, existing rows across the cluster need a default value or a backfill before the constraint can be enforced. Add the column as nullable first, backfill the data, then add the constraint in a subsequent migration.

- **Deploy schema changes before application code:** Roll out additive schema changes first, then deploy the application code that uses them. The schema is then in place across all nodes before any node starts sending queries that rely on it.

- **Keep DDL isolated:** Each DDL statement holds a cluster-wide lock. Run DDL statements separately rather than combining them in a single transaction, and never mix DDL with DML. Combining schema changes with data manipulation extends the lock duration to cover the entire transaction.

- **Resolve long-running transactions before running DDL:** Long-running transactions block DDL from acquiring its cluster-wide lock. Check for and resolve any long-running queries or transactions before running DDL.

- **Have a rollback plan:** Not all DDL can be rolled back after it's committed. For destructive changes like `DROP TABLE` or `DROP COLUMN`, have a tested rollback procedure before running the change in production.
