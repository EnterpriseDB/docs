Before migrating an existing application to PGD, assess its compatibility to identify schema patterns, backup gaps, and replication configurations that need to change. These steps apply to applications deployed on a Postgres-compatible database with no replication or at most a Physical Streaming Replication (PSR) setup.

## Using pgd assess

If you have a running Postgres instance serving your application, use `pgd assess` to evaluate its compatibility with PGD. The command connects to your Postgres instance, checks the schema and internal state of the database for the most common application patterns, and provides recommendations on how to adapt them for PGD. It doesn't analyze live traffic or actual queries of your application.

```
pgd assess --dsn <connection_string>
```

For full details on the command and its output, see [pgd assess](/pgd/current/reference/cli/command_ref/assess/).

## Reviewing backup strategies

Identify what tools are currently used for backup. A PGD cluster multiplies the number of nodes to back up, increasing disk space and backup bandwidth requirements. Plan for backups to cover all nodes.

## Reviewing replication and CDC setups

By default, outgoing logical replication to a data warehouse or event streaming pipeline is tied to a single PGD node. Logical replication slots aren't replicated across the cluster unless CDC Failover support is explicitly enabled. With CDC Failover support enabled, slots are replicated and consumers can failover to any node automatically. See [CDC Failover support](/pgd/current/cdc-failover/) for details.

Any synchronization with other databases through a two-phase commit (2PC) protocol or XA transactions isn't compatible with PGD.
