Use a custom scope when your cluster's node groups have different durability requirements. Predefined scopes apply the same policy to every group. A custom scope lets you combine policies, such as synchronous confirmation within your local data center and asynchronous replication to a remote site, with a degrade window that applies only to the cross-region leg.

!!!Note

Commit scopes control when PGD acknowledges a commit. They don't change row-level conflict semantics or application-visible transaction isolation. Two nodes can still commit conflicting changes independently under any scope kind except [Quorum Commit](#guaranteeing-distributed-consistency-with-quorum-commit), which prevents conflicts at commit time rather than resolving them afterward.
!!!

## Durability modes

Custom scopes are built from four durability modes:

- **Synchronous Commit** requires acknowledgment from the designated peers before the commit returns. Which peers, how many, and where they are located is defined by the commit scope group. Pair with `DEGRADE ON` to fall back to async after a timeout if the required peers don't respond.
- **Lag Control** provides async semantics gated by a maximum lag threshold. As the cluster approaches the threshold, PGD throttles application traffic to prevent runaway lag. The commit returns after local durability, but writes slow down before the threshold is breached.
- **Quorum Commit** requires all participating nodes to agree on the transaction outcome before any node commits, preventing conflicting changes from being committed independently across nodes. See [Guaranteeing distributed consistency with Quorum Commit](#guaranteeing-distributed-consistency-with-quorum-commit).
- **CAMO** (Commit At Most Once) enforces single-commit semantics across nodes, preventing a transaction from being applied more than once when a client retries after a connection failure. The client must implement a verification step before retrying.

## Example 1: Combining local majority with cross-region confirmation

These examples use the [two data groups, active-active](/pgd/current/planning/deployment-patterns/two-data-groups/) topology with a witness location: two active data groups (Region A and Region B, each with three data nodes) and a single witness node in Region C.

<img src="../images/commit-scope-anatomy.svg" style="width: 600px; height: auto;" alt="Anatomy of a commit scope" />

<img src="../images/commit-scope-example1.svg" style="width: 480px; height: auto;" alt="Topology diagram showing two selection boxes. The red box covers A2 and A3 in Region A as the local majority. The amber box spans A2 and B2 across regions as the cross-region confirmation clause." />

This scope has two clauses joined by `AND`. The red box shows the local clause: A2 (write leader) and A3 satisfy `MAJORITY ORIGIN GROUP`. The amber box shows the cross-region clause: A2, as the origin node, participates in both; B2 in Region B satisfies `ANY 1 NOT ORIGIN GROUP`. Each clause is evaluated independently and both must be satisfied before the commit returns. `DEGRADE ON` applies only to the cross-region leg — if B2 can't confirm within 10 seconds, that leg falls back to async while the local majority requirement stays enforced.

```sql
SELECT bdr.create_commit_scope(
    commit_scope_name := 'regional_ha',
    origin_node_group := 'region_a',
    rule := 'MAJORITY ORIGIN GROUP SYNCHRONOUS COMMIT
             AND ANY 1 NOT ORIGIN GROUP SYNCHRONOUS COMMIT
                 DEGRADE ON (timeout = 10s, require_write_lead = true) TO ASYNCHRONOUS COMMIT',
    wait_for_ready := true
);
```

Apply this scope as the default for the group:

```sql
SELECT bdr.alter_node_group_option(
    node_group_name := 'region_a',
    config_key := 'default_commit_scope',
    config_value := 'regional_ha'
);
```

## Example 2: Requiring all local nodes and two cross-region nodes

This example tightens both clauses. `ALL ORIGIN GROUP` requires every node in Region A to confirm before the commit returns, rather than a majority. `ANY 2 NOT ORIGIN GROUP` requires two Region B nodes to confirm, rather than one. The result is a stricter scope with no degrade fallback, suitable for workloads where partial confirmation isn't acceptable.

<img src="../images/commit-scope-example2.svg" style="width: 400px; height: auto;" alt="Topology diagram showing two selection boxes. The red box covers all three Region A nodes as the local ALL clause. The amber selection spans A2, B2, and B3, with two solid arrows showing the two required cross-region confirmations." />

The red box covers A1, A2, and A3 because `ALL ORIGIN GROUP` selects every node in the origin group, including A1, which plays no role in Example 1. The amber selection covers A2, B2, and B3, and the two arrows show that both B nodes must confirm. There's no `DEGRADE ON` clause, so if any of the required nodes can't confirm, the transaction waits.

```sql
SELECT bdr.create_commit_scope(
    commit_scope_name := 'strict_ha',
    origin_node_group := 'region_a',
    rule := 'ALL ORIGIN GROUP SYNCHRONOUS COMMIT
             AND ANY 2 NOT ORIGIN GROUP SYNCHRONOUS COMMIT',
    wait_for_ready := true
);
```

See [Commit scope rules](/pgd/current/commit-scopes/commit-scope-rules) for the full syntax, including `ALL`, `ANY n`, `NOT`, and CAMO options.

## Guaranteeing distributed consistency with Quorum Commit

All the scopes above — including Synchronous Commit and the predefined scopes — guarantee **durability**: confirmation that remote nodes received, persisted, or applied the data. What they don't prevent is two nodes independently committing conflicting changes. Conflicts surface after the fact and are resolved using timestamps or other post-commit strategies. Conflict resolution is a cure applied after data has already diverged.

Consider two nodes that simultaneously check an account balance. Both see $10,000. Neither knows the other is mid-transaction. Both commit locally and replicate. The bank now has to untangle an overspent account after the fact, with both purchases already completed.

**Quorum Commit** prevents this from happening. Before any node commits, all participating nodes must agree on the transaction outcome. The first transaction to win the quorum locks the row across the cluster. By the time the second transaction tries to commit, it sees the updated balance and is declined at the point of transaction, not resolved in a cleanup job afterward. Eager detection replaces after-the-fact cure.

If the cluster can't reach agreement, the transaction is rolled back everywhere. Partial commits aren't possible and data never diverges.

Enterprises running high-value workloads in payments, banking, and telecom have historically been forced onto proprietary databases for this guarantee. With Quorum Commit, Postgres delivers the same distributed consistency natively.

To create and use a Quorum Commit scope:

```sql
SELECT bdr.create_commit_scope(
    commit_scope_name := 'banking_scope',
    origin_node_group := 'top_group',
    rule := 'MAJORITY ORIGIN GROUP ON durable QUORUM COMMIT ABORT ON (timeout = 10s)',
    wait_for_ready := true
);
```

```sql
BEGIN;
SET LOCAL bdr.commit_scope = 'banking_scope';
-- ... application writes ...
COMMIT;
```

Because Quorum Commit uses eager conflict resolution, applications encounter serialization errors more often than with asynchronous replication. If `COMMIT` returns a serialization error, re-execute the transaction from `BEGIN`.

See [Quorum Commit](/pgd/current/commit-scopes/quorum-commit) for the full syntax reference, topology-specific examples, and application guidance.

## Choosing a scope kind

| Scope kind | Best for | Key trade-off |
|---|---|---|
| Synchronous Commit | Strong durability within or across groups, with optional degrade on timeout | Higher commit latency than async |
| Lag Control | High-throughput pipelines that need async performance with a bounded recovery point | Commits throttle or block when lag threshold is reached |
| Quorum Commit | High-value workloads in payments or banking where conflicts can't be resolved after the fact | Higher per-transaction latency, and a majority must participate |
| CAMO | Workloads where a retried transaction must never be applied twice | Client must implement a verification step before retrying |
