A physical streaming replication cluster can span multiple data centres — for example, to achieve geographical redundancy.
Failover Manager has no settings specific to node location.
It only requires that each agent can connect to every other agent and database in the cluster.
Keep the following in mind when planning a multi-data-centre deployment.

## Using two data centers

### Simple configuration

The simplest cluster with two data centers is as follows. In this example, a VIP is used for applications to
connect to the primary database. Generally, a different VIP would be needed in each location, and the applications
would need to be aware of each address. Failover Manager can use different VIPs for different nodes by setting
the `virtual.ip.single` property to false. For more information about using a virtual IP address, see
[Using Failover Manager with virtual IP addresses](/efm/latest/04_configuring_efm/05_using_vip_addresses/#using_vip_addresses).

* DC1:
  * Primary node with VIP pointing to it
  * Witness node
* DC2:
  * Standby node

This configuration has the following properties:

* A failure of the primary database or node will cause a failover to the standby database. (This behavior
is no different from the case of having all three nodes in the same location.)
* If the network fails between the two sites, the standby will not promote and the primary database will not
be fenced off. This is because there is a majority of nodes in DC1.
* If the primary site fails entirely, the standby in DC2 can be manually promoted with the
[`efm promote`](/efm/latest/07_using_efm_utility/#efm_promote) command.

The above configuration shares the same behavior as the three-node configuration in terms of network separation or
data center failure. There are more options to control promotion in the five node cluster:

* During a promotion, Failover Manager will promote the most up to date standby, using the standby priority list
to break ties. The [`priority.standbys`](/efm/latest/04_configuring_efm/01_cluster_properties/#priority_standbys)
property can be set to customize the standby priority list based on which node has been promoted to primary.
* To force only Standby1 to be promoted automatically, the other two standbys can be marked as non-promotable by
setting the [`promotable`](/efm/latest/04_configuring_efm/01_cluster_properties/#promotable) property to false.
* To promote a standby in DC2, e.g. to perform maintenance on DC1, Standby2 or Standby3 can be given the highest
priority using the [`efm set-priority`](/efm/latest/07_using_efm_utility/#efm-set-priority) command before performing a
switchover. This works even if the standby was originally marked non-promotable.

## More than two data centers

### Automatic failover for data center failure

For automatic failover in the case of a primary data center failure, the setup can be described simply: have more nodes
outside of the primary data center than in the primary data center. However, network outages in this case can cause
unwanted behavior as described below.

### Three data center case

TBD: diagram with three DCs
* DC1:
  * Primary node with VIP pointing to it
  * Standby1
* DC2:
  * Standby2
  * Standby3
* DC3:
  * Witness node

In this architecture, a complete failure of DC1 will trigger a failover to a standby in DC2. This is because DC2
and DC3 together hold more than half of the cluster's nodes. However, network outages can lead to the
following behvior:

* If DC1 is isolated from the rest of the cluster, the primary database will be fenced off - shut down and the VIP dropped
if one is in use - even though it is running normally. All applications will be disconnected, and a standby in DC2 will be promoted.
Applications running in DC1 will not be able to access the new primary until the network is restored.
(In contrast, a network outage in the two-location architecture has no effect on the running
primary or applications in DC1.)

* If the network between all sites goes down, each site becomes isolated. The primary
will be fenced, but no standby will be promoted. Restart the original primary and Failover Manager will resume monitoring once the nodes reconnect.

* If only DC2 or DC3 is isolated, DC1 continues to function normally.

* If the network allows two locations to lose connectivity with each other while both remain connected to a third, behavior is undefined. For example, if DC1 and DC2 can't reach each other but both can
reach the witness in DC3, agents in DC1 and DC2 may each believe they hold a majority of nodes. This can result in two simultaneous primary databases until the network
is restored.

For these reasons, spreading a cluster across three locations is generally not recommended unless network
outages between the locations are not expected.

#### More than three data centers

Given the complexities of network-related issues with three data centers, it isn't recommended to use more
than three locations.
