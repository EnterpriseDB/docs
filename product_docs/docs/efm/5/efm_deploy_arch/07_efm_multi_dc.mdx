A physical streaming replication cluster can span multiple data centers — for example, to achieve geographical redundancy.
Failover Manager has no settings specific to node location.
It only requires that each agent can connect to every other agent and database in the cluster.
Keep the following in mind when planning a multi-data-center deployment.

## Using two data centers

### Simple configuration

The simple two-data-center cluster setup is shown below.

In this example, applications connect to the primary database using a Virtual IP (VIP). Each data center typically needs its own VIP,  and applications must be configured to know both addresses. To assign different VIPS to different nodes, set the `virtual.ip.single` property to false. For more information, see
[Using Failover Manager with virtual IP addresses](/efm/latest/04_configuring_efm/05_using_vip_addresses/#using_vip_addresses).

* DC1:
  * Primary node with VIP pointing to it
  * Witness node
* DC2:
  * Standby node

This configuration has the following properties:

* If the primary database or node fails, Failover Manager will promote the standby. This behavior
is the same as in a single-location cluster.)
* If the network between the two data centers fails, the standby will not be promoted, and the primary database will not
be fenced. This is because DC1 still holds a majority of nodes.
* If the entire primary site fails, the standby in DC2 can be promoted manually using the
[`efm promote`](/efm/latest/07_using_efm_utility/#efm_promote) command.

### A more robust configuration

The following configuration provides more redundancy for the database servers and allows for more read scalability.

* DC1:
  * Primary node with VIP pointing to it
  * Standby1
  * Witness node
* DC2:
  * Standby2
  * Standby3

This configuration behaves the same as the three-node setup in the case of a network separation or
data center failure. The five-node cluster offers additional options for controlling promotion:

* During a promotion, Failover Manager promotes the most up to date standby, using the standby priority list
to break ties. Use the [`priority.standbys`](/efm/latest/04_configuring_efm/01_cluster_properties/#priority_standbys)
property to customize the priority list based on which node was last promoted to primary.
* To restrict automatic promotion to Standby1 only,  mark the other two standbys as non-promotable by
setting the [`promotable`](/efm/latest/04_configuring_efm/01_cluster_properties/#promotable) property to false.
* To promote a standby in DC2, for example, to perform maintenance on DC1, use the [`efm set-priority`](/efm/latest/07_using_efm_utility/#efm-set-priority) command to give Standby2 or Standby3 the highest priority before performing a
switchover. This works even if the standby was originally marked non-promotable.

## More than two data centers

### Automatic failover for data center failure

The simple rule for automatic failover when the primary data center fails: deploy more nodes
outside of the primary data center than within it. However, this setup can lead to
unintended behavior during network outages, as described below.

### Three data center case

TBD: diagram with three DCs
* DC1:
  * Primary node with VIP pointing to it
  * Standby1
* DC2:
  * Standby2
  * Standby3
* DC3:
  * Witness node

In this architecture, a complete failure of DC1 will trigger a failover to a standby in DC2. This is because DC2
and DC3 together hold more than half of the cluster's nodes. However, network outages can lead to the
following behavior:

* If DC1 is isolated from the rest of the cluster, the primary database will be fenced off - shut down and the VIP dropped
if one is in use - even though it is running normally. All applications will be disconnected, and a standby in DC2 will be promoted.
Applications running in DC1 will not be able to access the new primary until the network is restored.
(In contrast, a network outage in the two-location architecture has no effect on the running
primary or applications in DC1.)

* If the network between all sites goes down, each site becomes isolated. The primary
will be fenced, but no standby will be promoted. Failover Manager will restart the original primary and resume monitoring once the nodes reconnect.

* If only DC2 or DC3 is isolated, DC1 continues to function normally.

* If the network allows two locations to lose connectivity with each other while both remain connected to a third, behavior is undefined. For example, if DC1 and DC2 can't reach each other but both can
reach the witness in DC3, agents in DC1 and DC2 may each believe they hold a majority of nodes. This can result in two simultaneous primary databases until the network
is restored.

For these reasons, spreading a cluster across three locations is generally not recommended unless network
outages between the locations are not expected.

#### More than three data centers

Given the complexities of network-related issues with three data centers, it isn't recommended to use more
than three locations.
