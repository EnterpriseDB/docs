PGFS provides support for Azure Blob Storage and Azure Data Lake Storage Gen2 (ADLS Gen2) using: 

- **Static credential**: PGFS uses Azure's shared access key as static credentials. For more information, see [generic static credentials information](../configuring.mdx/#static-credentials).

- **Microsoft entra ID (service principal)**: PGFS uses modern identity-based access Microsoft entra ID (formerly Azure Active Directory (ADD)) that is client credentials as entra ID. 


## Syntax

PGFS uses the `protocol://path` format in URL for static credentials method and service principal both. Use the following structure to define a location for Azure:

```sql
SELECT pgfs.create_storage_location(
               'storage_location_name',
               'prefix://bucket_name',
               options => '{}',
               credentials => '{}'
       );
```

- PGFS recognizes several URL formats and prefixes to identify Azure resources:

  - **Standard**: `az://`, `adl://`, or `azure://`

  - **Filesystem (ADLS Gen2)**: `abs://` or `abfss://`

  - **Direct HTTPS**: `https://<account>.blob.core.windows.net`, `https://<account>.blob.core.windows.net/<container>`, or `https://<account>.dfs.core.windows.net`

- The `options` argument (JSON), which defines the connection behavior for the storage provider Azure.

  | Option           | Description                                                    |
  |------------------|----------------------------------------------------------------|
  | `account_name`   | The unique identifier for your entire Azure storage resource.  |
  | `container_name` | The specific name of the container within the storage account. |

- The `credentials` argument (JSON), which provides the actual authentication secrets.
  
   - For shared access key:

     | Option        | Description |
     | ------------- | ----------- |
     | `account_key` | The primary secret key used for shared access authentication. |
   
   - For client credentials:

     | Option          | Description                                                     |
     |-----------------|-----------------------------------------------------------------|
     | `client-id`     | The unique ID assigned to your app registration in ADD.    |
     | `client-secret` | The secret key generated for your Azure app registration.       |
     | `tenant-id`     | The unique identifier for your ADD instance. |


## Examples

These examples show how to use PGFS for Azure Blob Storage and Azure Data Lake Storage Gen2 (ADLS Gen2).

### Static credentials (shared access key)

The simplest method for connecting to an Azure storage account. You provide the account name and one of the two access keys found in the Azure portal.

Create a storage location in Azure Blob storage using a shared access key credential:

```sql
SELECT pgfs.create_storage_location('edb_ai_example_images', 'https://edb-account.blob.core.windows.net', 
    NULL, 
    '{
        "account_name": "edb-account",
        "container_name": "ai-images"      
    }', 
    '{
        "account_key": "<your azure storage key>"
    }'
);
```
## Microsoft entra ID (service principal)

For enterprise environments, you can use a service principal. This is more secure than shared keys and allows for granular role-based access control.

Create a storage location in Azure Blob storage using client credentials:

```sql
SELECT pgfs.create_storage_location('edb_ai_example_images', 'https://edb-account.blob.core.windows.net', 
    NULL, 
    '{
        "account_name": "edb-account",
        "container_name": "ai-images"      
    }', 
    '{
        "client_id": "<your azure clientID>", 
        "client_secret":  "<your azure client secret>",
        "tenant_id": "<your azure tenantID>"
    }'
);
```
