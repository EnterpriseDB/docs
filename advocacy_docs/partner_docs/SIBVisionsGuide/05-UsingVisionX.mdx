This section shows the following VisionX usage scenarios with EDB Postgres Advanced Server and EDB Postgres Extended.

**Semi-automated migration of an Oracle Forms application / Oracle APEX to an EDB Postgres / VisionX Low Code application:**
- Oracle Forms migration based on the FMB file.
- Oracle Forms / APEX migration based on the data model

**Use the VisionX Low-code platform to visually build web and mobile applications that use an EDB Postgres Advanced Server/EDB Postgres Extended instance as a data source:**
- Create a Screen that connects to an existing table from an EDB Postgres Advanced Server/EDB Postgres Extended instance.
- Create a Screen that creates a table in the connected EDB Postgres Advanced Server/EDB Postgres Extended instance.

## FMB-based migration of an Oracle Forms application to an EDB Postgres / VisionX Low Code application

This example shows how an Oracle Forms application is migrated to a VisionX low-code application using an FMB file (= Oracle Forms source file). The first step is to migrate the application. In the second step, the Oracle database is migrated to EDB Postgres Advanced Server/EDB Postgres Extended using the usual tools such as EDBs MTK.
Then the migrated application is connected to EDB Postgres Advanced Server/EDB Postgres Extended.

**Part 1: FMB based application migration**

The database user is used to create the tables for the migrated applications’ standard VisionX user management. This is used instead of the Oracle Forms user management. Optionally, other authentication systems such as Windows AD or Open ID system can easily be integrated.

**Part 2: Oracle database to EDB Postgres migration**

In the second step, the Oracle database is migrated to EDB Postgres Advanced Server/EDB Postgres Extended using the EDBs MTK migration tool.

The migration steps in connection with an Oracle Forms migration are not different from a pure Oracle to EDB database migration. Therefore, we do not discuss this topic in detail here. Here is the link to MTK:

https://www.enterprisedb.com/docs/migration_toolkit/latest/

**Part 3: Switch database connection to EDB Postgres**

Switching from the Oracle database is a simple step in VisionX. In the migrated VisionX application, simply click on the `VisionX Settings` screen and choose the `Datasources` tab. Click `Edit` to change the database connection from Oracle to PostgreSQL.

## Data model-based migration of an Oracle Forms / APEX application to an EDB Postgres / VisionX Low Code application

This example shows how an Oracle Forms / APEX application is migrated to a VisionX low-code application using the existing data model. The first step is to migrate the application. In the second step, the Oracle database is migrated to EDB Postgres Advanced Server/EDB Postgres Extended using the usual tools such as EDBs MTK.

Then the migrated application is connected to EDB Postgres Advanced Server/EDB Postgres Extended.

**Part 1: Data model based application migration**

Steps 1-5 are the same as for an FMB-based application migration. **See - Part 1: FMB based application migration steps 1-5.**

6. At the `Screen Infos` step, provide the required information for the screen.

<p align="center">
  <img  src="Images/ScreenInfos.png">
</p>

7. At the `Choose Layout` step, select the required layout for the screen.

<p align="center">
  <img  src="Images/ChooseLayout.png">
</p>

8. At the next step (`Select Data Source`), select the `Use existing data from database tables` option, which allows the user to select the tables used in the original Oracle Forms / APEX screen. The screen will be migrated based on these tables.

<p align="center">
  <img  src="Images/UseExistingData.png">
</p>

9. At the `Choose Database` step, provide the connection information for the Oracle database that is used in the Oracle Forms / APEX application.

<p align="center">
  <img  src="Images/OracleAPEX.png">
</p>

10. On the next screen, select the tables used in the original `Oracle Forms/APEX` screen. The screen will be migrated based on these tables. 

Much of the typically manually created logic in Oracle Forms and APEX is automatically recognized in VisionX. All drop-down lists are recognized based on the Foreign Keys to the master data. If these are missing, they can be defined manually in VisionX. Furthermore, all relevant detail tables are offered for each master table. This makes it very easy to define the master detail relations on the screen. We have selected the appropriate layout of the screen in the previous step. This layout can be changed in the in the UI Designerlater.

<p align="center">
  <img  src="Images/OracleAPEX2.png">
</p>

We choose `s_Ord` in this example because it is the master table of the `Oracle Forms` screen. **See Part 1: FMB-based application migration - step 10.**

<p align="center">
  <img  src="Images/OracleAPEX3.png">
</p>

The dropdown lists to `S_Customer`, `A_Payment Type` and `S_Emp` for the sales rep of the order are detected automatically.

In the next step, we click on `More` and add the Master/Detail relationship to the `S_Item` table.

<p align="center">
  <img  src="Images/SItemTable.png">
</p>

11. Once the screen is created, it will be displayed in Design Mode where the user can adjust it according to the requirements.

<p align="center">
  <img  src="Images/Orders.png">
</p>

To get the image for each product as in the original Oracle Form, we add the product table in the lower area using the `New Table` tab. Here we use the same steps as for the selection of the `S_Ord` table, except that we select the `S_Product` table instead. We choose `S_Product` because it is a detail to the `S_Item` table.

<p align="center">
  <img  src="Images/Orders2.png">
</p>

Repeat the same for the `S_Image` table, because `S_Image` is a detail to the `S_Product` table.

<p align="center">
  <img  src="Images/Orders3.png">
</p>

We then define the `Image` column as `Image` datatype by clicking on the datatype dropdown, so we can later position it on the screen as an image.

<p align="center">
  <img  src="Images/Orders4.png">
</p>

Now we connect the `S_Product` table as a detail to the `S_Item` table. To do this, we select the `S_Product` table at the bottom of the editor and click on the `Edit` icon. Then we click on `More` and select the `S_Item` as `Master Table`.

<p align="center">
  <img  src="Images/Orders5.png">
</p>

Repeat the same for the `S_Image` table, because `S_Image` is a detail to the `S_Product`.

<p align="center">
  <img  src="Images/Orders6.png">
</p>

Select the screen’s detail area by clicking `Details` on the orange bar at the top of the `Orders` screen. Then select the `S_Image` table in the lower area and position the image editor on the top right corner of the screen using drag & drop.

<p align="center">
  <img  src="Images/Orders7.png">
</p>

13. Open the screen in the Live Preview. With a few steps we created the `Oracle Forms` screen in VisionX based on the current data model. We even created an additional list view with a search function.

<p align="center">
  <img  src="Images/Orders8.png">
</p>

<p align="center">
  <img  src="Images/Orders9.png">
</p>

In the second step, the Oracle database is migrated to EDB Postgres Advanced Server/EDB Postgres Extended using the EDBs MTK migration tool.
