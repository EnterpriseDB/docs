Proactively verify cluster health by executing active tasks that simulate real-world operations using the *Monitoring** panel on the left sidebar. Canary checks ensure the database engine responds correctly and meets performance baselines.

### Ensuring system availability with canary checks

Canary checks are recurring, automated scripts that verify the end-to-end integrity of the system. You can configure these tests using the [Management panel](../system-access/management).

- Review the header metrics to get an instant snapshot of system integrity. Compare the **Success** count against the **Total Checks** count to identify if a specific subset of your monitoring is failing.
- Check the status of the **Scheduler** to ensure it is running. This confirms that the WarehousePG Enterprise Manager (WEM) engine is actively triggering your background probes. If the scheduler is stopped, your health data will become stale and you will lose proactive visibility.
- Trigger an immediate execution of any check in the list to verify a fix or test real-time connectivity. While you must go to the [Management panel](../system-access/management) to create or edit a check, the **Monitoring** panel allows you to run and stop them at any time to get an instant status update.
- Monitor the **Avg Duration** metric to establish a baseline for expected responsiveness. A sudden spike in this metric, even if checks are still successful, serves as an early warning of resource saturation or network latency.
- Audit the **Health Checks** table to isolate the root cause of a failure. By checking which specific probe is non-passing, you can determine if the issue is a total service outage or a localized subsystem failure.
