This how-to guide explains how to run **Hybrid Manager (HM)** on Kubernetes with a customer-managed **Prometheus Operator** in place of the bundled `upm-prometheus-operator` Deployment.

**Scope of "BYO Operator":** Only the Prometheus Operator *control plane* (the Deployment that reconciles `monitoring.coreos.com` CRDs) is replaced. HM's `Prometheus`, `Alertmanager`, `ThanosRuler`, `ServiceMonitor`, `PodMonitor` and `PrometheusRule` resources — together with HM's Grafana, dashboards, and alert rules — all remain HM-managed and continue to run unchanged. The BYO operator simply reconciles HM's existing CRs instead of the bundled operator Deployment.

For the bundled stack, see [Use the EDB observability stack](./use-edb-observability-stack).
For architectural guidance, see [Monitoring Kubernetes best practices](/portal/kubernetes/learn/explained/hybrid-manager/monitor-kubernetes-best-practices).

## What changes

-   The bundled `prometheus-operator` Deployment is **not installed**.
-   Your existing Prometheus Operator Deployment is the only reconciler of `monitoring.coreos.com` CRDs cluster-wide.
-   Your operator must reconcile HM's CRs in the namespaces HM uses (typically `edb-observability`).

## Unsupported: the OpenShift built-in Prometheus Operator

**The Prometheus Operator that ships with OpenShift (RHOS) as part of the [cluster monitoring stack](https://docs.redhat.com/en/documentation/openshift_container_platform/4.16/html-single/monitoring/index#understanding-the-monitoring-stack_monitoring-overview) is not a supported BYO target.** It is scoped to the `openshift-monitoring` namespace and cannot reconcile CRs created in `edb-observability` or other HM namespaces.

On OpenShift, HM deploys its own Prometheus Operator instance alongside the cluster monitoring stack and uses the RHOS-supplied CRDs only. If you want a single operator on OpenShift, deploy a separate `kube-prometheus-stack` (or equivalent) operator with cluster scope — do **not** point HM at the default `openshift-monitoring` operator Deployment.

## Prerequisites

Before starting, confirm:

-   **Hybrid Manager 1.4+ (LTS) or 2026.5+ (innovation).** On earlier releases, the bundled `upm-prometheus-operator` component also owns the `edb-observability/monitoring` namespace disabling this will break the observability stack. Do not attempt BYO on earlier releases.
-   A working **Prometheus Operator** Deployment in the target cluster, at a version at or above the HM-shipped baseline of `v0.90.1`. HM is validated against this version; older operators may not understand the CR fields HM sets.
-   The full upstream CRD set is installed and reconciled by your BYO operator. HM creates `Prometheus`, `Alertmanager`, `ThanosRuler`, `ServiceMonitor`, `PodMonitor`, `PrometheusRule`, and `ScrapeConfig` resources, so all of these must be supported by the BYO operator.
-   Your operator's reconcile scope (`--namespaces` / `--deny-namespaces` flags, or `kube-prometheus-stack` `operator.namespaces` values) includes the HM `edb-observability` namespace. Cluster-scoped is simplest.
-   Only one Prometheus Operator Deployment runs in the cluster managing resources in the `edb-observability` namespace — running both yours and HM's leads to dueling reconciles and CR field flapping.

## 1. Disable the bundled operator only

HM disables components via the `disabledComponents` list on the `HybridControlPlane` (HCP) custom resource. The HCP controller treats this list as a per-component opt-out. Add `upm-prometheus-operator` so the bundled Prometheus Operator Deployment is not installed, while the rest of HM's observability stack continues to reconcile.

For operator-managed installs, set the field directly on the HCP CR:

```yaml
apiVersion: edbpgai.edb.com/v1alpha1
kind: HybridControlPlane
metadata:
  name: hybridcontrolplane
spec:
  disabledComponents:
    - upm-prometheus-operator
```

For Helm-based bootstraps via `edbpgai-bootstrap`, set the top-level `disabledComponents` value:

```yaml
disabledComponents:
  - upm-prometheus-operator
```

## 2. Confirm reconcile scope covers HM namespaces

HM creates its monitoring CRs in the `edb-observability` namespace. Your BYO operator must be allowed to reconcile in this namespace.

For `kube-prometheus-stack`:

```yaml
prometheusOperator:
  namespaces:
    releaseNamespace: true
    additional:
      - edb-observability
```

For a vanilla Prometheus Operator install, check the Deployment args for `--namespaces` / `--deny-namespaces`. Cluster-scoped (no `--namespaces` flag) is the simplest and most common configuration.

Verify the operator sees HM's CRs:

```bash
# The operator's reconcile loop should mention edb-observability
kubectl -n <byo-operator-ns> logs deploy/prometheus-operator | grep edb-observability

# And HM's CRs should be visible to whatever operator is running
kubectl -n edb-observability get prometheus,alertmanager,thanosruler
```

## 3. Validate

After the bundled operator is disabled and the BYO operator is scoped:

```bash
# HM CRs are present and reconciled
kubectl -n edb-observability get prometheus,alertmanager,thanosruler

# Operator-managed StatefulSets exist (created by the BYO operator from HM CRs)
kubectl -n edb-observability get statefulset

# HM Prometheus pods are healthy
kubectl -n edb-observability get pods -l app.kubernetes.io/name=prometheus

# Alertmanager and ThanosRuler are healthy
kubectl -n edb-observability get pods -l app.kubernetes.io/name=alertmanager
kubectl -n edb-observability get pods -l app.kubernetes.io/name=thanos-ruler
```

Spot-check HM Grafana — dashboards should render with data because the underlying `Prometheus` instance is unchanged.

If a CR shows `status.conditions[].type=Reconciled` as `False`, the most common causes are:

-   The BYO operator's version is older than the CR's schema fields require.
-   The BYO operator's reconcile scope excludes the `edb-observability` namespace.
